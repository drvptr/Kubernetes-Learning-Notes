## 1. Введение

Kubernetes — это оркестратор контейнеров, который управляет запуском и жизненным циклом приложений. Он работает по декларативной модели: вы описываете желаемое состояние приложения — какие компоненты должны быть запущены и как они должны функционировать. На основе этого описания Kubernetes либо создаёт необходимые ресурсы, либо сообщает о невозможности их создания, а при отклонении от заданного состояния автоматически пересоздаёт или обновляет компоненты. Всякий раз когда вы вносите изменения в описание, Kubernetes предпримет необходимые шаги для
переконфигурации работающего приложения в соответствии с новым описанием.

Kubernetes предполагает на работу на нескольких серверах. Хотя его можно запустить и на одном сервере, основной смысл системы раскрывается при использовании нескольких машин, где она может автоматически распределять нагрузку и управлять приложениями. Группа связанных между собой серверов (физических или виртуальных) называется **кластером** Kubernetes. Сами серверы кластера, называются **нодами**. В случае программного сбоя приложения, kubernetes пересоздаёт его, в случае аппаратной недоступности одной из нод - перераспределяет их на другие ноды кластера.

Kubernetes используется для автоматизации управления контейнеризированными приложениями. Он позволяет не только распределять нагрузку между серверами, но и автоматически масштабировать приложения в зависимости от нагрузки, обновлять их без простоя и поддерживать их работоспособность. В отличие от ручного управления или простых средств оркестрации, Kubernetes берёт на себя ключевые операционные задачи: перезапуск сервисов при сбоях, перераспределение нагрузки и масштабирование.

## 2. Собираем Кластер

Кластер Kubernetes состоит из двух типов нод **control-plane** и **worker-нод**. Control-plane управляет состоянием кластера: хранит описание желаемого состояния системы (desired state) и принимает решения о размещении и состоянии приложений. Worker-ноды выполняют полезную нагрузку — на них запускаются контейнеры и работают приложения.  Логическая единица Kubernetes, описывающая набор контейнеров, которые запускаются на одной worker-ноде с общей сетью и ресурсами называется Pod'ом. Он хранится на control-plane как описание желаемого состояния. Worker-ноды получают Pod’ы через control-plane и выполняют необходимую работу для достижения описанного в Pod'ах состояния. Для сборки тестового Kubernetes-кластера необходимо развернуть пять виртуальных машин. Чтобы поднять Kubernetes-кластер на Linux, на каждую ноду необходимо установить базовые компоненты и правильно подготовить систему на уровне ядра и сети. 

### 2.1 Требования к Сети

Для работы Kubernetes требуется, чтобы все узлы кластера были взаимно доступны по сети. Это означает наличие корректной маршрутизации между нодами без NAT и сетевых ограничений, которые мешают прямому взаимодействию между компонентами. Зафиксируем инфраструктуру через /etc/hosts, чтобы все узлы резолвились по именам:

```hosts
192.168.56.101 control1
192.168.56.102 control2
192.168.56.103 control3
192.168.56.111 worker1
192.168.56.112 worker2
```
Кроме того, на уровне ядра Linux, на каждой из нод, необходимо включить следующие параметры:
- `sysctl net.bridge.bridge-nf-call-iptables=1` включает обработку bridged-трафика через iptables, то есть пакеты, проходящие через Linux bridge, попадают в цепочки фильтрации сети — это важно для работы NetworkPolicy и корректного маршрутизации трафика. 
- `sysctl net.ipv4.ip_forward = 1` включает IP forwarding в ядре, позволяя системе маршрутизировать пакеты между интерфейсами, что критично для взаимодействия Pod’ов между разными нодами.
- `modprobe br_netfilter` обеспечивает интеграцию bridge-сетей с netfilter подсистемой ядра, без него Kubernetes CNI не сможет корректно применять правила фильтрации к pod-to-pod трафику. 
Для того чтобы эти настройки сохранялись после перезагрузки системы, их необходимо добавить в файл: `/etc/sysctl.d/99-kubernetes-cri.conf`
```bash
cat <<- EOF | sudo tee /etc/sysctl.d/99-kubernetes-cri.conf
net.bridge.bridge-nf-call-iptables  = 1
net.ipv4.ip_forward                 = 1
net.bridge.bridge-nf-call-ip6tables = 1
EOF
```
Со стороны firewall, на worker-нодах важно разрешить соединения на порты 10250 - используются kubelet API, а также 80 и 443 для входящего трафика приложений через сервисы и ingress. На control-plane нодах ключевыми являются 6443 - Kubernetes API server, 2379–2380 - etcd — хранилище состояния кластера, а также 10250 для доступа к kubelet. Дополнительно используются порты для CoreDNS - 53, для metrics - 9153, а ingress-контроллеры часто используют NodePort диапазон (30000–32767) через который публикуются внешние сервисы, перечисленные порты можно открыть на всех нодах. А также порт 10248 Healtz.

### 2.2 Установка Пакетов

В первую очередь устанавливаются пакеты **kubelet**, **kubeadm** и **kubectl**, обычно из репозитория Kubernetes `https://pkgs.k8s.io/core:/stable:/<VERSION>/rpm`.
- **kubectl** — основной клиент для управления кластером.
- **containerd** (например из `https://download.docker.com/linux/centos/9/x86_64/stable`), является container runtime и непосредственно отвечает за создание контейнеров, также им можно пользоваться аналогично docker `crictl ps -a`.
- **kubelet** — это основной агент k8s на каждой ноде, который работает постоянно и отвечает за запуск и контроль Pod’ов на конкретной машине. Он взаимодействует с container runtime и с api-server. 
- **kubeadm** используется для инициализации кластера и добавления новых нод, то есть это инструмент bootstrap’а кластера. .

На каждой ноде должны быть запущены два ключевых сервиса - **containerd** и **kubelet**. Перед запуском containerd необходимо предварительно настроить конфигурацию, отредактировав файл /etc/containerd/config.toml со следующим содержанием:
```toml
version = 2
[plugins]
  [plugins."io.containerd.grpc.v1.cri"]
    [plugins."io.containerd.grpc.v1.cri".containerd]
      discard_unpacked_layers = true
      [plugins."io.containerd.grpc.v1.cri".containerd.runtimes]
        [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc]
          runtime_type = "io.containerd.runc.v2"
          [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
            SystemdCgroup = true
```
Затем, можно включить авозагрузку служб:
```bash
systemctl enable --now containerd
systemctl enable kubelet
```
Дополнительно обычно отключают swap, потому что Kubernetes не рассчитывает на своппинг памяти — это ломает предсказуемость ресурсов и может приводить к нестабильности scheduling. В учебных целях необходимо отключить SELinux или AppArmor, чтобы избежать дополнительных ограничений

### 2.3 Инициализация Кластера и Добавление Нод

Сразу включим автодополнение для kubectl. Эта команда подключает bash completion, позволяя использовать TAB для автодополнения команд, ресурсов и имён объектов Kubernetes.
```bash
source <(kubectl completion bash)
```
Для инициализации control-plane ноды необходимо зайти на виртуальную машину control1 и выполнить команду от пользователя root:
```bash
kubeadm init
```
После завершения инициализации создаётся административный конфигурационный файл `/etc/kubernetes/admin.conf`
Он используется для доступа к API server с правами администратора кластера. Чтобы использовать его через kubectl, необходимо скопировать его в стандартное расположение:
```bash
mkdir -p $HOME/.kube
cp /etc/kubernetes/admin.conf $HOME/.kube/config
```
Также в процессе инициализации создаётся конфигурация для _kubelet_, /var/lib/kubelet/config.yaml, теперь мы можем его запустить:
```bash
systemctl start kubelet
```
После этого становится доступно управление кластером через kubectl. Для проверки состояния кластера можно выполнить:
```bash
kubectl get all -A
```
Данная команда отображает все основные ресурсы: Pod'ы, сервисы, ноды. Центральным компонентом Kubernetes является kube-apiserver. Как видно из вывода, он представлен как Pod в namespace kube-system. Именно через него проходят все операции управления кластером: создание Pod’ов, Service’ов и других объектов. Состояние всего кластера хранится в базе данных etcd, которая также запускается как отдельный Pod на control-plane ноде. Состояние Pod’а в поле READY 1/1 означает, что контейнер внутри Pod’а успешно запущен и функционирует. Если состояние отличается, это указывает на проблему запуска. Скорее всего вы увидите статус NotReady. Если посмотреть подробнее:
```bash
kubectl describe node control1
```
в выводе будет что-то вроде:
```
NetworkPluginNotReady: network plugin is not ready
```
Это связано с тем, что Kubernetes сам по себе не реализует сетевое взаимодействие между Pod’ами. Он не назначает им IP-адреса и не настраивает маршрутизацию — эти задачи выполняются внешними CNI-плагинами. Без установленного CNI kubelet не может подтвердить готовность сетевого стека, поэтому ноды остаются в состоянии NotReady. При этом control-plane продолжает работать, и на ранних этапах это может быть неочевидно. Для исправления этой ситуации установим сетевой плагин Calico:
```bash
kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml
```
После установки через некоторое время ноды перейдут в состояние Ready.
Также, если кластер развёрнут на RHEL, как в моём случае для работы Calico потребуется добавить интерфейсы вида cali+ в текущую зону firewall:
```bash
firewall-cmd --add-interface=cali+ --permenent
firewall-cmd --reload
```
Для более низкоуровневой диагностики можно использовать:
```bash
crictl ps -a
``` 
Данная команда показывает реальные контейнеры, запущенные через _container runtime_, их состояния и количество перезапусков. Важно понимать, что containerd не знает о концепции Pod’ов Kubernetes — он работает только с контейнерами. Связь между Pod’ами и контейнерами обеспечивает kubelet. До выполнения kubeadm init на ноде работает только kubelet как системный агент. Сам по себе он не формирует кластер и не управляет распределённой системой. Его основные функции:
- взаимодействие с container runtime (containerd),
- запуск и контроль Pod’ов взаимодействуя с apiserver,
- обработка static pod манифестов.

Именно kubeadm init подготавливает Kubernetes-окружение: создаёт директорию /etc/kubernetes/manifests/, в которую помещаются static pod манифесты control-plane компонентов, генерирует сертификаты и создаёт административный kubeconfig. Кроме того, kubeadm создаёт bootstrap token, который используется для подключения worker-нод к кластеру. Если токен был утерян, его можно пересоздать командой:
```bash
kubeadm token create --print-join-command
```
В результате будет выведена готовая команда для подключения worker-ноды к кластеру, например:
```bash
kubeadm join 192.168.56.101:6443 --token 8z6ram.52inp92e19to4nvn --discovery-token-ca-cert-hash sha256:df509938ded24842cc2f57f28cc2a6d43e6043ebda904e5eee00854c1df3c0a0

```
Зайдём на worker1 и на worker2 машины и введём данную команду. После выполнения этой команды на worker-ноду она подключается к control-plane и регистрируется в кластере. Проверить список нод можно командой:
```bash
kubectl get nodes 
```
Если какая-либо нода не переходит в состояние Ready, её состояние можно диагностировать через:
```bash
kubectl describe node worker1
```
Данная команда применима не только к нодам, но и к любым Kubernetes-ресурсам. Также важно понимать, что kubectl не привязан к конкретной ноде. Он может использоваться с любой машины, имеющей доступ к API server. Для этого достаточно иметь корректный kubeconfig-файл (~/.kube/config или /etc/kubernetes/admin.conf). Например можно выполнить `kubectl describe pod kube-apiserver-control1 -n kube-system`. Аргумент `-n` в означает _namespace_ о них чуть позже. Посмотреть ресурс во всех namespace можно через `-A`. Также важно понимать, что kubectl не привязан к конкретной ноде. Он может использоваться с любой машины, имеющей доступ к API server. Для этого достаточно иметь корректный kubeconfig-файл `~/.kube/config`.

## 3. Создание Static Pods

Pod’ы в Kubernetes описываются манифестами — YAML-файлами, в которых задаётся желаемое состояние какого-нибудь приложения. В обычной ситуации этот манифест отправляется в kube-apiserver, где он проверяется и сохраняется в etcd. Всё это время kube-scheduler, подключен к kube-apiserver, и как только он получает ответ о наличии пода без назначенной ноды, он вычисляет ноду, на которой Pod должен быть запущен, после чего возарщает её kube-apiserver. Kubelet на этой ноде, аналогично, наблюдая за API server, получает это описание и запускает контейнеры через container runtime. У этой схемы есть проблема, сам kube-apiserver и другие компоненты control-plane тоже являются Pod’ами. 

Static pod — это Pod, который создаётся не через API server, а напрямую kubelet’ом. Для этого kubelet отслеживает специальную директорию, по умолчанию `/etc/kubernetes/manifests/`. Все YAML-файлы, которые туда попадают, kubelet воспринимает как манифест Pod’ов и сразу же пытается их запустить через container runtime. Путь в котором хранятся манифесты _static pods_ вприципе можно изменить в конфигурации самого kubelet `/var/lib/kubelet/config.yaml`. _Static pods_ - это единственный способ “заставить kubelet что-то делать без API server”. Именно так при kubeadm init запускаются kube-apiserver, etcd, kube-controller-manager и kube-scheduler. kubeadm просто кладёт их манифесты в /etc/kubernetes/manifests/, а kubelet поднимает их как контейнеры.

Использовать static pods для пользовательских приложений не имеет смысла. Они не участвуют в нормальной логике Kubernetes: их не планирует scheduler, ими нельзя управлять через API, и они жёстко привязаны к конкретной ноде. Это инструмент именно для bootstrap’а и системных компонентов.

Чтобы увидеть, как это работает, можно создать свой static pod. Сначала сгенерируем манифест:
```bash
kubectl run static-pod --image=nginx --dry-run=client -o yaml > static-pod.yaml
cat static-pod.yaml
```
В этом файле описан обычный Pod с контейнером nginx. Если переместить его в директорию /etc/kubernetes/manifests/, kubelet автоматически обнаружит новый файл и создаст Pod без каких-либо дополнительных команд:
```bash
mv static-pod.yaml /etc/kubernetes/manifests/
kubectl get pods static-pod-control1
```
Через вывод команды `kubectl get pods` вы можете убедиться что Pod автоматически создан. С ним можно работать, смотреть логи `kubectl logs`, выполнять команды `kubectl exec`, синтаксис клиента в данном случае похож на docker:
```bash
kubectl exec -it static-pod-control1 -- /bin/sh
```
Удаление происходит просто удалением файла:
```bash
rm -f /etc/kubernetes/manifests/static-pod.yaml
kubectl get pods
```
После этого kubelet автоматически остановит и удалит Pod.

## 4. Управление Состоянием Нод

В Kubernetes можно управлять тем, участвует ли нода в размещении новых Pod’ов и что происходит с уже запущенными. Для этого используются команды `cordon`, `drain` и `uncordon`.

Команда `kubectl cordon` переводит ноду в состояние **unschedulable**. Это означает, что новые Pod’ы на неё больше не будут назначаться, однако уже запущенные продолжают работать без изменений. Фактически scheduler просто перестаёт рассматривать эту ноду при размещении.

Команда `kubectl drain` делает более жёсткое действие. Она не только запрещает размещение новых Pod’ов, но и удаляет уже запущенные, чтобы они были пересозданы на других нодах. Это используется, например, перед обслуживанием или выключением сервера.

При использовании `drain` могут возникать нюансы. По умолчанию DaemonSet Pod’ы не удаляются, поэтому часто используется ключ:` --ignore-daemonsets`. Также, если Pod использует проброшенные volume `emptyDir`, его данные будут потеряны при удалении. Для явного подтверждения используется ` kubectl drain <node> --delete-emptydir-data`

Команда `kubectl uncordon` возвращает ноду в нормальное состояние, после чего scheduler снова начинает использовать её для размещения Pod’ов. Рассмотрим поведение scheduler на практике. Сначала переведём одну из нод в состояние unschedulable:

```bash
kubectl cordon worker1
kubectl describe node worker1
```
Обратим внимание на вывод `describe`, в поле Taint можно заметить значение `NoSchedule` - это означает что нода выведена из планировщика. Помимо этого `descibe` может дать и другую полезную информацию:
```
Conditions:
  Type                 Status  LastHeartbeatTime                 LastTransitionTime                Reason
                     Message
  ----                 ------  -----------------                 ------------------                ------
                     -------
  NetworkUnavailable   False   Thu, 16 Apr 2026 15:58:29 -0400   Thu, 16 Apr 2026 15:58:29 -0400   CalicoIsUp                   Calico is running on this node
  MemoryPressure       False   Thu, 16 Apr 2026 20:31:27 -0400   Mon, 23 Mar 2026 21:26:11 -0400   KubeletHasSufficientMemory   kubelet has sufficient memory available
  DiskPressure         False   Thu, 16 Apr 2026 20:31:27 -0400   Mon, 23 Mar 2026 21:26:11 -0400   KubeletHasNoDiskPressure     kubelet has no disk pressure
  PIDPressure          False   Thu, 16 Apr 2026 20:31:27 -0400   Mon, 23 Mar 2026 21:26:11 -0400   KubeletHasSufficientPID      kubelet has sufficient PID available
  Ready                True    Thu, 16 Apr 2026 20:31:27 -0400   Sat, 11 Apr 2026 01:48:45 -0400   KubeletReady                 kubelet is posting ready status
```
Как видно, сеть доступна, нехватки памяти, диска, pid не наблюдается. Теперь создадим Deployment с несколькими репликами:

```bash
kubectl create deploy test --image=nginx --replicas=5
kubectl get pods -o wide
```
Можно заметить, что все Pod’ы запущены только на одной ноде. Это происходит потому, что `worker1` исключён из планирования, и scheduler использует только доступные ноды. Удалим Deployment:

```bash
kubectl delete deployment test
```
Вернём ноду в работу:

```bash
kubectl uncordon worker1
```
И создадим Deployment снова. Теперь Pod’ы будут распределены между нодами более равномерно, так как обе ноды снова доступны для scheduler.

## 5. Резервное Копирование etcd

Как уже говорилось ранее, etcd — это база данных, в которой хранится всё состояние Kubernetes-кластера: Pod’ы, Deployment’ы, Service’ы и другие ресурсы. Создание резервной копии etcd фактически означает сохранение состояния всего кластера. Для работы с резервными копиями используются утилиты `etcdctl` и `etcdutl`, которые входят в состав проекта [etcd-io](https://github.com/etcd-io/etcd). По умолчанию доступ к etcd защищён с помощью TLS. Узнать параметры запуска etcd можно командой:

```bash
ps aux | grep etcd
```

Из вывода видно, что используются два порта:
* `2379` — клиентский порт, через который происходит доступ к базе данных. Именно к нему выполняется подключение при создании резервной копии.
* `2380` — порт для взаимодействия между узлами etcd в кластере (например, между несколькими control plane узлами).

Для выполнения резервного копирования используется порт `2379`. Создадим snapshot работающего etcd:

```bash
etcdctl \
  --endpoints=localhost:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /tmp/etcdbackup.db
```

Проверим созданную резервную копию:

```bash
etcdutl snapshot status /tmp/etcdbackup.db
```

Для проверки создадим тестовый Deployment:

```bash
kubectl create deploy test --image=nginx --replicas=5
kubectl get deploy
```

Теперь восстановим состояние кластера из резервной копии. Перед восстановлением необходимо остановить static pods control plane, поскольку kube-apiserver и etcd будут использовать каталог данных, который мы собираемся заменить:

```bash
mkdir ~/backup
mv /etc/kubernetes/manifests/*.yaml ~/backup/
crictl ps -a
```

После остановки static pods восстановим snapshot в новый каталог данных:

```bash
mv /var/lib/etcd /var/lib/etcd-old
etcdutl snapshot restore /tmp/etcdbackup.db --data-dir /var/lib/etcd
```

После завершения восстановления вернём static pods:

```bash
mv ~/backup/*.yaml /etc/kubernetes/manifests/
```

Проверим состояние кластера:

```bash
kubectl get deploy
```

Из вывода видно, что Deployment, созданный после создания snapshot, исчез, так как состояние кластера было возвращено к моменту создания резервной копии.

## 6. Обновление Kubernetes

Обновление Kubernetes выполняется без пропуска minor-версий. Нельзя перескочить, например, с 1.29 сразу на 1.32 — необходимо проходить каждую промежуточную версию (1.29 → 1.30 → 1.31 → 1.32). Обновление всегда происходит в следующем порядке
1. Cначала обновляется _kubeadm_
2. Затем control-plane
3. После этого worker-ноды.

### 6.1 Обновление Control Plane Node

Проверим текущую версию:
```bash
kubectl get nodes
```
```
NAME            STATUS   ROLES           AGE   VERSION
control1        Ready    control-plane   26d   v1.28.15

```
Из вывода видно, что в данный момент установлена версия 1.28. Обновление начинается с настройки репозитория. Для этого, например на RHEL можно отредактировать файл `/etc/yum.repos.d/rhel.repo`. Нас интересует следующая строка:
```
baseurl = https://pkgs.k8s.io/core:/stable:/v1.28/rpm/
```
Необходимо изменить ссылку на версию выше, в нашем случае, конечная версия будет выглядеть следующми образом:
```
baseurl = https://pkgs.k8s.io/core:/stable:/v1.29/rpm/
```
После этого можно посмотреть доступные версии kubeadm и установить нужную:
```bash
dnf --showduplicates list kubeadm
dnf install kubeadm-1.29.15-*
kubeadm version
```
Вывод последней команды из представленного листинга, должен показать новую версию. Таким образом мы обновили утилиту _kubeadm_. Теперь необходимо обновить control-plane. Перед тем как это сделать, имеет смысл проверить возможно ли обновление. Сделать фактическую проверку можно командой:
```bash
kubeadm upgrade plan
```
Если ошибок обнаружено не было, можно запускать обновление. Производится это командой:
```bash
kubeadm upgrade apply v1.29.15
```
На этом этапе обновляются основные компоненты control-plane: _kube-apiserver, kube-controller-manager и kube-scheduler_. После успешного обновления, должнен появиться вывод ` [upgrade/successful] SUCCESS! Your cluster was upgraded to "v1.29.15". Enjoy!`. Это наиболее опасная часть, потому что мы видим успех - звучит будто бы всё готово. Но это не так. Если присмотреться внимательнее можно обнаружить строку `[upgrade/kubelet] Now that your control plane is upgraded, please proceed with upgrading your kubelets if you haven't already done so.` . Обновление ещё не закончено — kubelet на этой ноде всё ещё работает на старой версии. Чтобы корректно завершить обновление, необходимо обновить kubelet. Перед этим ноду переводят в состояние, в котором на неё не будут назначаться новые Pod’ы:
```bash
kubectl drain control1 --ignore-daemonsets
```
После этого обновляются пакеты:
```bash
dnf install kubelet-1.29.15-* kubectl-1.29.15-*
```
И перезапускается kubelet:
```bash
systemctl daemon-reload
systemctl restart kubelet
```
Когда kubelet обновлён, ноду возвращают в работу:
```bash
kubectl uncordon control1
```
Если теперь снова посмотреть список нод `kubectl get nodes`, будет видно, что control-plane уже работает на версии v1.29.15. При этом worker-ноды всё ещё будут оставаться на старой версии — это допустимо на время обновления, но в итоге все ноды должны быть приведены к одной версии.

### 6.2 Обновление Worker Node

~~Обновление worker-ноды выполняется аналогично, но с одним отличием — вместо `kubeadm upgrade apply` используется команда~~ это команда для второго и третьего control plane.:
```bash
kubeadm upgrade node
```
Перед этим также обновляется kubeadm через репозиторий, как и на control-plane. Команда kubeadm upgrade node обновляет локальную конфигурацию ноды, чтобы она соответствовала новой версии кластера. Для обновления worker-node нам надо также к ней подключиться, скорректировать репозиторий на новую версию, остальные шаги аналогичны и приведены в листинге ниже:
```bash
dnf install kubeadm-1.29.15-*
kubeadm upgrade node
kubectl drain worker1 --ignore-daemonsets
dnf install kubelet-1.29.15-* kubectl-1.29.15-*
systemctl daemon-reload
systemctl restart kubelet
kubectl uncordon worker1
kubectl get nodes
```
Вывод последней команды должен показать новую версию также и у worker-нод.

## 7. Отказоустойчивый Кластер

В текущей конфигурации Kubernetes используется одна control-plane нода. В случае недоступности control-plane ноды кластер формально  продолжает работает - worker-ноды выполняют уже запущенные Pod’ы. Однако, управление кластером, в таком случае, полностью теряется. Становится невозможным создание новыx Pod’ов, кластер перестаёт реагировать на сбои.

Причина в том, что все управляющие компоненты — _kube-apiserver, kube-scheduler и kube-controller-manager_ — работают именно на control-plane. Особенно важен _kube-controller-manager_, который постоянно приводит текущее состояние кластера к желаемому (desired state). Если control-plane недоступен, этот механизм перестаёт работать, и со временем кластер начинает “расходиться” с описанным состоянием. В случае сбоев восстановление уже не гарантируется.

Чтобы избежать этой проблемы, control-plane делают отказоустойчивым, добавляя несколько нод - нечётное количество, например три. Нечётное число связано с тем, что Kubernetes использует распределённое хранилище etcd, которое работает по принципу кворума. Для корректной работы необходимо большинство доступных узлов, поэтому в случае с чётным числом, например 2, кворум не достигается при отказе одного из улов - 1/2 не достаточно.

Существует два основных варианта построения отказоустойчивого control-plane. 
— Stacked Control Plane: etcd работает на тех же нодах, что и остальные компоненты Kubernetes. 
- External etcd topology: etcd развёрнут на отдельных серверах.
Первый вариант проще в настройке и чаще используется в учебных и небольших инсталляциях, именно его мы будем использовать. Второй — более надёжный и применяется в production.

Ноды control-plane присоединяются аналогично worker-нодам, но со специальным флагом '_--control-plane_'
```
kubeadm join 192.168.0.101:6443 --control-plane
```

При этом важно понимать, что Kubernetes сам по себе не предоставляет готового механизма высокой надёжности. Он позволяет объединить несколько control-plane нод в кластер, но не решает задачу балансировки запросов. Когда мы добавляем вторую/третью control-plane ноду, происходит не “магия”, а вполне конкретные вещи - поднимаются static pod’ы такие же как на первой ноде, etcd становится реплицируемым - был один узел, а становится 3 узла с репликацией. Если просто присоединить три control-plane ноды в кластер, каждая из них будет работать, но запрос к кластеру всё равно будет идти по конкретному адресу, например вы можете заметить это в `/etc/kubernetes/kubelet.conf`. В случае отказа этой основной ноды доступ к кластеру снова будет потерян. Получается ситуация, что формально средства для отказоустойчивости есть, но на практике они не задействованы.

Для решения этой проблемы используется стороннее ПО, никак не связанное с kubernetes. Например самый простой вариант — использование Keepalived, который реализует виртуальный IP-адрес. Этот адрес “переезжает” между нодами при отказе, позволяя сохранить доступ к API server даже при падении одной из control-plane нод. Такая схема уже обеспечивает базовую отказоустойчивость. Например, представим что имеется виртуальный IP 10.0.0.88, который прикреплён к ноде control1. Как только нода становится недоступной - этот адрес принимает на себя control2. У данной схемы можно сразу же заметить недостаток - в каждый момент времени весь трафик идёт только на одну ноду - на ту ноду что обладает virtual IP. Остальные control-plane ноды при этом простаивают и не участвуют в обработке запросов. Это означает, что ресурсы используются неэффективно, а нагрузка не распределяется. Поэтому на control-plane нодах дополнительно разворачивается балансировщик нагрузки, например HAProxy. На порту 6443, он ожидает TCP-трафик, и распределяет его по нодам. В этом случае, после поступления на сервер control1, запросы к API server распределяются между всеми control-plane нодами, что позволяет не только обеспечить отказоустойчивость, но и использовать ресурсы кластера более эффективно.


```ASCII
+-----------------------+       +-----------------------+       +-----------------------+ 
|    CONTROL PLANE 1    |       |    CONTROL PLANE 2    |       |    CONTROL PLANE 3    |
|     ( 10.0.0.1 )      |       |     ( 10.0.0.2 )      |       |     ( 10.0.0.3 )      |
|        haproxy <----------------------> haproxy <---------------------> haproxy       |
|            |          |       |            |          |       |            |          |
|       keepalived      |       |       keepalived      |       |       keepalived      |
|     ( 10.0.0.88 )     |       |     (~10.0.0.88~)     |       |     (~10.0.0.88~)     |
+------------|----------+       +------------|----------+       +------------|----------+
             |                               |                               |
             | ~  ~~  ~~  (reserved)  ~~  ~~  ~~  ~~  ~~  (reserved)  ~~  ~~
             | 
             +-------------------------------+
             |                               |
+------------|----------+       +------------|----------+       
|            |          |       |            |          |       
|       WORKER 1        |       |       WORKER 2        |       
|                       |       |                       |       
+-----------------------+       +-----------------------+      
```
Далее будет рассмотрена практическая реализация такой схемы. Итак, необходимо полностью стереть кластер:
```bash
kubeadm reset -f
rm -rf /etc/cni/net.d
rm -rf /var/lib/cni
rm -rf /var/lib/etcd
rm -rf ~/.kube
iptables -F && iptables -t nat -F && iptables -t mangle -F && iptables -X
systemctl restart containerd
```
На **каждой control plane** ноде устанавливается _keepalived_ и _haproxy_.
```bash
dnf install haproxy keepalived
```
Как уже говорилось ранее, для обеспечения высокой доступности control-plane мы будем использовать Keepalived. В его основе лежит идея виртуального IP-адреса (VIP), через который осуществляется доступ к Kubernetes API. Этот адрес не привязан жёстко к конкретной машине и может “переезжать” между нодами. В качестве такого адреса можно использовать любой свободный IP из нашей сети, например 192.168.56.88/24.

По умолчанию keepalived отслеживает только доступность самой ноды. Это означает, что переключение VIP произойдёт только в случае полного отказа сервера. Однако возможна ситуация, когда нода продолжает работать, но kube-apiserver на ней недоступен. Например ситуация Memory Pressure или высокого IOPS - нода отвечает на ICMP, но при этом любые приложения на запросы не отвечают, так как процессор выполняет системные или программные прерывания. В таком случае keepalived не выполнит переключение, и кластер фактически станет недоступен, несмотря на наличие других control-plane нод.

Чтобы этого избежать, добавим простой механизм проверки состояния API. Для этого создадим скрипт, который будет периодически выполнять HTTP-запрос к kube-apiserver и в случае ошибки сигнализировать keepalived о проблеме:
```bash
#!/bin/bash
APISERVER_VIP=192.168.56.88
APISERVER_DEST_PORT=6443

errorExit() { 
	echo "* * * $*" 1>&2
	exit 1
}

curl --silent --max-time 2 --insecure https://localhost:${APISERVER_DEST_PORT}/ -o /dev/null || errorExit "Error GET https://localhost:${APISERVER_DEST_PORT}/"
if ip addr | grep -q ${APISERVER_VIP}; then
	curl --silent --max-time 2 --insecure https://${APISERVER_VIP}:${APISERVER_DEST_PORT}/ -o /dev/null || errorExit "Error GET https://${APISERVER_VIP}:${APISERVER_DEST_PORT}/"
fi
```
Сохраним его в файл `/etc/keepalived/check_apiserver.sh` и сделаем исполняемым:
```bash
chmod +x /etc/keepalived/check_apiserver.sh
```
Далее настроим минимальную конфигурацию keepalived. Предварительно сохраним исходный конфиг и создадим новый:
```bash
mv /etc/keepalived/keepalived.conf /etc/keepalived/keepalived.conf.bak
vim /etc/keepalived/keepalived.conf
```
Ниже приведён шаблон конфигурации. Адаптируйте конфигурацию под свою инфраструктуру. В параметре `interface` необходимо указать реальный сетевой интерфейс ноды (например, `enp0s5`), иначе keepalived просто не сможет повесить виртуальный IP. Узнать имя интерфейса можно командой `ip link`. Значение `priority` задаёт “вес” ноды при выборе владельца VIP: на основной ноде обычно указывают максимальное значение (например, 255), на второй — 254, на третьей — 253, чтобы был предсказуемый порядок переключения. Соответственно, на первой ноде указывается `state MASTER`, а на остальных — `state SLAVE`. Также необходимо, чтобы виртуальный IP находился в той же подсети, что и интерфейс: если у вас используются адреса вида `10.x.x.x` или `172.x.x.x`, то и VIP должен быть из этой сети, иначе он не будет корректно анонсироваться и использоваться.

```
global_defs {
    enable_script_security
    router_id LVS_DEVEL
}

vrrp_script check_apiserver {
  script "/etc/keepalived/check_apiserver.sh"
  interval 3
  weight -2
  fall 10
  rise 2
}

vrrp_instance VI_1 {
    state <SLAVE_or_MASTER>
    interface <YOUR_IFNAME>
    virtual_router_id 151
    priority <CHOOSE DEPENDING ON THE NODE 255, 254, 253 >
    authentication {
        auth_type PASS
        auth_pass Password
    }
    virtual_ipaddress {
        <YOUR_IP_CIDR>
    }
    track_script {
        check_apiserver
    }
}
```
После настройки конфигурации необходимо запустить сервис *keepalived* на каждой control-plane ноде и убедиться, что он корректно работает:

```bash
systemctl enable --now keepalived
systemctl status keepalived
ip a
```

В выводе команды `ip a` на одной из нод должен появиться виртуальный IP-адрес (VIP), например:

```
inet 192.168.56.88/24 scope global secondary enp0s5
```
Это означает, что данная нода владеет виртуальным IP.

Далее, также необходимо конфигурировать балансировщик haproxy.
```bash
vim /etc/haproxy/haproxy.cfg
```
Желательно удалить или закоментировать настроки связанные с системными шифрами, на всякий случай
```
# utilize system-wide crypto-policies
ssl-default-bind-ciphers PROFILE=SYSTEM
ssl-default-server-ciphers PROFILE=SYSTEM
```
Также удаляются стандартные frontend/backend блоки, так как они не имеют отношения к нашей задаче. Вместо них описывается собственная конфигурация, ориентированная исключительно на проксирование API-сервера Kubernetes.
```
frontend apiserver
    bind *:8443
    mode tcp
    option tcplog
    default_backend apiserver

backend apiserver
    option httpchk GET /healthz
    http-check expect status 200
    mode tcp
    option ssl-hello-chk
    balance     roundrobin
        server control1 192.168.56.101:6443 check
        server control2 192.168.56.102:6443 check
        server control3 192.168.56.103:6443 check
```
HAProxy работает в режиме tcp. Kubernetes API использует HTTPS, но для балансировщика это всего лишь TCP-трафик — он не расшифровывает его и не вмешивается в содержимое. Это означает, что HAProxy остаётся максимально «прозрачным» и не требует настройки сертификатов. После настройки конфигурации запускаем сервис:
```bash
systemctl enable --now haproxy
```
Проверка на данном этапе сводится к проверке доступности виртуального IP:
```bash
ping 192.168.56.88
```
Это означает, что keepalived корректно назначил VIP на одну из нод, а HAProxy принимает соединения. При этом kube-apiserver на данном этапе ещё не запущен, поэтому проверка через curl пока не даст результата.

Далее поднимается сам кластер. В отличие от предыдущей конфигурации, теперь инициализация выполняется с учётом виртуального IP, через который в дальнейшем будет происходить весь доступ к control-plane.

На каждой control-plane ноде выполняется команда:
```bash
kubeadm init --control-plane-endpoint "192.168.56.88:6443" --upload-certs
```
Здесь есть два ключевых момента. Во-первых, в параметре `--control-plane-endpoint` указывается виртуальный IP, а не адрес конкретной ноды. Если этого не сделать, кластер инициализируется с IP текущей ноды, и вся дальнейшая схема с keepalived и HAProxy теряет смысл — при отказе этой ноды доступ снова будет потерян. 

Во-вторых, параметр `--upload-certs`. По умолчанию сертификаты control-plane остаются только на той ноде, где был выполнен `kubeadm init` и хранятся в `/etc/kubernetes/pki`. Это делает невозможным простое добавление новых control-plane нод — у них просто нет необходимых ключей для запуска тех же компонентов. Флаг `--upload-certs` меняет это поведение. kubeadm загружает сертификаты в кластер, в *etcd*, после чего другие control-plane ноды могут получить их при присоединении.  Благодаря этому параметру, в выводе команды `kubeadm init` появляется дополнительный `join` с параметром `--control-plane` и `--certificate-key` :
```bash
kubeadm join 192.168.56.88:6443 --token xynm1c.5w149edorr87kluq \
        --discovery-token-ca-cert-hash sha256:e3993dd17b95be4303b7695877c92a006a83e6de40846d1ae71e8fe94c91b6ee \
        --control-plane --certificate-key f6f38b663c7f59150cf8242101c0c57eb504cd2b5adfa72215f6910a066b8410
```
Выполним данную команду. При добавлении, на второй и третьей control-ноде можно отключить временно haproxy и keepalived, чтобы соединение с kube-apiserver произошло корректно, ведь join может пойти через VIP → HAProxy на ещё не готовый apiserver. Также можно установить плагин Calico для сети между подами.
```bash
kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml
```
Теперь можно добавлять worker-ноды:
```bash
kubeadm 192.168.56.88:6443 --token zfz61z.h12nu9a8849qblg5 --discovery-token-ca-cert-hash sha256:e3993dd17b95be4303b7695877c92a006a83e6de40846d1ae71e8fe94c91b6ee
```
Потребуется некоторое время пока разворачивается calico плагин. В конечном итоге, получим следующее состояние нод:
```
[root@control1 ~]# kubectl get nodes
NAME       STATUS   ROLES           AGE     VERSION
control1   Ready    control-plane   83m     v1.32.13
control2   Ready    control-plane   37m     v1.32.13

worker1    Ready    <none>          5m54s   v1.32.13
worker2    Ready    <none>          5m48s   v1.32.13
```
Высокодоступный кластер поднят! Теперь мы можем протестировать его работу. Для начала проверим на какой из control-нод находится VIP, для этого выполним команду `ip a` на каждой ноде:

## 8. Управление Настройками безопасности.

Перед тем как переходить к RBAC, важно понимать, как происходит обращение к кластеру. Все взаимодействие с Kubernetes выполняется через kube-apiserver, который представляет собой REST API. Любая команда kubectl в конечном итоге превращается в HTTP-запрос к API-серверу. Соединение устанавливается по HTTPS, поэтому весь трафик шифруется. При подключении клиент проходит аутентификацию — то есть API-сервер определяет, кто именно выполняет запрос. В случае с kubectl используются клиентские сертификаты из kubeconfig. Pod’ы внутри кластера используют другой механизм — ServiceAccount и связанный с ним токен. Несмотря на различия в способах аутентификации, результат один и тот же — API-сервер получает некоторую identity, от имени которой выполняется запрос. Далее вступает в силу авторизация. Она отвечает уже не на вопрос "кто ты", а на вопрос "что тебе можно". В Kubernetes для этого используется механизм RBAC, который определяет, какие действия разрешены для данной identity.

### 8.1 Настройка Прав Доступа для Pod'ов

Продемонстрируем, о чём говорилось в предыдущем абзаце. Для этого создадим простой Pod:

```bash
kubectl run mypod --image=curlimages/curl -- sleep 3600
kubectl get pod mypod -o yaml
```

В результате выполнения последней команды мы получим описание Pod’а. Обратим внимание на следующие строки:

```yaml
securityContext: {}
serviceAccount: default
serviceAccountName: default
```

Используется стандартный `ServiceAccount`. Это ожидаемо — никаких своих аккаунтов мы пока не создавали, поэтому все Pod’ы запускаются от имени `default`:

```bash
kubectl get sa
```

Теперь попробуем обратиться к API изнутри Pod’а:

```bash
kubectl exec -it mypod -- sh
curl https://kubernetes/api/v1 --insecure
```

Получим ответ:

```json
{
  "kind": "Status",
  "status": "Failure",
  "message": "forbidden: User \"system:anonymous\" cannot get path \"/api/v1\"",
  "code": 403
}
```

Запрос был выполнен без аутентификации, поэтому API воспринимает нас как `system:anonymous` и отклоняет его. Попробуем передать токен, который автоматически монтируется в Pod:

```bash
TOKEN=$(cat /run/secrets/kubernetes.io/serviceaccount/token)
```

И выполним запрос снова:

```bash
curl -H "Authorization: Bearer $TOKEN" https://kubernetes/api/v1 --insecure
```

Теперь API отвечает — мы успешно прошли аутентификацию. Можно, например, попробовать получить список Pod’ов:

```bash
curl -H "Authorization: Bearer $TOKEN" https://kubernetes/api/v1/namespaces/default/pods --insecure
```

Но получим ошибку:

```json
{
  "kind": "Status",
  "status": "Failure",
  "message": "pods is forbidden: User \"system:serviceaccount:default:default\" cannot list resource \"pods\"",
  "code": 403
}
```

Теперь ситуация другая. Мы уже не `anonymous`, а вполне конкретный пользователь `default:default`. То есть аутентификация прошла успешно, но доступ всё равно запрещён. Причина — отсутствие прав. Здесь вступает в силу RBAC. Все ресурсы в Kubernetes существуют в рамках пространств имён (namespaces). Это логическая изоляция: Pod’ы, Service’ы, Deployment’ы — всё создаётся внутри конкретного namespace. Посмотреть список можно так:

```bash
kubectl get ns
```

Наш Pod работает в `default`, и все запросы сейчас выполняются именно в этом пространстве, как раз namespace в данной нотации указан через `:` - `<namespace>:<account>`. 

RBAC строится вокруг трёх сущностей.
- **Role** — это набор правил, какие действия (`verbs`) разрешены и над какими ресурсами (`resources`). Например, “читать Pod’ы”.
- **ServiceAccount** — это идентификатор, от имени которого работает Pod. В отличие от клиентов (kubectl), где используется аутентификация через сертификаты, внутри кластера применяется более простой подход — токены. Этот токен автоматически монтируется в Pod и используется при обращении к API server.
- **RoleBinding** — связка между ними. Он определяет, какая Role применяется к какому ServiceAccount.

Продемонстрируем это на практике. Для этого создадим собственный ServiceAccount:

```bash
kubectl create sa mysa
```

Теперь создадим роль, которая позволит просматривать Pod’ы (как мы помним, в предыдущем примере просмотреть Pod'ы нам не удалось). Список доступных `verbs` и `resources` можно посмотреть через `kubectl create role -h`. Создадим роль:

```bash
kubectl create role myrole --resource=pods --verb=list
```

Далее свяжем созданную роль с нашим ServiceAccount. Для этого создаётся RoleBinding:

```bash
kubectl create rolebinding mybinding --role=myrole --serviceaccount=default:mysa
```

Перед созданием нового Pod’а сохраним его манифест в файл:

```bash
kubectl run newpod --dry-run=client -o yaml --image=curlimages/curl -- sleep 3600 > newpod.yaml
vim newpod.yaml
```

Добавим в секцию `spec` параметр `serviceAccountName`, указав созданный ранее аккаунт. В итоге манифест будет выглядеть следующим образом:

```yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: newpod
  name: newpod
spec:
  serviceAccountName: mysa
  containers:
  - args:
    - sleep
    - "3600"
    image: curlimages/curl
    name: newpod
  dnsPolicy: ClusterFirst
  restartPolicy: Always
```

Запустим Pod:

```bash
kubectl apply -f newpod.yaml
```

Теперь снова попробуем выполнить запрос, который ранее завершался ошибкой 403:

```bash
kubectl exec -it newpod -- sh
TOKEN=$(cat /run/secrets/kubernetes.io/serviceaccount/token)
curl -H "Authorization: Bearer $TOKEN" https://kubernetes/api/v1/namespaces/default/pods --insecure
```

В этот раз API возвращает список Pod’ов. Это означает, что запрос успешно прошёл не только аутентификацию, но и авторизацию — благодаря связке `Role` и `ServiceAccount`.

### 8.2 Роли Кластера

Важно обратить внимание на один момент. Всё, что мы сделали до этого, работает только в рамках одного namespace — `default`. Role и RoleBinding по своей природе ограничены пространством имён. Это означает, что даже если мы дали доступ к Pod’ам, этот доступ распространяется только на Pod’ы внутри конкретного namespace. Например, наш `mysa` не сможет посмотреть Pod’ы в `kube-system` или любом другом namespace.

Однако может потребоваться смотреть ресурсы во всех namespace или работать с cluster-wide ресурсами такими как ноды. В таких случаях используются **ClusterRole** и **ClusterRoleBinding**. ClusterRole — это та же самая роль, но которая действует ко всему кластеру. 

Рассмотрим какие ClusterRole уже существуют:
```bash
kubectl get clusterroles
```
Тут можно увидеть множество ролей, некоторые из которых создаются по-умолчанию, такие как `cluster-admin`, а некоторые создаются для сторонних дополнений типо плагина calico.

Все роли являются только разрешающими. Если хотя бы одна Role или ClusterRole предоставляет доступ к ресурсу, он будет разрешён независимо от других ролей.

### 8.3 Настройка Прав Доступа для Клиента

До этого момента мы работали с ServiceAccount — идентичностями внутри кластера. Однако при работе с Kubernetes извне (например, через kubectl) используется другой механизм. В Kubernetes не существует объектов “пользователь” в привычном смысле. Нет ресурса типа User, который можно создать командой kubectl. Вместо этого пользователь определяется на этапе аутентификации.

При подключении к кластеру клиент использует сертификат. API server проверяет его и извлекает из него имя пользователя. Таким образом, пользователь “не хранится” в кластере, а передаётся вместе с запросом. После успешной аутентификации Kubernetes применяет механизм авторизации — RBAC. Именно здесь уже задаётся, какие действия разрешены данному пользователю.

Ранее уже упоминалось, что при работе с Kubernetes используется namespace по умолчанию. Однако на практике параметров подключения больше: это не только namespace, но и сам кластер, а также пользователь, от имени которого выполняются запросы. Все эти параметры можно объединить в одну конфигурацию — так называемый context. Посмотреть доступные context’ы можно командой:
```bash
kubectl config get-contexts
```
Каждый из них представляет собой отдельный "профиль" для клиента, с определёнными правами и настройками. Переключение между ними позволяет, например, работать с разными кластерами или под разными пользователями, не меняя конфигурацию вручную в `~/.kube/`.

Рассмотрим создание пользователей на практике. Для начала создадим два namespace, они понадобятся в будущем:

```bash
kubectl create ns staff
kubectl create ns customers
```
Далее создадим пользователя на уровне операционной системы и войдём в сабшелл:
```bash
useradd -G wheel anna
passwd anna
su - anna
```
Сгенерируем приватный ключ и запрос на подпись сертификата:

```bash
openssl genrsa -out anna.key 2048
openssl req -new -key anna.key -out anna.csr -subj "/CN=anna/O=k8s"
```
Теперь подпишем сертификат с помощью центра сертификации Kubernetes:
```bash
sudo openssl x509 -req -in anna.csr -CA /etc/kubernetes/pki/ca.crt -CAkey /etc/kubernetes/pki/ca.key -CAcreateserial -out anna.crt -days 1800
```
После этого у нас появляется сертификат, с помощью которого kube-apiserver сможет аутентифицировать пользователя anna. Настроим клиент для работы с этим сертификатом. В качестве шаблона возьмём админский kubeconfig:
```bash
mkdir -p ~/.kube
sudo cp /etc/kubernetes/admin.conf ~/.kube/config
sudo chown anna:anna ~/.kube/config
```
Теперь заменим учётные данные с помощью `kubectl config`:
```bash
kubectl config set-credentials anna --client-certificate=anna.crt --client-key=anna.key
```
Если сейчас выполнить `kubectl get pods`, доступ всё ещё будет админским. Это связано не с сертификатом, а с тем, что используется админский контекст из `admin.conf`. Чтобы работать именно от имени пользователя anna, создадим отдельный контекст и переключимся на него:
```bash
kubectl config set-context anna-context --cluster=kubernetes --namespace=staff --user=anna
kubectl config use-context anna-context
kubectl config get-contexts
```
Теперь все запросы к API выполняются от имени пользователя anna. **Выйдем из пользователя anna** и вернёмся к **root**, чтобы настроить права доступа. Создадим роль в namespace staff:
```bash
kubectl create role tech_support --verb=get,list,watch,create,update,patch,delete --resource=pods -n staff
```
И свяжем эту роль с пользователем anna:
```bash
kubectl create rolebinding ts-to-staff --user=anna --role=tech_support -n staff
```
Теперь снова зайдём под пользователем anna и проверим доступ:
```bash
su - anna
kubectl create deployment testdep --image=nginx
```
В результате должны получить ошибку `deployments.apps is forbidden: User "anna" cannot create resource "deployments"`, - у пользователя нет прав на ресурс deployments. Роль разрешает работать только с pod'ами.
Создадим pod:
```bash
kubectl run testpod --image=busybox -- sleep 3600
```
Проверим:
```bash
kubectl get pods
```
Pod успешно создаётся, значит права применились. Но мы заметим только один под. А ранее создавалось ещё два. Они находятся в другом namespace Попробуем посмотреть pod'ы в другом namespace:
```bash
kubectl get pods -n default
```
Получим ошибку доступа, так как роль действует только в namespace staff. Имеет смысл добавить права на просмотр ресурсов в namespace default:
```bash
kubectl create role guests --verb=get,list,watch --resource=pods,deployments,replicasets -n default
kubectl create rolebinding ts-to-def --user=anna --role=guests -n default
```
Снова проверим под пользователем anna:
```bash
kubectl get pods -n default
```
Теперь доступ есть. Это демонстрирует, что права в Kubernetes задаются через RBAC и ограничиваются namespace, если используется обычная Role. К слову, хорошей особенностью namespace является то что при его удалении, удаляются и все ресурсы содержащиеся в этом пространстве имён. Поэтому теперь мы можем удалить весь созданный мусор одной командой:
```bash
kubectl delete ns staff
```

### 8.4 Контексты Безопасности

В первой части, в описании Pod можно было заметить параметр `securityContext`. Он отвечает за настройки безопасности Pod'a на уровне операционной системы: от какого пользователя запускается процесс, какие у него группы, а также дополнительные механизмы вроде SELinux или AppArmor.

`securityContext` может быть задан как на уровне всего Pod, так и на уровне отдельного контейнера. При этом настройки контейнера имеют приоритет, так как они более специфичны.

Для примера рассмотрим следующий Pod из официальной документации:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: security-context-demo
spec:
  securityContext:
    runAsUser: 1000
    runAsGroup: 1000
    fsGroup: 2000
    supplementalGroups: [4000]
  volumes:
  - name: sec-ctx-vol
    emptyDir: {}
  containers:
  - name: sec-ctx-demo
    image: busybox:1.28
    command: [ "sh", "-c", "sleep 1h" ]
    volumeMounts:
    - name: sec-ctx-vol
      mountPath: /data/demo
    securityContext:
      allowPrivilegeEscalation: false
```

Запустим данный Pod и зайдём внутрь контейнера:

```bash
kubectl apply -f security-context.yaml
kubectl exec -it security-context-demo -- sh
```

Попробуем создать файлы:

```bash
touch /data/demo/test.txt
touch /data/test.txt
```

Первая команда выполнится успешно, а вторая завершится ошибкой. Это связано с тем, что процесс внутри контейнера запущен не от root-пользователя, а с UID 1000, и у него нет прав на запись в `/data`.

Проверим текущего пользователя и права доступа:

```bash
id
ls -lahd /data
ls -lahd /data/demo
```

Можно заметить, что каталог `/data/demo` доступен для записи благодаря параметру `fsGroup`. Kubernetes автоматически меняет группу файловой системы для примонтированного тома, и процесс, входящий в эту группу, получает доступ на запись. В то время как `/data` остаётся недоступным, так как соответствующих прав у пользователя нет.

Таким образом, `securityContext` позволяет управлять правами доступа внутри контейнера и изолировать процессы, не давая им избыточных привилегий, например запускаться от root.

## 9. Запуск Приложений

В предыдущих разделах мы уже запускали Pod’ы и Deployment’ы, но не акцентировали на этом внимания. По сути всё, что мы запускали — это и есть приложения. Например, контейнер с nginx — это обычный веб-сервер, который работает внутри Pod’а. Kubernetes в данном случае просто управляет его запуском, перезапуском и размещением.

Однако запуск приложения в Kubernetes можно организовать разными способами. В зависимости от задачи используются разные типы ресурсов. Основные из них:

* Pod — самый простой вариант, одиночный экземпляр контейнера
* Deployment — управление ReplicaSet’ами и масштабированием stateless-приложений
* DaemonSet — запуск одного экземпляра на каждой ноде
* StatefulSet — запуск stateful-приложений с сохранением состояния

Каждый из этих ресурсов решает свою задачу и используется в разных сценариях. Далее рассмотрим их подробнее и разберём, в чём между ними разница и когда какой вариант применять.

### 9.1 Deployments

Deployment — это основной способ запуска приложений в Kubernetes. В отличие от одиночного Pod’а, который мы создавали ранее, Deployment управляет сразу несколькими Pod’ами и следит за их состоянием. Deployment поддерживает нужное количество Pod’ов, так называемых реплик. Replica — это просто копия Pod’а. Если указано, например, 3 реплики, это значит, что одновременно должно работать 3 одинаковых Pod’а с одним и тем же контейнером и конфигурацией.

Главная задача Deployment — обеспечить, чтобы всегда было запущено нужное количество Pod’ов. Если какой-то Pod упадёт или нода отвалится, Kubernetes автоматически создаст новый Pod, чтобы восстановить заданное количество реплик.

Ещё одна важная особенность Deployment — обновления. Если изменить образ контейнера, Deployment не просто убьёт старые Pod’ы и создаст новые, а сделает это постепенно, с помощью механизма RollingUpdate. Это позволяет обновлять приложение без простоя: часть Pod’ов работает на старой версии, часть — на новой, и постепенно происходит замена.

Создать Deployment можно, например, так:

```bash
kubectl create deploy mondeploy --image=nginx:1.17 --replicas=3
```

Проверим, что создалось:

```bash
kubectl get all --selector app=mondeploy
```

Можно изменить количество реплик:

```bash
kubectl scale deploy mondeploy --replicas=4
```

Или обновить версию образа контейнера:

```bash
kubectl set image deploy mondeploy nginx=nginx:latest
```

Таким образом, Deployment — это не просто “набор Pod’ов”, а механизм управления жизненным циклом приложения: запуск, масштабирование, обновление и восстановление после сбоев.

### 9.2 DaemonSets

DaemonSet — это ещё один способ запуска приложений в Kubernetes, похожий на Deployment, но с другим принципом работы. Если Deployment управляет количеством реплик (например, 3 или 5 Pod’ов), то DaemonSet работает иначе: он гарантирует, что **на каждой ноде кластера будет запущен ровно один Pod**. Количество реплик здесь не задаётся вручную — оно автоматически равно количеству нод в кластере. Если в кластер добавляется новая нода, DaemonSet сразу запускает на ней Pod. Если нода удаляется — соответствующий Pod тоже исчезает. Такой подход нужен, обычно, для служебных задач. DaemonSet используют в случаях, когда какой-то компонент должен работать **на каждой ноде**. Например, мониторинг, возмём - zabbix-agent, то тут для него такой вариант подходит как раз кстати.

По умолчанию DaemonSet не запускается на control-plane нодах, так как они обычно имеют taint. Если нужно запускать Pod’ы и там, необходимо добавить соответствующую toleration.

Создать манифест можно, например, на основе шаблона deploy:

```bash
kubectl create deploy daemon --image=nginx --dry-run=client -o yaml > daemon.yaml
vim daemon.yaml
```
Но только после этого нужно изменить `kind` на `DaemonSet` и убрать `spec.replicas`. В итоге при применении такого манифеста Kubernetes сам распределит Pod’ы по всем нодам.

Таким образом, DaemonSet — это способ гарантированно запустить одинаковый Pod на каждой ноде кластера, что делает его удобным для инфраструктурных и системных задач.

### 9.3 StatefulSets

Перед тем как рассматривать StatefulSet, важно понять разницу между stateless и stateful приложениями.

- stateless - приложение которое **не** зависит от состояния.
- stateful - приложение которое **зависит** от состояния.

На первый взгляд может показаться, что практически любое приложение хранит какое-то состояние, ведь большинство клиент-серверных программ написаны в парадигме событийно-ориентированного программирования - то есть как конечный автомат Мура имеют состояния. Но в контексте Kubernetes под “состоянием” понимаются только те данные, потеря которых влияет на работу конечного пользователя.

Простой пример stateless-приложения — веб-сервер nginx. Он обрабатывает HTTP-запросы и не хранит данные между ними. Если Pod с nginx перезапустится или будет пересоздан, пользователь этого не заметит — новый экземпляр просто продолжит принимать запросы. Такие приложения легко масштабировать и заменять.

Но бывает и иначе. Stateful-приложения зависят от данных, которые должны сохраняться между перезапусками. Например, если приложение на условном *php* хранит пользовательские сессии внутри контейнера (условно в `/var/lib/php/session`), то при перезапуске Pod’а эти данные исчезнут. Пользователь потеряет сессию, придётся заново логиниться, а несохранённые данные пропадут. Пользователь мог, например, редактировать статьи на своём сайте на Wordpress - при удалении сессий, всё что он написал не сохранится, и условный пользователь-редактор-газеты будет вынужден проделать дневную работу заново. Ещё более очевидный пример — базы данных. Если потерять их данные при перезапуске, то условный сайт на Wordpress просто перестанет работать корректно. Для таких приложений в Kubernetes используется StatefulSet.

Хотя stateful-приложение можно запустить и через Deployment, это плохая идея. Deployment не гарантирует сохранность данных, стабильные имена Pod’ов и порядок запуска. StatefulSet решает эти проблемы. StatefulSet, в отличие от Deploy имеет следующие качества для обеспечения сохранности состояния:
* каждый Pod получает **уникальное и стабильное имя** (например `app-0`, `app-1`). Это означает, что даже если Pod умрёт - после пересоздания он восстановится со старым именем.
* каждому Pod’у соответствует **свой постоянный том (PersistentVolume)**
* при перезапуске Pod получает **тот же самый том**, данные не теряются
* запуск и остановка происходят **последовательно** (сначала `app-0`, потом `app-1` и т.д.)
* обновления также выполняются **поочерёдно**
* при удалении StatefulSet сами тома **не удаляются автоматически**

Важно понимать, что сохранность данных — это не “магия StatefulSet”. За это отвечает механизм хранения — PersistentVolume и PersistentVolumeClaim. StatefulSet лишь гарантирует, перечисленные свойства, а их корректное использование остаётся за администратором.

Перед использованием StatefulSet обычно необходимо настроить StorageClass — это способ автоматически выдавать дисковое пространство для Pod’ов. Об этом будет подробнее рассказано в следующих главах. Вообще, задавать *mountVolume* для Pod'ов можно и без него, однако без автоматического выделения пространства, обьяснение StatefulSet теряет смысл, поэтому пока что возьмём самый простой StorageClass:

```bash
kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/master/deploy/local-path-storage.yaml
```

После этого можно проверить доступные классы хранения и созданные тома:

```bash
kubectl get storageclass
kubectl get pvc
kubectl get pv
```

Далее рассмотрим образец манифеста StatefulSet:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  ports:
  - port: 80
    name: web
  clusterIP: None
  selector:
    app: nginx
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  selector:
    matchLabels:
      app: nginx # has to match .spec.template.metadata.labels
  serviceName: "nginx"
  replicas: 3 # by default is 1
  minReadySeconds: 10 # by default is 0
  template:
    metadata:
      labels:
        app: nginx # has to match .spec.selector.matchLabels
    spec:
      terminationGracePeriodSeconds: 10
      containers:
      - name: nginx
        image: registry.k8s.io/nginx-slim:0.24
        ports:
        - containerPort: 80
          name: web
        volumeMounts:
        - name: www
          mountPath: /usr/share/nginx/html
  volumeClaimTemplates:
  - metadata:
      name: www
    spec:
      accessModes: [ "ReadWriteOnce" ]
      storageClassName: "local-path"
      resources:
        requests:
          storage: 64M
```

В данном примере появляется новый ресурс — **Service**. Service — это ресурс Kubernetes, который предоставляет сетевой доступ Pod'ам. Демонсет kube-proxy реализует Service, деплой CoreDNS даёт DNS-имя сервису. Посмотрим, какие Service уже существуют в кластере:

```bash
kubectl get svc -n default
```

Вывод будет примерно таким:

```text
NAME         TYPE        CLUSTER-IP   PORT(S)
kubernetes   ClusterIP   10.96.0.1    443/TCP
```

Это стандартный Service с именем `kubernetes`. Он создаётся автоматически и указывает на kube-apiserver. Важно понимать, что сам по себе Service — это просто описание (ресурс в API). Он не выполняет никаких действий. Когда Service создаётся, другие компоненты Kubernetes настраивают инфраструктуру под него - *kube-proxy* на нодах настраивает правила (iptables/ipvs), чтобы трафик на ClusterIP перенаправлялся на нужные Pod’ы, *CoreDNS* создаёт DNS-запись для имени сервиса. В результате чего внутри Pod’а можно обратиться к API-серверу по имени, как мы это и делали ранее: `curl https://kubernetes`. CoreDNS преобразует имя `kubernetes` в IP сервиса, а kube-proxy перенаправляет запрос на реальный endpoint (kube-apiserver). Pod’ы при этом не взаимодействуют друг с другом напрямую по IP, а используют Service как стабильную точку входа.

Обратим внимание на параметр нашего StatefulSet:

```yaml
clusterIP: None
```

Это так называемый **headless Service**. В обычном Service Kubernetes выдаёт один общий ClusterIP и распределяет трафик между Pod’ами. Однако для StatefulSet это не подходит, так как каждому Pod’у нужна собственная идентичность. Поэтому при использовании headless Service, Kubernetes не создаёт общий IP, вместо этого создаются DNS-записи для каждого Pod’а отдельно. В результате каждый Pod получает стабильное DNS-имя:

```
web-0.nginx
web-1.nginx
web-2.nginx
```

Это позволяет приложениям обращаться друг к другу напрямую, что особенно важно для stateful-систем (например, баз данных или кластеров).

Теперь обратим внимание на параметр:

```yaml
storageClassName: "local-path"
```

Здесь указывается StorageClass, который будет использоваться для автоматического выделения хранилища. Для каждого Pod’а StatefulSet будет создан отдельный PersistentVolumeClaim, а Kubernetes с помощью StorageClass автоматически создаст соответствующий PersistentVolume.

Итак, создадим наш StatefulSet:
```bash
kubectl apply -f statefulset-demo.yaml
kubectl get svc
kubectl get pvc,pv
kubectl get statefulset
kubectl get pods
```
Если вы довольно быстро выполнили последнюю команду, вы могли наблюдать как последовательно создаются Pod'ы. В отличии от Deploy, они создаются не сразу все, а каждый поочерёдно.

### 9.4 Одиночные Pod'ы

На ранних этапах мы уже запускали Pod’ы напрямую с помощью `kubectl run`. Это удобно для тестирования, однако в реальной работе такой подход почти не используется. Pod — это самая базовая единица в Kubernetes, но сам по себе он не управляется никаким контроллером. Это означает, что его жизненный цикл никак не контролируется. Если Pod будет удалён или упадёт, Kubernetes не создаст его заново. В отличие от Deployment или StatefulSet, здесь нет механизма, который следит за количеством запущенных экземпляров. Также отсутствует масштабирование. Один Pod — это всегда один экземпляр приложения. Если требуется несколько копий, их придётся создавать вручную, что не имеет практического смысла. Кроме того, нет механизма обновления. При изменении образа или конфигурации Pod нужно удалить и создать заново, что приводит к простою приложения. Наконец, отсутствует балансировка нагрузки. Pod не имеет встроенной точки входа, и без использования Service к нему нельзя удобно обращаться из других частей кластера.

По этим причинам отдельные Pod’ы используются только в вспомогательных задачах:
* для тестирования
* для отладки
* для разовых операций (например, выполнить команду внутри контейнера)

Во всех остальных случаях используются контроллеры более высокого уровня — такие как Deployment, DaemonSet или StatefulSet, которые управляют Pod’ами автоматически.

## 10 Многоконтейнерные Pod'ы

По умолчанию в Kubernetes используется модель “один контейнер — один Pod”. Это упрощает масштабирование и управление. Однако Pod может содержать несколько контейнеров, если они логически связаны и должны работать вместе. Важно понимать, что речь идёт не об отдельном типе ресурса. Многоконтейнерный Pod может быть частью Deployment, StatefulSet или любого другого контроллера. Это просто способ организации контейнеров внутри одного Pod.

Контейнеры внутри Pod’а:
* используют **общий network namespace** (один IP и localhost)
* могут общаться друг с другом напрямую через `localhost`
* могут использовать **общие volume** для обмена данными

Это позволяет строить шаблоны, где один контейнер дополняет другой. На практике выделяют несколько типовых сценариев.

**Sidecar** — дополнительный контейнер, который расширяет функциональность основного. Например, второй под может выполнять логирование. Он работает “рядом” с основным контейнером и разделяет с ним окружение.

**Ambassador** — контейнер-прокси, который отвечает за взаимодействие с внешними сервисами.
Например, может проксировать запросы к базе данных или внешнему API, скрывая детали подключения от основного приложения.

**Adapter** — контейнер, который преобразует данные основного приложения в нужный формат.
Например, приводит метрики или логи к стандартному виду.

Отдельно стоит упомянуть init-контейнеры. Это контейнеры, которые выполняются перед запуском основного приложения. Начиная с Kubernetes 1.29, init-контейнер с `restartPolicy: Always` фактически работает как sidecar, так как не завершается. С них, пожалуй и начнём.

### 10.1 Init-контейнеры

Продемонстрируем пример использования init-контейнеров:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
  labels:
    app.kubernetes.io/name: MyApp
spec:
  containers:
  - name: myapp-container
    image: busybox
    command: ['sh', '-c', 'echo The app is running! && sleep 3600']
  initContainers:
  - name: init-myservice
    image: busybox
    command: ['sleep', '20']
  - name: init-mydb
    image: busybox
    command: ['sleep', '10']
```

Создадим Pod:

```bash
kubectl apply -f init-demo.yaml
kubectl get pods
```

Сначала Pod будет в состоянии `Init:0/2` - это означает, что ни один из init-контейнеров ещё не завершился. Init-контейнеры выполняются **строго последовательно**:сначала запускается `init-myservice`, после его завершения запускается `init-mydb`. Когда первый контейнер завершится, статус станет `Init:1/2` - это означает, что один init-контейнер уже успешно отработал, и сейчас выполняется второй. После завершения всех init-контейнеров Pod перейдёт в состояние: `Running` И только в этот момент будет запущен основной контейнер `myapp-container`. Init-контейнеры используются для подготовки окружения перед запуском приложения. Пока они не завершатся успешно, основной контейнер даже не начнёт запускаться.

### 10.2 Sidecar-контейнеры

Теперь создадим Pod с sidecar контейнером:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-sidecar
spec:
  containers:
  - image: busybox
    name: test-foo
    volumeMounts:
    - mountPath: /messages
      name: messages-volume
    args: [ 'sh', '-c', 'echo hello > /messages/hello.txt && sleep 3600' ]
  - image: nginx
    name: test-bar
    volumeMounts:
    - mountPath: /usr/share/nginx/html
      name: messages-volume
  volumes:
  - name: messages-volume
    emptyDir:
      sizeLimit: 32Mi
```

Как мы видим, в параметре `containers` указано два контейнера — `test-foo` и `test-bar`. Они работают внутри одного Pod’а и используют общий volume.

Запустим Pod:

```bash
kubectl apply -f sidecar-demo.yaml
```

Проверим, создалось ли сообщение:

```bash
kubectl exec -it my-sidecar -c test-bar -- cat /usr/share/nginx/html/hello.txt
```

Результат `hello` - это означает, что контейнер `test-foo` создал файл, а контейнер `test-bar` смог его прочитать. Несмотря на то, что это разные контейнеры, они работают внутри одного Pod’а.

## 11. Автоматическое масштабирование (Autoscaling)

В Kubernetes мы можем управлять количеством Pod’ов у Deployment. Самый простой способ — вручную:

```bash
kubectl scale deployment mondeploy --replicas=5
```

Выполнив эту команду, через некоторое время количество реплик нашего деплоя должно стать равным пяти. Также у нас есть автоматическое масштабирование — **Horizontal Pod Autoscaler (HPA)**. HPA следит за потребляемыми ресурсами сервера и изменяет количество Pod’ов в зависимости от нагрузки, по следующему приципу:
* если CPU < target → уменьшает количество реплик до значения min
* если CPU ≈ target → ничего не делает
* если CPU > target → увеличивает количество реплик

HPA не работает сам по себе. Он должен узнавать фактическое потребление ресурсов - т.е. метрики. Метрики предоставляет предоставляет отдельный деплой **metrics-server**. Без него команда `kubectl top` не будет работать, и HPA просто не сможет принимать решения. Для начала установим *metrics-server*.

```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
```
В учебных целях отключим проверку TLS:
```bash
kubectl edit -n kube-system deployment metrics-server
```
И добавить в spec.template.spec.containers.args:

```yaml
- --kubelet-insecure-tls=true
```
Теперь мы можем смотреть нагрузку по кажлому Pod'у:
```bash
kubectl top pods -A
```
Итак, создадим Deployment с автоматическим масштабированием:

```bash
kubectl create deployment webstress --image=nginx
```
Создадим autoscaler:

```bash
kubectl autoscale deployment webstress --min=2 --max=5 --cpu=80%
```
Это значит, что будет создано минимум 2 Pod’а в случае минимальной нагрузки, и в случае максимальной нагрузки - 5 Pod'ов, ориентир-target нагрузки — 80%. Посмотрим состояние, и также рассмотрим yaml созданного hpa:

```bash
kubectl get hpa
kubectl get deployment webstress
kubectl get hpa webstress -o yaml
```
Как мы видим, значение реплик не изменилось, hpa показывает `<unknown>/80%`. При принятии решения о масштабировании Horizontal Pod Autoscaler использует два источника данных: фактическое потребление ресурсов - **metrics** и значения ресурсов, указанные в спецификации Pod - **requests**. Фактическое потребление - это среднее значение, полученное от `metric-server`. Ожидаемое значение - это среднее `resources.requests`. 

Количество необходимых реплик HPA вычисляет по формуле:
```txt
desiredReplicas = round( currentReplicas × (currentVal/(desiredValue × k) ) )
```
где, `currentVal` - фактическое потребление, `desiredValue` - ожидаемое потребление, `k` - коэффициент который мы задаём, например как было ранее через `--cpu=80%`, `currentReplicas` - текущее количество реплик.
Для примера, зададим нашему деплою значения ресурсов
```bash
kubectl set resources deploy webstress --requests=cpu=20m
kubectl get hpa webstress
kubectl get pods --selector='app=webstress'
```
Теперь у нас количество реплик стало 2. Проанализируем почему так произошло.
```bash
kubectl top pods --selector='app=webstress'
```
Наши Pod'ы ничего не потребляют. Следовательно текущее среднее значение CPU собранное метриками равно 0m, ожидаемое среднее равно 20m, коэффцицент 0.8 то есть получается 
```txt
0÷(20×0.8) = 0
```
Значение меньше 1±0.1 поэтому масштабировать нечего. Попробуем создать нагрузку, представим что наш сайт стал популярным, и на него заходят множество посетителей:
```
kubectl expose deploy webstress --port=80
APP_IP=$(kubectl get svc webstress --no-headers | awk '{print $3}')
while true ;
  do curl -s -o /dev/null http://$APP_IP ;
done &
kubectl top pods --selector='app=webstress'
```
Как мы видим, нагрузка в первые же секунды поднялась до 24m и 32m, в моём случае. Следовательно:
```txt
( (24+32)/2 ) ÷ (20×0.8) = 28 / 16 = 1.75 ≈ 2
```
Таким образом kube-controller-manager увеличит количество реплик в два раза. Итоговое количество реплик должно получиться 4. Важно отметить что если итоговое количество реплик получается больше чем `--max`, то масштабирование не выходит за рамки максимального значения, а если соотношение `current/desired` близко к единице масштабирование в большую сторону не выполняется. Существует ситуация "дребезжания" - ситуация, когда количество реплик постоянно изменяется из-за незначительных колебаний. Например, если целевое значение CPU составляет 50%, а текущая загрузка колеблется между 49% и 51%, то без дополнительной защиты HPA мог бы непрерывно увеличивать и уменьшать количество Pod'ов, создавая бессмысленную нагрузку. Чтобы этого избежать у ресурса `hpa`, предусмотрен параметр `spec.behavior.scaleUp.tolerance` - допустимое отклонение от единицы (в процентах). По умолчанию этот параметр равен 10%.


Кроме того, чтобы дополнительно избежать "дребезжания" уменьшение происходит не сразу. Kubernetes ждёт примерно 5 минут перед тем, как уменьшить количество Pod’ов. Это поведение можно изменить двумя способами. Первый способ — на уровне конкретного HPA, через тот же `spec.behavior`:

```bash
kubectl edit hpa webstress
```
И добавляем:

```yaml
spec:
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 30
```
Это означает, что именно для этого HPA ожидание перед уменьшением будет 30 секунд. Второй способ — глобально, редактируя сам `kube-controller-manager`:

```bash
sudo vim /etc/kubernetes/manifests/kube-controller-manager.yaml
```
Добавляем в `spec.template.spec.containers.args` параметр:

```yaml
- --horizontal-pod-autoscaler-downscale-delay=30s
```

После сохранения static Pod автоматически перезапустится. Настройка в kube-controller-manager — это **глобальное значение по умолчанию** для всего кластера. Она применяется ко всем HPA, у которых не задано собственное поведение. Настройка внутри самого HPA (`spec.behavior`) — это **более приоритетный (override)** вариант. Если он задан, то именно он используется, а глобальная настройка игнорируется для этого конкретного autoscaler’а. На практике почти всегда удобнее управлять через HPA, потому что можно гибко настраивать разные приложения по-разному, не трогая весь кластер.

Некоторые приложения могут создавать значительную нагрузку при запуске, однако на "рабочих оборотах" потреблять незначительно. Например JVM может компилировать классы сразу после запуска, и потреблять порядка 800m CPU. Если у нас в этот момент сработал бы HPA - он бы заметил 800% utilization, и срочно создал бы ещё 100 Pod, хотя это просто старт приложения. Чтобы избежать подобной ситуации HPA не всегда учитывает метрики недавно запущенных Pod. Для этого используется параметры `kube-controller-manager`:
```text
--horizontal-pod-autoscaler-cpu-initialization-period=5m
--horizontal-pod-autoscaler-initial-readiness-delay=30s
```
Первый параметр игнорирует значение метрик CPU, если Pod имеет статус NotReady. Таким образом, пока приложение проходит инициализацию оно не будет масштабироваться. Второй параметр задаёт время после запуска Pod, в течение которого HPA может считать его "ещё не полностью готовым" и полностью игнорировать его метрики - это необходимо для предотвращения масштабирования Pod'ов, которые часто "моргают" в состояниях Ready/NotReady при запуске. Для Pod'ов которые сразу получили статус Ready и не меняли его - метрики будут собираться игнорируя этот интервал. Но если Pod переключился из статуса Ready в NotReady, то для такого Pod'а масштабирование происходить не будет пока не пройдёт 30 секунд начиная с его создания.

Помимо всего прочего, масштабирование может происходить и без заданного `requests`. Мы могли заметить следующие параметры нашего yaml манифеста HPA:
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: webstress
spec:
  maxReplicas: 8
  metrics:
  - resource:
      name: cpu
      target:
        averageUtilization: 80
        type: Utilization
    type: Resource
  minReplicas: 2
```
Если мы зададим ожидаемое потребление сразу абсолютных значениях, то логично что значения `requests` нам не нужны. Для этого используется target типа `AverageValue`. Соответственно если мы явно установим`averageValue: 30m` - то вместо `20×0.8` у нас сразу будет подставлено 30, игнорируя реквесты.

Также, тут полезно рассмотреть параметр `type: Resource` - он отвечает за то, с какими метриками будет сравниваться значение реквестов. Стандартное `Resource` значит что HPA использует встроенные ресурсы Kubernetes (cpu или memory) как источник метрик. Однако в общем случае, мы можем в качестве ориентира брать и показатели из других источников, например вообще с какого-нибудь произвольного prometeus. И строго говоря, эти показатели не обязательно должны быть именно вычислительными ресурсами, может быть вообще произвольная метрика, например, пакеты в секунду, IOPS, или вообще уровень заполнения воды в зумпфе карьера. Всё это сравнивается с `requests` в нашем деплое.

Если для некоторых Pod не удалось получить метрики, то перед принятием решения выполняется предварительный расчёт на основании существующих метрик. Если по предварительному расчёту необходимо **уменьшение** количества реплик, HPA считает, что Pod без метрик потребляют ресурсы на уровне ожидаемого значения. Это уменьшает вероятность слишком агрессивного scale down. Если же предполагается **увеличение** количества реплик, то HPA считает, что Pod без метрик не потребляют ресурсов вовсе, что также уменьшает вероятность чрезмерного scale up. Задавшись значениями для отуствующих метрик, производится повтороне вычисление. После повторного вычисления коэффициента, HPA сравнивает новый результат с исходным. Если после учёта отсутствующих метрик и недавно запущенных Pod **направление масштабирования изменилось**, то масштабирование не выполняется. Таким образом, HPA предпочитает отказаться от масштабирования, чем принять потенциально ошибочное решение.

Если для Horizontal Pod Autoscaler указано несколько метрик одновременно, например `--cpu` и `--memory`, то HPA вычисляет требуемое количество реплик для каждой метрики отдельно, после чего выбирает наибольшее полученное значение. Такой подход гарантирует, что приложение будет масштабироваться по наиболее критичному ресурсу. Например, если по CPU достаточно и 3 реплики, а по памяти необходимо смасштабировать до 8 реплик, то итоговое количество реплик будет всё равно 8. Если при использовании нескольких метрик одна из них полностью недоступна ни для одного Pod, то HPA не выполняет уменьшение количества реплик вообще, даже если остальные реплики этого требуют. Однако увеличение количества реплик по-прежнему допускается. Это сделано для того, чтобы ошибка получения метрик не привела к случайному уменьшению ресурсов приложения.

Для ограничения скорости масштабирования HPA поддерживает `policies`, которые задают максимально допустимое изменение количества реплик за определённый промежуток времени. Политика может задаваться либо в абсолютном количестве Pod `type: Pods`, либо в процентах от текущего числа реплик `type: Percent`.

Например:

```yaml
behavior:
  scaleDown:
    selectPolicy: Max
    policies:
    - type: Pods
      value: 4
      periodSeconds: 60
    - type: Percent
      value: 10
      periodSeconds: 60
```

В данном случае HPA может уменьшить не более 4 Pod за минуту или не более 10% текущих Pod за минуту. Если задано несколько политик одновременно, то по умолчанию используется политика, допускающая наибольшее изменение количества реплик (`selectPolicy: Max`). Это означает, что HPA выбирает наиболее агрессивный вариант масштабирования. Значение `Min` заставляет HPA выбирать наиболее консервативную политику, а значение `Disabled` полностью запрещает масштабирование в данном направлении. Аналогичные политики могут быть заданы как для `scaleDown`, так и для `scaleUp`.

## 12. Helm и Kustomize

### 12.1 Helm

Helm — это пакетный менеджер для Kubernetes, который позволяет устанавливать приложения как готовые пакеты (charts). Chart — это по сути набор шаблонов Kubernetes-манифестов плюс файл значений values.yaml. В этих шаблонах используются переменные, а конкретные значения подставляются из values.yaml, за счёт чего один и тот же chart можно переиспользовать с разной конфигурацией. Обычно работа с Helm начинается с добавления репозитория, в котором лежат charts. Скачать heml можно из [официального репозитория](https://github.com/helm/helm). Например, добавим популярный репозиторий *Bitnami* и обновим список пакетов:

```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
```

После этого можно искать нужные приложения:

```bash
helm search repo nginx
```

И установить, например, nginx:

```bash
helm install my-nginx bitnami/nginx
```

Здесь `my-nginx` — это имя release, то есть конкретной установленной версии приложения в кластере, а `bitnami/nginx` — это сам chart. Важно понимать разницу: chart — это шаблон, а release — это результат установки. Один и тот же chart можно установить несколько раз под разными именами, и это будут независимые экземпляры.

Проверить, что установилось:

```bash
helm list
kubectl get all
```

Посмотреть подробности:

```bash
helm status my-nginx
```

Удалить:

```bash
helm uninstall my-nginx
```

Теперь самое важное. У любого chart’а есть values.yaml с дефолтными значениями, его можно посмотреть так:

```bash
helm show values bitnami/nginx
```

Там будет большой YAML, где задаются параметры вроде количества реплик, типа сервиса и т.д. Эти значения используются внутри шаблонов chart’а. На практике ты почти никогда не используешь дефолт как есть, а создаёшь свой values.yaml:

```yaml
replicaCount: 3
service:
  type: NodePort
```

И устанавливаешь chart с этими значениями:

```bash
helm install my-nginx bitnami/nginx -f values.yaml
```

Здесь `-f` (или `--values`) говорит Helm взять твой файл и поверх дефолтных значений применить изменения. Это основной и нормальный способ конфигурации. Есть ещё вариант через `--set`, например:

```bash
helm install my-nginx bitnami/nginx --set replicaCount=3 --set service.type=NodePort
```

Он делает то же самое, но через CLI, и быстро становится неудобным, если параметров много или структура вложенная, поэтому его используют в основном для быстрых тестов. После установки можно посмотреть, какие значения реально применились:

```bash
helm get values my-nginx
helm get values --all my-nginx
```

Если нужно изменить конфигурацию — используется upgrade:

```bash
helm upgrade my-nginx bitnami/nginx -f values.yaml
```

Важно, что при upgrade нужно снова передавать values.yaml, иначе Helm возьмёт дефолтные значения и настройки потеряются. Ещё одна полезная команда — `helm template`. Она рендерит chart в обычный Kubernetes YAML, но при этом ничего не применяет в кластер, то есть ты можешь посмотреть, что именно будет создано:

```bash
helm template my-nginx bitnami/nginx -f values.yaml > nginx.yaml
```

После этого можно уже руками применить:

```bash
kubectl apply -f nginx.yaml
```

### 12.2 Kustomize

Представим ситуацию: у нас есть YAML-файл и нам необходимо создать несколько "версий" Pod'а, отличающихся лишь некоторыми параметрами. Для этого в kubernetes сущуествует способ не копироать YAML-файлы заново при каждом изменении, а накладывать изменения в виде патчей. Вместо того чтобы копировать и редактировать манифесты для каждого окружения, мы берём уже готовый YAML и описываем, что именно в нём нужно изменить: image, labels, namespace, ресурсы или другие поля. Эти изменения описываются отдельно в специальном файле `kustomization.yaml`, который работает как “инструкция по модификации” исходных манифестов. В результате Kubernetes применяет не исходный YAML напрямую, а уже собранную версию с наложенными изменениями. Продемонстрирем как это работает. Создадим  пробный Deploy:
```bash
kubectl create deploy test-dep --image=nginx --dry-run=client -o yaml > test-dep.yaml
vim test-dep.yaml
```
Получили стандартный yaml c одной репликой:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  creationTimestamp: null
  labels:
    app: test-dep
  name: test-dep
spec:
  replicas: 1
  selector:
    matchLabels:
      app: test-dep
  strategy: {}
  template:
    metadata:
      creationTimestamp: null
      labels:
        app: test-dep
    spec:
      containers:
      - image: nginx
        name: nginx
        resources: {}
status: {}

```
Теперь разберём как нам написать `kustomization.yaml` файл. Он состоит из двух основных частей, первая представляет собой список файлов которые мы будем патчить, а вторая описывает модифицирующие действия. Первая часть называется `resources:`. В данном случае укажем в ней наш test-dep.yaml. Во второй части могут быть применены различные модификаторы, подробнее каждый из которых будет рассмотрен ниже. Например, наш `kustomization.yaml` может выглядеть так:
```yaml
resources:
  - test-dep.yaml
patches:
  - target:
      kind: Deployment
      name: test-dep
    patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: test-dep
      spec:
        replicas: 3
```
Посмотреть что произойдёт после его применения можно следующей командой:
```bash
kubectl apply -k . --dry-run=client -o yaml
```
Как можно заметить из вывода, Deploy будет иметь три реплики. Команда похожа на `kubectl apply -f` но вместо буквы "f" тут "k" - "kustomize", и собственно в качестве аргумента указывается не сам kustomize.yaml, а каталог в котором он находится.

В этом примере, в качестве модифицироующей части использовался `patches:`, который вносит точечные изменения в файл. Он состоит из списка "патчей", каждый из которых имеет `target` - то есть обьект к которому будет применяться патч, в данном случае это "Deployment/test-dep", и собственно секция `patch` в которой содержится сам тескт того что необходимо изменить. Если мы попробуем изменить параметр `metadata.name` - у нас ничего не получится. Дело в том что в данном случае наш патч **не изменяет** просто содержимое yaml-файла, как это могло показаться. В данном примере производится так называемый  "*Strategic Merge Patch (SMP)*", о котором подробнее будет рассказано позже, т.е. он способен изменить только всё то что можно сделать при помощи `kubectl scale` или `kubectl set image` и других уже известных модификаторов. 

Всего существует четыре механизма kustomize:
- (A) Resource composition - композиция ресурсов, "склеивает" несколько yaml в один файл.
- (B) Transformers - глобальные изменения. При помощи него можно изменять метаданные, лейблы, префиксы имени, пространства имён.
- (C) Patches - точечные изменения.
- (D) Resource Generator - создание ресурсов.

Рассмотрим механизм композиции. Например, создадим для нашего деплоя service:
```bash
kubectl create svc nodeport test-dep-svc --tcp='80:80' --node-port=30080 --dry-run=client -o yaml > test-dep-svc.yaml
```
В файле `kustomization.yaml`, удалим всё кроме секции `resources:`, в которую добавим наш service:
```yaml
resources:
  - test-dep.yaml
  - test-dep-svc.yaml
```
Проверим что получится:
```bash
kubectl apply -k . --dry-run=client -o yaml
```
Как видим, в результате получился файл в котором содержится и deployment и service. Таким образом, всё что указано в секции `resource` объединяется в один yaml.

Далее, к глобальным параметрам относятся следующие секции:
```yaml
namespace: my-namespace
namePrefix: dev-
nameSuffix: "-001"
labels:
  - pairs:
      app: bingo
    includeSelectors: true 
commonAnnotations:
  oncallPager: 800-555-1212
```
Данные модификаторы действуют на все yaml указанные в `resources`, например, следующий `kustomization.yaml` добавим префикс "kustom-" в названия delpoy и services:
```yaml
resources:
  - test-dep.yaml
  - test-dep-svc.yaml
namePrefix: kustom-
```
Проверить это можно следующей командой:
```bash
kubectl apply -k . --dry-run=client -o yaml
```

Теперь рассмотрим точечные изменения. За них отвечает секция `patches`, которая использовалась в первом примере. В отличии от глобальных модификаций, каждиый "патч" состоит из двух параметров, и параметр `target` - отвечает конкретно какой ресурс, или группа ресурсов будет попадать под модификацию. Таким образом, мы можем изменить содержимое только нашего deployment'a. Второй параметр патча может быть `patch: |-` - для inline текста, или `path: filename.yaml` - для текста в отдельном файле. Сам текст патча принимается в двух форматах: **strategicMerge** и **Json6902**. StrategicMerge был продемонстрирован в самом первом примере - он сопостовляет поля по `name` и обновляет их. Собственно, возниакет вопрос, а что делать если нам необходимо изменить название контейнера? Каким образом strategicMerge поймёт что нужно сделать, добавить новый sidecar-контейнер, или изменить имя старого? В таком случае, предусмотрены JSON-патчи. Чтобы найти правильный ресурс для патча Json6902. Например продемонстрируем следующий Json6902-патч:
```yaml
resources:
  - test-dep.yaml
  - test-dep-svc.yaml
namePrefix: kustom-
patches:
  - target:
      kind: Deployment
      name: test-dep
    patch: |-
      - op: replace
        path: /metadata/name
        value: aboba
```
Применим и посмотрим что получится:
```bash
kubectl apply -k .
kubectl get deploy,svc
```
В итоге будут созданы следующие ресурсы:
```txt
service/kustom-test-dep-svc
deployment.apps/kustom-aboba
```
Как мы видим, ресурс у наз назван не `aboba` как ожидалось, а `kustom-aboba`, потому что у нас также применён префикс на глобальном уровне.

На практике, гораздо чаще используют четвёртый механизм - генерация ресурсов. Для него изначально и был придуман kustomize. В kubernetes есть такой ресурс как `configMap`, он содержит заготовленные файлы конфигурации для различных служб. Выглядит он примерно так:
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  nginx.conf: |
    server {
        listen 80;
    }
```
То есть фактически такой файл состоит из двух частей: название и произвольные данные, в общем случае даже не обязательно конфигурационные. Благодаря нему, например мы можем автоматически загружать уже готовый index.html в различные Pod'ы, или даже задавать переменные среды. Просто, самое целесообразное применение этому именно конфигурация, поэтому он назван *configMap*.

Собственно, усть геренации, в том чтобы создать ресурс из заготовленных данных, файлов, литералов или переменных среды. Вместо того чтобы писать:
```bash
kubectl create cm nginx-config --from-file=nginx.conf --dry-run=client -o yaml
```
Мы можем создать следующий `kustomization.yaml` файл:
```yaml
resources:
  - test-dep.yaml
  - test-dep-svc.yaml
namePrefix: kustom-
patches:
  - target:
      kind: Deployment
      name: test-dep
    patch: |-
      kind: Deployment
      metadata:
        name: test-dep
      spec:
        template:
          spec:
            containers:
            - name: nginx
              volumeMounts:
                - mountPath: /etc/nginx/sites-avalible/default
                  name: default-site
            volumes:
            - name: nginx-conf
              configMap:
                name: nginx-conf
configMapGenerator:
  - name: nginx-conf
    files:
      - nginx.conf
```
И в результате kustomize он будет добавлен к resources:
```bash
kubectl apply -k . --dry-run=client -o yaml
```
Как можно заметить, произошла магия - в названии сгенерированного configMap автоматически добавился суффикс `nginx-conf-t76h69d47t`, хотя оригинальное название было просто `nginx-conf`, при чём этот суффикс был также автоматически добавлен в `spec.volumes`. Kustomize применяет встроенные reference rules, "если этот ConfigMap сгенерирован мной  -> значит я могу установить его alias". Суффикс с хэшем позволяет избежать конфликтов имён: даже если в кластере уже существует ConfigMap с базовым именем, Kustomize создаёт уникальный ресурс. Однако это добавляет возможные конфликты при ручном управлении ресурсами вне Kustomize. Если ConfigMap создаётся через Kustomize, а затем вручную создаётся ресурс с тем же базовым именем, это не гарантирует совместимость. Pod’ы всегда ссылаются на итоговое имя с хэшем, поэтому любые ручные копии ConfigMap не будут автоматически подхватываться. Чтобы отключить это поведение, можно использовать опцию `disableNameSuffixHash: true`:
```yaml
configMapGenerator:
- name: example-configmap-3
  literals:
  - FOO=Bar
generatorOptions:
  disableNameSuffixHash: true
  labels:
    type: generated
  annotations:
    note: generated
```
Но лучше не изменять ConfigMap, созданный через Kustomize, вручную. Также, в этом примере продемонстрировано, что вместо секции `files`, можно также указать `literals` - т.е. произвольные данные и `envs`. Подробнее об этом будет раскрыто в главе про configMap.

Обычно kuztomize не разбрасывают по всему каталогу как мы только что делали. Вместо этого файлы упорядочивают в структуру базовых каталогов и наслоений. Выглядит эта структура следующим образом:

```txt
- ./base
     |- test-cm.yaml
     |- test-svc.yaml
     |- test-pod.yaml
     \._ kustomization.yaml
- ./overlays
    |- dev
    |    \._ kustomization.yaml
    \._ prod
         \._ kustomization.yam|
```
Файл `base/kustomization.yaml`, как правило содержит список *resources*, которые генерируются для каждого наслоения. Например, создадим обычный манифест:

```bash
mkdir kustomize-demo
cd kustomize-demo
mkdir base
mkdir -p overlays/dev
mkdir overlays/prod
vim base/test-pod.yaml
```
Допустим, со следующим содержианием:
```yaml
apiVersion: v1
kind: Pod
metadata:
  creationTimestamp: null
  labels:
    run: test-pod
  name: test-pod
spec:
  containers:
  - args:
    - sleep
    - "3600"
    image: busybox
    name: test-pod
    resources: {}
  dnsPolicy: ClusterFirst
  restartPolicy: Always
status: {}
```
Файл `base/kustomization.yaml` будет содержать только наш манифест:

```yaml
resources:
  - test-pod.yaml
```
Теперь добавим overlay для *dev* `overlays/dev/kustomization.yaml`:
```yaml
resources:
  - ../../base
namePrefix: dev-
```
И overlay для *prod* - `overlays/prod/kustomization.yaml`:
```yaml
resources:
  - ../../base
namePrefix: prod-
```
Создадим поды:
```bash
kubectl apply -k overlays/dev
kubectl apply -k overlays/prod
kubectl get pods
```
Не смотря на то что в *overlays/dev/kustomization.yaml* указан только каталог *base*, Kustomize рекурсивно включает все ресурсы и применяет поверх них дополнительные модификации.

## 13. Управление Планированием

Когда в Kubernetes создаётся Pod, он не сразу попадает на какой-то узел. Сначала он появляется в состоянии Pending, потому что у него ещё нет назначенного node. Дальше в дело вступает scheduler — компонент, который решает, на каком узле этот Pod должен запуститься. Процесс планирования идёт в несколько этапов. Сначала scheduler ищет подходящие узлы, то есть такие, которые вообще могут запустить этот Pod. На этом этапе отсеиваются ноды, которые не подходят: например, не хватает CPU или памяти, не совпадают taints/tolerations, не удовлетворяются nodeSelector или affinity-правила. Это называется фильтрация (feasible nodes). После этого из оставшихся узлов scheduler выбирает лучший. Для этого он каждому узлу выставляет “оценку” (score) на основе разных факторов, например равномерность нагрузки или предпочтения (preferred rules). Узел с самым высоким score считается наиболее подходящим.

Когда узел выбран, происходит процесс, который называется binding — scheduler сообщает API Server, что этот Pod должен быть запущен на конкретном node. После этого в объекте Pod появляется поле `nodeName`. Если на каком-то этапе что-то пошло не так (например, вообще нет подходящих узлов), Pod так и остаётся в состоянии Pending, и это первый сигнал, что есть проблема с планированием.

После того как scheduler принял решение, управление переходит к kubelet на выбранной ноде. kubelet видит, что ему назначен новый Pod, и начинает его запуск. Для этого он через CRI (container runtime interface) скачивает нужный образ контейнера (например, из Docker Hub), сохраняет его локально на ноде и затем создаёт и запускает контейнер.

В итоге цепочка выглядит как-то так:
```ASCII
CONTROL:                        (4)               WORKER NODE:
                       ,-----------------------------------. 
[ kubectl create ... ] v                                   |
     |   +---------------+ (3) +---------------+    +------+------+
     `-->|   api-server  | <---+   scheluder   |    +   kubelet   |
         +-------+-------+     +---------------+    +------+------+
             (1) |                (2)  ^                   | (5)
                 v                     |                   v
         +-----------------------------+------+     +-------------+
         |               etcd                 |     |     cri     |
         +------------------------------------+     +-------------+
```

Управление планирование — это способы повлиять на то какие узлы считаются подходящими и какой узел получит больший приоритет. С этого и начнём.

### 13.1 NodeSelector

Самый тривиальный способ повлиять на планировщик — это параметр `spec.nodeSelector`. Он позволяет сказать: “этот Pod можно запускать только на нодах с определённым label”. Механика простая: на нодах есть labels (ключ-значение), и Pod в своём spec указывает, какие labels ему подходят. Scheduler при фильтрации узлов просто отбрасывает все, которые не совпадают.

Сначала повесим label на ноду:

```bash
kubectl label nodes worker2 disktype=ssd
kubectl get nodes --show-labels
```

Теперь создаём Pod, который требует именно ноды c лейблом `ssd`:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: selector-pod
spec:
  containers:
  - name: nginx
    image: nginx
    imagePullPolicy: IfNotPresent
  nodeSelector:
    disktype: ssd
```

Применяем и смотрим на какую ноду он попал:
```bash
kubectl apply -f selector-pod.yaml
kubectl get pod selector-pod -o wide
```

А ни на какую ноду он не попал. Если **нет ни одной ноды**, которая подходит под `nodeSelector`, Pod останется в состоянии Pending. Теперь разрешим планировщику размещать Pod'ы на второй ноде:

```bash
kubectl uncordon worker2
kubectl get pod selector-pod -o wide
```
Как мы видим, спустя некоторое время Pod создаётся именно на этой ноде. Убрать лейбл можно следующей командой:
```bash
kubectl label nodes worker2 disktype-
```

Таже отдельно есть поле `spec.nodeName`:
```yaml
spec:
  nodeName: worker1
```
Оно жёстко указывает, на какой ноде запускать Pod, вообще без участия scheduler. Это почти всегда плохая идея, потому что если нода недоступна или заcordon’ена — Pod просто никогда не запустится.

### 13.2 Affinity и Anti-Affinity

Если `nodeSelector` — это простой фильтр “ключ=значение”, то affinity — это более гибкий и мощный механизм, который позволяет задавать сложные правила планирования. Есть два основных типа Affinity:
* **nodeAffinity** — работает на основании labels нод
* **podAffinity / podAntiAffinity** — планирование на основании уже запущеных Pod - рядом с кем можно или нельзя запускаться.

Начнём с nodeAffinity, это практически расширенная версия nodeSelector. Если ранее наш селектор выглядел следующим образом:

```yaml
nodeSelector:
  disktype: ssd
```

То теперь можно писать более сложные условия через `spec.affinity.nodeAffinity`. Пример:

```yaml
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: type
          operator: In
          values:
          - blue
          - green
```
Здесь `key: type` — label на ноде, `operator: In` — оператор выбора, означает что значение должно быть в списке, `values: [blue, green]` — допустимые значения для этого оператора. То есть Pod запустится только на нодах, где есть label `type=blue` или `type=green`. 

Существует также два режима Affinity:
* requiredDuringSchedulingIgnoredDuringExecution - Это жёсткое правило. Если ни одна нода не подходит — Pod не запустится (останется Pending). Это аналог nodeSelector, только мощнее.
* preferredDuringSchedulingIgnoredDuringExecution - Это мягкое правило. Scheduler попытается его выполнить, но если не получится — всё равно запустит Pod. 
Название `IgnoredDuringExecution` - означает, что правило обязательно учитывается при выборе ноды, но после запуска Pod’а изменения лейблов ноды больше не влияют на уже запущенный Pod. Важно понимать ограничение affinity применяется **только в момент планирования**. Если Pod уже запущен — он не будет автоматически переезжать, даже если условия изменились. Kubernetes не делает “рескейджулинг” по affinity.

Также есть разные операторы:
* `In` — значение входит в список
* `NotIn` — не входит
* `Exists` — label просто существует (значение не важно)
* `DoesNotExist` — label отсутствует

Логика здесь такая, правило affinity может состоять из нескольких независимых наборов условий. Таким образом **nodeSelectorTerms** - это OR-логика, т.е. набор альтернативных “групп условий”. А отдельный **matchExpressions внутри каждого term** - это AND-логика, набор условий внутри одной группы.

То есть структура выглядит так:

```yaml
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:      # Элемент списков №1 (Условие А)
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["zone-A"]
      - matchExpressions:      # Элемент списка №2 (Условие Б) -> Между ними логика OR
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["zone-B"]
# Эквивалентно: (Находится в zone-A) OR (Находится в zone-B)
```

А внутри каждого term:

```yaml
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:      # Один term
        - key: disktype        # Элемент массива №1 (Условие А)
          operator: In
          values: ["ssd"]
        - key: environment     # Элемент массива №2 (Условие Б) -> Между ними логика AND
          operator: In
          values: ["production"]
# Эквивалентно: (disktype == ssd) AND (environment == production)
```

Подобная логика используется во многих объектах Kubernetes, например в labelSelector, affinity, NetworkPolicy и других API-ресурсах использующих matchExpressions.

Рассмотрим инимальный практический пример nodeAffinity. Сначала повесим label:

```bash
kubectl label nodes worker1 type=good_pod
```

Теперь создадим Pod:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: node-affinity
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: color
            operator: In
            values:
            - red
            - blue
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 1
        preference:
          matchExpressions:
          - key: type
            operator: In
            values:
            - good_pod
  containers:
  - name: node-affinity
    image: k8s.gcr.io/pause:2.0
```

Здесь, `preference` — само условие, `weight` — “насколько сильно мы этого хотим” чем больше, тем выше приоритет. Можно заметить некоторую странность в API Kubernetes: в отличие от `preferred..`, как к слову и от `podAffinity` и `podAntiAffinity` , где список правил указывается непосредственно в `...DuringSchedulingIgnoredDuringExecution`, для `nodeAffinity` требуется дополнительный уровень вложенности через `nodeSelectorTerms`. С практической точки зрения это выглядит нелепо и избыточно, поскольку в большинстве случаев мы всё равно создаём один `nodeSelectorTerms`, внутри которых уже описываются условия через `matchExpressions`. Исторически это связано с тем, что механизм `nodeAffinity` был построен поверх уже существовавшей структуры `NodeSelector`, которая содержала поле `nodeSelectorTerms`, и разработчики Kubernetes решили переиспользовать готовый API вместо создания новой структуры специально для affinity. В результате появился дополнительный уровень вложенности, который не несёт особой пользы для пользователя и часто вызывает вопросы у начинающих администраторов. Для сравнения, в `podAffinity` и `podAntiAffinity` используется более прямолинейная схема, где список правил задаётся непосредственно в `requiredDuringSchedulingIgnoredDuringExecution` без промежуточных объектов. По этой причине многие считают структуру `nodeAffinity` несколько перегруженной и менее интуитивной, чем аналогичный механизм для Pod'ов.

Применяем:

```bash
kubectl apply -f node-affinity.yaml
kubectl get pod node-affinity -o wide
kubectl describe pod node-affinity
```

Как мы видим, Pod находится в состоянии pending, планировщик показывает `FailedScheduling`, с причиной `0/5 nodes are available: 2 node(s) didn't match Pod's node affinity/selector`. Потому что условия накладываются друг на друга `required` означает обязательное условие, если оно не соблюдено то `preferred` условие не выполняется. Чтобы это исправить, удалим под, и добавим ещё лейблы:
```bash
kubectl delete -f node-affinity.yaml
kubectl label nodes worker1 color=blue
kubectl label nodes worker2 color=red
kubectl apply -f node-affinity.yaml
kubectl get pods -o wide
```
Scheduler предпочтёт первую ноду, так как она удовлетворяет условию preferred affinity. Если других факторов влияющих на выбор нет, Pod скорее всего будет размещён именно на ней.

Теперь про podAffinity. Он работает не с нодами, а с Pod’ами. То есть мы говорим: “запусти этот Pod на ноде, где уже есть Pod с таким label”. Например, если есть Pod с `app=backend`, новый Pod тоже должен попасть на ту же ноду. Anti-affinity — наоборот: “не клади меня рядом с такими Pod’ами”. Это часто используют для отказоустойчивости, например чтобы реплики одного приложения не сидели на одной ноде. 

Продемонстрируем. Для начала повесим *label* на ранее созданный Pod и запретим планирование на первой ноде:
```bash
kubectl label pod selector-pod app=store
kubectl cordon worker1
```

Рассмотрим следующий Deployment:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: pod-anti-affinity
spec:
  selector:
    matchLabels:
      app: store
  replicas: 3
  template:
    metadata:
      labels:
        app: store
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - store
            topologyKey: "kubernetes.io/hostname"
      containers:
      - name: redis-server
        image: redis:3.2-alpine
```
Здесь указан `podAntiAffinity` - то есть Pod'ы данного деплоя не будут создаваться там же где существуют Pod'ы с лейблом `app=store`. Кроме того, когда мы говорим про pod(Anti)Affinity, обязательным параметром тут служит `topologyKey`. Зачем же он нужен? Мы в самом начале говорили, что в данном виде affinity, Pod'ы притяниваются к месту где находятся другие Pod'ы с нужным label. Как раз `topologyKey` — это способ определить, что Kubernetes считает “одной зоной размещения”. Например, рассмотрим следующие ноды с label:
```text
worker1 - color=red
worker2 - color=red
worker3 - color=green
```
Если мы в podAntiAffinity поставим topologyKey: color, в таком случае, суммарно только **один** Pod будет размещён на worker1 и worker2. Так получится поскольку, у нас настроен antiAffinity на те места где уже есть поды с label `app=store`. Следовательно после размещения первого Pod, мы не можем уже размещать в том же "месте" второй. Группировка по "месту" у нас определяется topologyKey, следовательно, worker1 и worker2 считаются за одно и то же "место", так как у них одинаковый label. 

Запустим наш предыдущий Pod, и проверим что получится:
```bash
kubectl apply -f pod-anti-affinity.yaml
kubectl get pods
```
Все три пода находятся в pending и не хотят создаваться, потому что *selector-pod* у нас размёщён на второй ноде, а планирование на первой ноде запрещено. Разрешим планирование на первой ноде:
```bash
kubectl uncordon worker1
kubectl get pod -o wide
```
И что мы увидим? Создалась всего одна реплика, а остальные две находятся в состоянии Pending. Если внимательно посмотреть, сам деплой pod-anti-affinity имеет лейбл `app=store`, следовательно после запуска первой реплики **он сам начинает попадать под правило anti-affinity**, и поэтому остальные реплики уже не могут быть размещены ни на одной ноде.

### 13.3 Taints и Tolerations

Если AntiAffinity — это способ запланировать *Pod* чтобы он не попал на определённые *ноды*, то taints работают наоборот, это способ *ноды* оттолкнуть от себя определённые *Pod'ы*. Taint вешается на ноду и говорит: “*не размещай здесь Pod’ы, если у них нет соответствующего toleration*”. Toleration, в свою очередь, является своего рода исключением - он добавляется в Pod и означает: “*мне можно на ноды с таким taint’ом*”. Важно отметить что toleration **разрешает**, но **не заставляет** размещать на данной ноде Pod'ы. То есть Pod *может* попасть на такую ноду, но не обязан — scheduler всё равно выбирает лучшую из доступных.

Есть три типа taint’ов:
* `NoSchedule` — новые Pod’ы не будут размещаться на ноде
* `PreferNoSchedule` — scheduler постарается не размещать, но может, если нет других вариантов
* `NoExecute` — более жёсткий вариант: новые Pod’ы не запускаются, а уже запущенные будут удалены с ноды

Повесим taint на ноду и проверим:
```bash
kubectl taint nodes worker1 storage=ssd:NoSchedule
kubectl describe node worker1
```

Теперь создадим обычный deployment:
```bash
kubectl create deployment nginx-taint --image=nginx
kubectl scale deployment nginx-taint --replicas=3
kubectl get pods -o wide
```
И заметим, что ни один Pod не попал на `worker1`, даже если он свободен — taint его “отталкивает”.

Теперь разрешим Pod’у туда попасть через toleration. Создаём файл:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: toleration-pod
spec:
  tolerations:
  - key: "storage"
    operator: "Equal"
    value: "ssd"
    effect: "NoSchedule"
  containers:
  - name: nginx
    image: nginx
```

Применяем:
```bash
kubectl apply -f taint-toleration.yaml
kubectl get pods -o wide
```

Теперь этот Pod уже может быть запущен на `worker1`. Разберём, что здесь важно: `key` — должен совпадать с taint’ом; `operator`: `Equal` — key и value должны совпадать `Exists` — достаточно совпадения key, value игнорируется;`value` — значение taint’а; `effect` — должен совпадать с типом taint’а

Удалить taint можно точно также как удаляем лейблы:

```bash
kubectl taint nodes worker1 storage=ssd:NoSchedule-
```

Отдельный важный момент: `NoExecute`. Если применить taint `storage=ssd:NoExecute` то, новые Pod’ы туда не попадут, а уже запущенные Pod’ы **будут удалены**, если у них нет toleration. Также можно даже задать время, сколько Pod может “пережить” на ноде при помощи параметра `tolerationSeconds: 30`.

**Примечание:** не стоит путать `taint` и `label`. Несмотря на похожий внешний вид `key=value`, это совершенно разные механизмы. `label` участвует в работе `nodeSelector`, `affinity`, `service selector` и других механизмов выбора объектов. `taint` же используется только совместно с `toleration`. Наличие на Pod метки `color=red` никак не позволит ему попасть на ноду с taint `color=red:NoSchedule`. Для этого Pod должен содержать соответствующую запись в секции `spec.tolerations`. Аналогично наличие `toleration` не означает, что Pod обязательно будет запущен на такой ноде — toleration лишь снимает запрет планирования, но не указывает Scheduler'у предпочтительное место размещения. Если требуется не только разрешить запуск на определённой ноде, но и направить Pod именно туда, необходимо дополнительно использовать `nodeSelector` или `nodeAffinity`.

## 14. Запросы Ресурсов, Лимиты и Квоты

Kubernetes не умеет “угадывать” сколько ресурсов нужно приложению. Поэтому контейнеры могут начать потреблять слишком много памяти или CPU и мешать другим приложениям. Чтобы этого избежать используются requests, limits, quotas и limit ranges. `requests` используются планировщиком при выборе ноды, а `limits` уже ограничивают контейнер во время работы. Например если контейнер превысит memory limit — он может получить OOMKilled.

**Limit range** - обьект api который ограничивает ресурсы на каждый контейнер или под в пределах пространства имён. Так что если вы хотите чтобы ваши поды никогда не выходили за 1024 Мб на каждый под - вам необходимо использовать limit range. У limit range есть следующие параметры:
* `type` - определяет к чему применяется ограничение, к подам или к контейнерам.
* `defaultRequest` - это резервированные ресурсы, которые будут предоставлены приложению по умолчанию. При их превышении ничего страшного не произойдёт, но лишь значение того, что гарантированно выделено, что соседние контейнеры эти ресурсы не своруют.
* `default` - это предел ресусров, если он превышен то приложение будет тормозить, так как больше ресурсов ему не дано.
* `max` и `min` - максимальные и минимальные допустимые значения ресурсов.

**Quota** - объект api который ограничивает общее количество ресурсов, на все приложения, в пределах одного пространства. Если пространство имён настроено на квоту, то приложения в этом пространстве должны быть настроены с параметром `pod.spec.containers.resources`, иначе API Server просто запретит создание Pod'а. Это необходимо потому что Kubernetes должен понимать сколько ресурсов потребляет namespace, иначе невозможно проверить превышена квота или нет.

Невнимательный читатель может задаться вопросом, а в чём разница между *limit range* и *quota*. Разница в том что *limit range* используется для ограничения самого приложения, а *quota* используется чтобы задать максимум который может использовать весь namespace. То есть, если по совокупности в кластере у вас доступно не более 16 Гб ОЗУ - необходимо использовать *quota* чтобы приложения не вышли за пределы и не вызвали out of memory. А если вам нужно сделать так чтоб одно приложение не занимало все 16 ГБ, то можно использовать *limit range* который разрешит каждому приложению использовать не более 1024 МБ допустим.


### 14.1 Явное указание ресурсов

Обычно ресурсы для контейнера задаются непосредственно в манифесте Pod, Deployment или другого workload-объекта через секцию `resources`. Именно эти значения будут использоваться планировщиком и механизмами контроля ресурсов Kubernetes.

Рассмотрим пример:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: resource-pod
spec:
  containers:
  - name: demo
    image: registry.k8s.io/pause:3.8
    resources:
      requests:
        cpu: 700m
        memory: 256Mi
      limits:
        cpu: 1000m
        memory: 512Mi
````

Параметр `requests` используется планировщиком Kubernetes. Перед созданием Pod планировщик должен убедиться, что на ноде имеется не менее 700m CPU и 256 MiB памяти. Если подходящей ноды нет, Pod останется в состоянии `Pending`. Параметр `limits` используется уже после запуска контейнера. Контейнер не сможет потреблять больше указанного количества ресурсов. Для памяти это ограничение является жёстким, при превышении лимита контейнер обычно завершается с причиной `OOMKilled`.

Проверить выделенные ресурсы можно любой из следующих команд:

```bash
kubectl describe pod example-pod
kubectl get pod example-pod -o yaml
```

Если значения `requests` и `limits` одинаковы то Kubernetes резервирует ресурсы и одновременно запрещает контейнеру использовать больше указанного объёма. Такой подход часто используется для критически важных приложений, потребление ресурсов которых хорошо известно заранее. Стоит помнить, что Kubernetes не требует обязательного указания `requests` и `limits`. Если они не заданы, контейнер может использовать ресурсы без каких-либо ограничений. Именно поэтому в production-кластерах часто используются объекты `LimitRange` и `ResourceQuota`, которые будут рассмотрены далее.


### 14.2 Quota

Далее, рассмотрим как создать квоты. Начнём с того что создадим отдельный namespace:

```bash
kubectl create ns limited-demo
```

Перед созданием квоты можно посмотреть подсказку-параметры `kubectl create quota -h | less`. Скопируем пример и немного модифицируем.

```bash
kubectl create quota my-quota --hard=cpu=100m,memory=500Mi,pods=3 -n limited-demo
kubectl describe ns limited-demo
```

Протестируем квоту, для этого создадим деплой:

```bash
kubectl create deploy nginx --image nginx --replicas=3 -n limited-demo
kubectl get all -n limited-demo
```

Мы увидим репликасеты, мы увидим деплои, но мы не увидим подов. Посмотрим в чём же проблема:

```bash
kubectl describe -n limited-demo rs nginx-5869d7778c
```

И мы увидим вывод:

```text
Error creating: pods "nginx" is forbidden: failed quota: my-quota: must specify cpu for: nginx; memory for: nginx
```

Собственно проблема тут в том что у нас нет никаких ограничений по ресурсам для самого приложения. API Server запрещает создавать Pod'ы без requests/limits, потому что иначе quota не сможет посчитать потребление ресурсов namespace. Простой способ это исправить — задать ресурсы через `kubectl set resources -h | less`. Посмотрим подсказку и сделаем аналогично примерам:

```bash
kubectl set resources deployment nginx \
  --requests cpu=100m,memory=5Mi \
  --limits cpu=200m,memory=20Mi \
  -n limited-demo
kubectl get all -n limited-demo
```

Вывод команды нам покажет что всего один под запущен. Как это возможно, нужно же три пода. Посмотрим на namespace:

```bash
kubectl describe ns limited-demo
```

И мы можем увидеть что quota на CPU уже полностью исчерпана, ведь у нас установлен `requests.cpu=100m`, а quota разрешает только `100m` CPU на весь namespace. Один Pod занял весь доступный CPU budget. Мы можем изменить это:

```bash
kubectl get quota -n limited-demo
kubectl edit quota my-quota -n limited-demo
kubectl get all -n limited-demo
```

Поменяем `100m` например на `500m`. И что мы увидим — теперь поды успешно создались. Вот так примерно работают квоты. Не забывайте что чтобы работать с квотами необходимо установить ресурсы на ваши приложения. Возможно ещё будет вопрос, а что будет если задать две квоты — ответ: ограничения квот накладываются как логическое “и”, в итоге будет работать только то что соответствует всем квотам одновременно.

Важный факт: Pod без requests и limits получает так называемый класс качества BestEffort и будет одним из первых кандидатов на удаление при нехватке ресурсов на ноде. Этот механизм подробнее будет описан следующей главе.

### 14.3 Limit Range

Теперь разберём *limit range*. Начнём с того что посмотрим как это используется:

```bash
kubectl explain limitrange.spec
kubectl explain limitrange.spec.limits
```

Как можно заметить параметров много и они достаточно запутанные, поэтому рациональнее использовать YAML, к тому же он нагляднее показывает limit range как объект API.

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: mem-limit-range-demo
spec:
  limits:
  - default:
      memory: 512Mi
      cpu: 100m
    defaultRequest:
      cpu: 50m
      memory: 256Mi
    type: Container
```

Стоит обратить внимание, что параметр `spec.limits` объекта **LimitRange** никак не связан с параметром `resources.limits` контейнера. Это просто совпадение названий. Первый используется для описания политики пространства имён, а второй определяет ограничения конкретного контейнера.

Теперь создадим Pod:

```bash
kubectl run limitrange-test -n limited-demo --image=nginx
```

И ничего не получится, так как у нас работает quota из предыдущего примера. Тогда применим limit range:

```bash
kubectl apply -f limit-range-demo.yaml -n limited-demo
kubectl run limitrange-test -n limited-demo --image=nginx
kubectl describe pod limitrange-test -n limited-demo
```

Pod успешно создался. И мы можем заметить что у контейнера автоматически появились requests и limits, хотя вручную мы их не задавали. Это происходит на этапе создания объекта через API Server. Когда пользователь отправляет Pod без секции `resources`, компонент api-server - так называемый Admission Controller - проверяет наличие подходящего LimitRange в данном namespace и автоматически дополняет манифест значениями по умолчанию.

Фактически Kubernetes преобразует исходный Pod примерно в следующий вид:

```yaml
resources:
  requests:
    cpu: 50m
    memory: 256Mi
  limits:
    cpu: 100m
    memory: 512Mi
```

То есть, `defaultRequest` определяет значения, которые будут автоматически записаны в `resources.requests`, если пользователь их не указал. `default` определяет значения, которые будут автоматически записаны в `resources.limits`, если пользователь их не указал.

Кроме значений по умолчанию LimitRange позволяет задавать минимальные и максимальные допустимые ресурсы:

```yaml
limits:
- min:
    cpu: 50m
    memory: 128Mi
  max:
    cpu: 500m
    memory: 1Gi
  type: Container
```

В отличие от `default` и `defaultRequest`, параметры `min` и `max` ничего не подставляют автоматически. Они используются только для проверки. Если пользователь попытается создать контейнер с ресурсами за пределами допустимого диапазона, API Server отклонит запрос ещё на этапе создания объекта. Таким образом min/max запрещает создавать контейнеры с ресурсами ниже/выше указанного значения. Именно это и делает LimitRange — автоматически подставляет значения по умолчанию и одновременно контролирует допустимые границы ресурсов в пределах namespace.

## 15. Приоритет Планирования Pod'ов

Мы можем явно определить порядок, в котором Pod'ы будут планироваться и вытесняться при нехватке вычислительных мощностей - в таком случае вам поможет такой ресурс api как *PriorityClass*.Каждый *PriorityClass* это числовое значение. Увеличивая это значение - увеличивается приоритет. Когда приложения кластера выходят за пределы доступных ресурсов, Pod'ы c наименьшим Priority будут уничтожены в пользу Pod'ам с большим. 

К слову, что касается вытеснения. В предыдущей главе мы упомянули, что на основании параметров `requests` и `limits` Kubernetes автоматически присваивает Pod'ам класс качества обслуживания QoS. Этот класс используется при нехватке ресурсов на ноде и определяет какие Pod'ы будут завершены первыми. Существует три класса QoS:
- **Guaranteed** — для всех контейнеров заданы `requests` и `limits`, причём их значения совпадают.
- **Burstable** — ресурсы указаны, но `requests` и `limits` отличаются либо заданы не для всех ресурсов.
- **BestEffort** — контейнеры вообще не имеют секции `resources`.

При планировании Pod'а Kubernetes использует только значения `requests`. Планировщик проверяет, имеется ли на ноде достаточное количество свободных ресурсов для размещения нового Pod'а. Значения `limits` при выборе ноды не учитываются. Например, если на ноде доступно 2 CPU и уже размещены Pod'ы с суммарными `requests` равными 1.8 CPU, то новый Pod с `requests.cpu: 500m` не будет запланирован на эту ноду, даже если фактическое потребление процессора в данный момент значительно меньше. При нехватке памяти или процессорного времени Kubernetes старается сохранить наиболее важные Pod'ы. Первыми кандидатами на удаление становятся Pod'ы класса **BestEffort**, так как они не зарезервировали для себя никаких ресурсов. Затем рассматриваются Pod'ы класса **Burstable**, а Pod'ы класса **Guaranteed** удаляются в последнюю очередь. Именно поэтому в production-кластерах не рекомендуется запускать приложения без указания `requests` и `limits`: такие Pod'ы получают класс QoS **BestEffort** и с наибольшей вероятностью будут завершены при нехватке ресурсов на ноде.

QoS и PriorityClass часто путают, потому что оба механизма как будто “решают судьбу Pod’ов”, но на самом деле они работают в разных частях системы и никогда напрямую друг с другом не сравниваются. QoS относится к поведению Pod’а уже на ноде и используется kubelet’ом в ситуациях нехватки ресурсов: когда начинается memory/CPU pressure, система решает кого вытеснить, и здесь учитываются только классы BestEffort, Burstable и Guaranteed — приоритет Pod’а при этом вообще не играет роли. PriorityClass же работает на уровне scheduler’а и влияет только на планирование и возможное вытеснение других Pod’ов ради размещения более важного: если новому Pod’у не хватает места, scheduler может вытеснить менее приоритетные Pod’ы с ноды, даже если они имеют “лучший” QoS. Поэтому возможна ситуация, когда Pod с BestEffort и высоким priority будет спокойно вытеснять другие при планировании, но если он уже запущен и нода под давлением, он всё равно будет удалён первым при эвикции, потому что QoS всегда имеет приоритет над PriorityClass в контексте выживания на ноде. Я добавил QoS именно в эту главу, а не в предыдущюю исключительно для того чтобы подчеркнуть разницу. QoS не относится к планировщику.

Для того чтобы настроить определённый *PrioriyClass* поды дожны быть сконфигурированы с параметром `PriorityClassName`. Также *PriortyClass* может быть настроен глобально по-умолчанию, в таком случае Pod'ы которые не имеют определённого *PriorityClass* будут планироваться с этим *PriorityClass*. Также, при создании *PriorityClass* можно указать `Policy: never`, чтобы Pod'ы вообще не удалялись. Продемонстрируем как это использовать.
 
Для начала создадим, собственно два *PriorityClass*:
```bash
kubectl create priorityclass demo-hight --value=1000 --description="hight priority demonstration" --preemption-policy="Never"
kubectl create priorityclass demo-mid --value=500 --description="mid priority demonstration" --global-default=trye
```
Теперь создадим Pod и рассмотрим его описание:
```bash
kubectl run mid-pod --image=nginx
kubectl get pod mid-pod -o yaml | grep -B2 -i 'priorityclass'
```
Теперь создадим Deployment, и зададим ему приоритет через параемер `spec.template.spec.priorityClassName`:
```bash
kubectl create deploy hi-pod --image=nginx
kubectl edit deploy hi-pod
```
Фрагмент описания будет выглядеть примерно так:
```yaml
...
spec:
  template:
    spec:
      priorityClassName: demo-high
...
```
Сохраним, и проверим изменения:
```bash
kubectl get pods hi-pod-5658c7cbd6-tdx5f -o yaml | grep -B2 -i 'priorityclass'
```
Как мы видим, теперь Pod'ы деплоя созданы с повышенным приоритетом.

## 16. Сеть

Сеть в Kubernetes работает сразу на нескольких уровнях. Взаимодействие может происходить:
* между контейнерами внутри одного Pod
* между самими Pod'ами
* между Pod'ами и Service
* между внешними пользователями и приложениями внутри кластера

Связь между контейнерами внутри одного Pod обычно реализуется через IPC (*inter-process communication*), shared volumes или localhost. Отдельная “настоящая” сеть для этого не требуется, так как контейнеры одного Pod используют общий network namespace и фактически видят одну и ту же сеть. Связь между Pod'ами реализуется через CNI-плагины (*Container Network Interface*), например Calico, Flannel или Cilium. Именно они обеспечивают маршрутизацию, IP-адресацию и сетевое взаимодействие между нодами и Pod'ами. Например, при создании Pod:

```bash
kubectl run test --image=nginx
kubectl get pod -o wide
```

Можно увидеть, что Pod получает IP вроде `192.168.235.131`. Этот IP выдаётся CNI-плагином и маршрутизируется между нодами, и доступен только в пределах кластера. Находясь на ноде или другом Pod'е мы можем проверить доступность нашего приложения:
```bash
curl -I 192.168.235.131
```
Перед выполнением данного примера необходимо убедиться что firewall разрешает входящие соединения по 80/TCP. Если удалить Pod и создать его заново, IP почти всегда изменится:
```bash
kubectl delete pod test
kubectl run test --image=nginx
```
В моём случае IP изменился на `192.168.235.132`. В связи с изменением IP его невозможно использовать как стабильную точку доступа, ведь предполагается что Pod'ы могут свободно пересоздаваться при необходимости.

Для решения этой проблемы Kubernetes использует объект **Service**. Они предоставляют стабильную точку доступа к Pod'ам.
Создадим Service типа `clusterip` вручную:

```bash
kubectl create service clusterip test-svc --tcp=80:80
kubectl get svc
```
Полученный IP отличается от того что нам предоставлял Calico. В моём случае это `10.102.118.85`. Попробуем обратиться к приложению по данному IP:
```bash
curl -I 10.102.118.85
```
И у нас ничего не сработает. Потому что Service ещё не знает, куда направлять трафик. 
Если посмотреть Service:
```bash
kubectl describe svc test-svc
kubectl get endpoints
```
Можно увидеть строку `Endpoints: <none>`  - Service не имеет backend’ов, у него нет Endpoints. Попробуем это исправить, для этого отредактируем service:
```yaml
kubectl edit svc test-svc
```
В поле `spec.selector` укажем label нашего Pod'a, в моём случае это 'run: test'. После чего, проверим доступ повторно:
```bash
curl -I 10.102.118.85
kubectl get endpoints
```
Теперь приложение отвечает на запросы.  Также заметим что у нас появились `endpoints`. После создания service включается ключевой механизм Kubernetes — Endpoint Controller. Это control plane компонент, который постоянно наблюдает за Service’ами и Pod’ами. Как только у Service появляется selector, например run=test, Endpoint Controller начинает работать в режиме reconciliation: он находит все Pod’ы, подходящие под selector, извлекает их IP и формирует объект Endpoints То есть причиной появления Endpoints является не сеть и не kube-proxy, а именно наличие Service с selector’ом, который запускает reconciliation-логику контроллера. Чтобы продемонстрировать суть, пересоздадим наш Pod:
```bash
kubectl delete pod test
kubectl get endpoints
kubectl run test --image=nginx
kubectl get pod -o wide
kubectl get endpoints
curl -I 10.102.118.85
```
Как мы видим, calico-IP адрес нашего Pod'а изменился уже на `192.168.235.133`, в моём случае. Однако доступ к нему обеспечен по прежнему IP `10.102.118.85`. Пойдём дальше, и посмотрим что произойдёт если мы создадим второй Pod с тем же label, и изменим содержимое его страницы для наглядности:
```bash
kubectl run test2 --image=nginx --labels='run=test'
kubectl exec -it test -- sh -c "echo 'TEST_1' > /usr/share/nginx/html/index.html"
kubectl exec -it test2 -- sh -c "echo 'TEST_2' > /usr/share/nginx/html/index.html"
```
Проверим что будет если отправить GET-запрос нашему приложению:
```bash
root@control1:~# curl 10.102.118.85
#TEST_1
root@control1:~# curl 10.102.118.85
#TEST_2
```
Ответы могут чередоваться. Это происходит потому, что Service выполняет балансировку между Endpoints.

При создании service мы указывали тип `clusterip`. Если мы посмотрим подсказку, то можем заметить несколько типов service:
```bash
kubectl create svc --help
```
Собственно из них:
- `ClusterIP` — тип по умолчанию. Такой Service доступен только внутри кластера. Kubernetes создаёт виртуальный IP-адрес и внутренний DNS, через который Pod'ы могут обращаться друг к другу. Снаружи такой Service недоступен.
- `NodePort` — Kubernetes открывает определённый порт на каждой ноде кластера. После этого приложение становится доступно по IP самой ноды, на заданом порту. Когда запрос приходит на ноду — kube-proxy перенаправляет трафик на один из Pod'ов Service.
- `LoadBalancer` — обычно используется в облаках. Kubernetes просит cloud provider создать внешний балансировщик нагрузки и связать его с Service. Внутри такой Service обычно всё равно использует ClusterIP или NodePort.
- `ExternalName` — особый тип Service. Он вообще не проксирует трафик, а просто создаёт DNS CNAME запись на внешний ресурс.

### 16.1 NodePort

До этого момента, все наши приложения были доступны только внутри кластера. Само собой разумеется, нам необходимо обеспечить возможность отправлять запросы приложению находясь вне сети кластера - конечный пользователь так или иначе должен иметь доступ к нашему сайту. Попробуем зайти на сайт по IP нашей ноды. Для этого с хост-системы (или с виртуальной машины в той же сети) выполним следующую команду:
```bash
curl 192.168.56.111
```
Разумеется, у нас ничего не получится, ведь мы ничего не сделали чтобы доступ к Pod'ам обеспечивался с внешних хостов. Можно задаться вопросом: если ClusterIP уже даёт стабильную точку доступа к набору Pod’ов, почему бы не использовать его напрямую снаружи кластера? Чтобы ответить на это, нужно понять, где вообще “живёт” ClusterIP. Когда создаётся Service, ClusterIP представляет собой набор правил iptables - он не имеет физического или виртуального интерфейса, он не маршрутизируем - через него нет ARP/route - он не участвует в L2/L3 сетевых уровнях вообще. Вместо этого kube-proxy на каждой ноде создаёт правила в ядро Linux - iptables или ipvs, которые говорят: если пакет пришёл на ClusterIP и порт Service — перенаправь его на один из Pod IP, указанных в Endpoints. То есть ClusterIP — это полностью виртуальная конструкция, реализованная через правила NAT внутри kernel space на каждой ноде. Он “живёт” одновременно на всех нодах, но физически не привязан ни к одной из них и не существует в L2/L3 сети. Одним словом в ядре Linux нет интерфейса и нет маршрута до него, но в kube-proxy он присутствует как match-условие. Поэтому пакет “никогда не идёт к ClusterIP”, он всегда перехватывается на ноде через netfilter до стадии маршрутизации Собственно, поэтому, мы фактически не можем напрямую для него настроить передачу пакетов на L2 уровне, ведь у нас он просто отсутвтует. 

Если попытаться решить задачу, один из очевидных вариантов — выбрать одну ноду и настроить на этой ноде NAT: добавить iptables правило вида:

```bash
iptables -t nat -N MY-NODEPORT
iptables -t nat -A MY-NODEPORT -j MARK --set-xmark 0x4000/0x4000
iptables -t nat -A MY-NODEPORT -p tcp -j DNAT --to-destination 192.168.235.133:80
iptables -t nat -A PREROUTING -p tcp  -d 192.168.56.111 --dport 8081 -j MY-NODEPORT
iptables -t nat -A OUTPUT -p tcp -d 192.168.56.111 --dport 8081 -j MY-NODEPORT
```
Теперь с хост-системы или виртуальной машины имеющей маршрутизацию до нашей сети выполним команду:
```bash
curl 192.168.56.111:8081
```
Таким хитрым образом мы создали доступ к endpoint для клиентов - пользователи могут заходить на наш сайт. Решение хорошее, но необходимость в его реализации отсутствует - Kubernetes уже обеспечил подобного рода механизм NodePort как стандартизированное решение этой схемы. Вместо того чтобы вручную делать внешний → Node → endpoint NAT, Kubernetes просто добавляет ещё одну входную точку прямо в kube-proxy: открывает порт на всех нодах и добавляет iptables правило на уровне NodeIP:NodePort. То есть тип service NodePort НЕ создаёт новый Service и НЕ меняет Endpoints - он просто добавляет ещё одну точку входа в те же самые iptables правила.

Продемонстрируем на практике, что делает сервис типа *NodePort*. Для этого изменим у нашего сервиса `spec.type` на `NodePort`:

```bash
kubectl edit svc test-svc
kubectl describe svc test-svc
```
В выводе `describe` нас интересует строка `NodePort:` в моём случае Kubernetes автоматически назначил 32684/TCP. Теперь приложение доступно для внешнего пользователя по данному порту:

```bash
curl 192.168.56.111:32684
```

Kubernetes открывает указанный порт на каждой ноде кластера. При обращении клиента на `<NodeIP>:<NodePort>` запрос сначала попадает на саму ноду, а затем обрабатывается компонентом *kube-proxy*. 
```bash
kubectl get pods -n kube-system -o wide | grep 'kube-proxy'
```
Именно kube-proxy отслеживает Service и список доступных Pod'ов через API Kubernetes и настраивает правила iptables или IPVS для перенаправления трафика. После получения запроса kube-proxy выбирает один из Pod'ов Service и перенаправляет трафик на его IP-адрес. Endpoint Controller и kube-proxy работают на разных уровнях. Endpoint Controller является частью Pod'а `kube-controller-manager` и занимается тем, что наблюдает за состоянием кластера: он смотрит на Service с label-селектором, находит подходящие Pod’ы и формирует объект Endpoints, то есть по сути фиксирует “живые адреса” приложений. kube-proxy же вообще не принимает решений о том, какие Pod’ы должны обслуживать трафик — он просто берёт уже готовые Endpoints и превращает их в правила ядра Linux (iptables или ipvs), которые реально перенаправляют пакеты. Поэтому можно сказать так: Endpoint Controller отвечает за то, кто сейчас является частью сервиса, а kube-proxy отвечает за то, как именно сетевой трафик до этих участников будет доставлен.

Кстати, вручную создавать yaml Service не обязательно — Kubernetes предоставляет более простой способ через `kubectl expose`. Эта команда автоматически создаёт объект Service на основе уже существующего Pod или Deployment, подставляя нужный selector и порты. Например, вместо написания YAML можно просто выполнить 
```bash
kubectl expose pod test --type=NodePort --port=80
```
и Kubernetes сам создаст Service, свяжет его с Pod’ами через labels.

### 16.2 Ingress

NodePort в Kubernetes открывает фиксированный порт на каждой ноде кластера из диапазона 30000–32767. Этот диапазон выбран специально, чтобы избежать конфликтов с системными и стандартными сервисами, которые уже могут использовать “обычные” порты вроде 80 или 443. Формально этот диапазон можно изменить, но на практике это редко делают, так как это легко может привести к конфликтам с другими сервисами или уже существующей инфраструктурой, где порты заранее зарезервированы.

Проблема NodePort заключается не в самом диапазоне, проблема заключается количестве портов. Доступ к приложению по различным портам неудобен с точки зрения пользователя и архитектуры приложения. Особенно это становится заметно, когда в кластере появляется несколько независимых частей системы: например, отдельный deployment для нашего вымышленного интернет-магазина webshop и отдельный для веб-клиента электронной почты mail. В таком случае пользователю пришлось бы объяснять, что разные части одного продукта находятся на разных портах, например http://site.com:30091 для почты и http://site.com:30080 для магазина. Даже если не думать о пользователях, такая схема плохо масштабируется: получается что подобные ссылки с портами, нам необходимо интегрировать сразу в код, документацию и конфигурации, и любая смена порта превращается в проблему. Кроме того, при объединении сервисов или изменении архитектуры можно легко получить конфликт портов или необходимость переписывать ссылки. Адекватным действием в данной ситуации становится введение обратного прокси. Например, можно поднять обычный nginx и настроить маршрутизацию по путям. Так и сделаем:
```bash
apt install nginx
cat  > /etc/nginx/sites-available/default <<EOF
server { 
        listen 80 default_server; 
        location / { 
           proxy_pass http://127.0.0.1:32684; 
        } 
} 
EOF
nginx -t
systemctl start nginx
nginx -s reload
curl 192.168.56.111
```
Теперь с любого внешнего устройства наше приложение доступно по нормальному порту, и можно спокойно добавить proxy_pass на location типо */webshop* или */mail*. Однако нельзя не заметить серьёзный архитектурный недостаток: весь входящий трафик начинает зависеть от одной ноды `control1`, на которой собственно запущен этот nginx! Если эта нода выходит из строя, падает весь входной слой системы, и фактически вся доступность кластера становится равна доступности одного сервера. Всё для чего мы использовали kubernetes идёт в мусор, ведь отказоустойчивость мы потеряли. Кроме того, при изменении порта, нам всё же придётся вручную копаться с конфигурацией nginx и svc.

Очевидно, чтобы такой проблемы не возникало, сам слой реверс-прокси должен быть частью Kubernetes и подчиняться тем же принципам, что и остальные компоненты: перезапускаться при сбоях, перемещаться между нодами и масштабироваться при необходимости. Именно для этого и был введён механизм Ingress — он позволяет описывать правила маршрутизации на уровне кластера, а реализация этих правил выполняется Ingress Controller’ом, который работает как управляемый Kubernetes компонент и не привязан к одной конкретной ноде.

Сам Kubernetes не умеет обрабатывать Ingress “из коробки”, однако по умолчанию он предоставляет "пустой" объект API с правилами маршрутизации. Чтобы он реально работал нужен *Ingress Controller* — отдельное приложение которое читает ресурсы Ingress и настраивает nginx, haproxy или другой reverse proxy.

Не смотря на то что создавать правила ingress мы можем и на чистом kubernetes, это не делает смысла так как работать без контроллера они не будут. Поэтому установим ingress-nginx controller через Helm, имеет смысл сделать это в отдельном namespace, чтобы потом всё можно было удобно удалить:
```bash
helm upgrade --install ingress-nginx ingress-nginx \
  --repo https://kubernetes.github.io/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.service.type=ClusterIP \
  --set controller.hostPort.enabled=true \
  --set controller.replicaCount=2 
  
kubectl get pods --o wide -n ingress-nginx
```

Теперь создадим приложение:
```bash
kubectl create deploy webshop --image=nginx
kubectl expose deploy webshop --port=80 --name='webshop-svc'
kubectl describe svc webshop
```

Создадим ingress:
```bash
kubectl create ing webshop-ing --class=nginx --rule=webshop.com/*=webshop-svc:80
```
Здесь:`--class=nginx` — какой ingress controller должен обработать ресурс. После установки контроллера, нам была выдана подсказка что используется класс nginx; `webshop.com/*` — домен и путь, аналог location в конфигурации nginx,`webshop-svc:80` — backend service.

Добавим запись в `/etc/hosts`:
```bash
echo '192.168.56.121 webshop.com' >> /etc/hosts
```

Теперь проверим:
```bash
curl http://webshop.com/
curl http://webshop.com/
curl http://webshop.com/
```
Отлично, теперь мы можем заходить на сайт по нормальному, балансировка работает. Внимательный читатель мог заметить что в `/etc/hosts` мы указали IP именно нашей ноды worker1. Почему не conrtol1? В данном случае мы используем ingress-controller `hostport` который слушает 80 порт прямо на ноде, на которой он расположен. Попробуем выключить ноду worker1 и посмотерть что из этого получится:
```bash
ssh worker1
shutdown now
sed -i 's|192.168.56.121 webshop.com|192.168.56.122 webshop.com|g' /etc/hosts
curl http://webshop.com/
```
Как мы видим, доступ к сайту мы не потеряли за счёт контроллера на второй ноде, ведь мы при установке указали value `controller.replicaCount=2`. Недоступность старого IP в данном случае не проблема, так как мы бы могли установить keepalived на каждой ноде. Таким образом ingress controller обладает всеми теми же преимуществами kubernetes что и deployment, чего нельзя было сказать про nginx на conrtol1. Это не единственный плюс, кроме того ingress-controller'ы имеют разные способы публикации, например мы бы могли сделать контроллер способа *nodeport* который создаёт одноимённого типа сервисы автоматически.

Ingress поддерживает несколько правил одновременно. Для этого создадим ещё backend:
```bash
echo '192.168.56.121 site.com' >> /etc/hosts
kubectl create deploy mail --image=gcr.io/google-samples/hello-app:2.0
kubectl expose deploy mail --port=80 --target-port=8080 --name='mail-svc'
kubectl create ing site-ing  --class=nginx --rule=site.com/=webshop-svc:80 --rule=site.com/mail=mail-svc:80
```

И протестируем:
```bash
curl http://site.com/
curl http://site.com/
curl http://site.com/mail
```
Ingress Controller — это обычное приложение, которое понимает объект Ingress и умеет превращать его в реальную конфигурацию прокси. Например, мы можем посмотреть его конфигурацию при помощи следующей команды:
```bash
kubectl exec -n ingress-nginx   deploy/ingress-nginx-controller   -- cat /etc/nginx/nginx.conf | less
```

### 16.3 Gateway API

Долгое время *Ingress* был основным способом управления входящим HTTP-трафиком в Kubernetes, однако сейчас Ingress находится в состоянии *feature freeze*, то есть новый функционал в него практически не добавляется. Постепенно Kubernetes переходит на *Gateway API* — более современную и расширяемую модель сетевой маршрутизации. 

Основной недостаток Ingress заключается в том, что более гибкие возможности маршрутизации не стандартизированы. Мы, как администраторы кластера, можем только создавать правила для HTTP-маршрутизации. На практике, нам может потребоваться более сложные правила. Например, представим что мы выкатываем новую версию магазина и хотим отправлять на неё только 10% пользователей. Либо хотим маршрутизировать пользователей по какому-нибудь HTTP-заголовку. Сам объект Ingress ничего подобного описывать не умеет, реализацию данных правил берёт на себя ingress-контроллер и его внутренняя конфигурация. Зачастую, подобную конфигурацию выводят в annotations и CRD. Например, наш nginx из предыдущего примера вовсе не выполняет балансировку запросов - балансировка происходит уже на этапе service. Но это можно было бы и исправить, например установить ingress-контроллер на основе haproxy, и задать разновидность балансировки round-robin. В результате появляется неприятная ситуация: формально мы используем стандартный ресурс Kubernetes, однако реальные возможности оказываются привязаны к конкретному ingress-controller. Перейти с nginx на HAProxy или Traefik становится значительно сложнее, поскольку приходится переписывать конфигурацию. Gateway API решает эту проблему, создавая более разнообразные механизмы маршрутизации в стандартные ресурсы Kubernetes и разделяя инфраструктурную часть (Gateway) от правил маршрутизации приложений (HTTPRoute).

В Gateway API появляются новые ресурсы:
* `GatewayClass` - описывает сам gateway controller.
* `Gateway` - описывает точку входа трафика.
* `HTTPRoute` - описывает правила маршрутизации к Service.

Для начала установим Gateway API CRD — это набор Custom Resource Definitions для Kubernetes, который добавляет в кластер новые ресурсы Gateway API:
```bash
kubectl kustomize https://github.com/nginxinc/nginx-gateway-fabric/config/crd/gateway-api/standard?ref=v1.5.1 | kubectl apply -f -
```

А также установим gateway controller:
```bash
helm install ngf oci://ghcr.io/nginxinc/charts/nginx-gateway-fabric --create-namespace -n nginx-gateway --set service.type=NodePort
```

Создадим приложение:
```bash
kubectl create deploy myblog --image=nginx --replicas=3
kubectl expose deploy myblog --port=80 --name='myblog-svc'
```

В отличие от Ingress, где точка входа и правила маршрутизации описываются одним объектом, Gateway API разделяет эти обязанности на несколько независимых ресурсов. Объект Gateway отвечает только за создание точки входа в кластер, которая может использоваться для обработки трафика, то есть для фильтрации, балансировки, разделения и т. д. для бэкэндов. Напишем Gateway для нашего приложения:
```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: myblog-gw
spec:
  gatewayClassName: nginx
  listeners:
  - name: http
    protocol: HTTP
    port: 80
```
В данном примере создаётся HTTP-шлюз, который принимает соединения на 80 порту. Однако пока что он ничего не знает ни о сервисах, ни о путях вроде /mail или /shop. Для описания самих правил маршрутизации используются отдельные ресурсы, например HTTPRoute, которые мы рассмотрим далее. Такое разделение позволяет администраторам управлять сетевой инфраструктурой независимо от разработчиков приложений, которым требуется лишь добавлять собственные маршруты.

Если объект Gateway отвечает за создание точки входа в кластер, то ресурс HTTPRoute содержит непосредственно правила маршрутизации. Именно здесь указывается какие домены должны обрабатываться, какие URL-пути необходимо сопоставлять и в какой backend-service следует передавать запросы. По сути, HTTPRoute играет ту же роль, что и раздел rules в объекте Ingress.

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: myblog-route
spec:
  parentRefs:
  - name: myblog-gw
  hostnames:
  - "myblog.info"
  rules:
  - backendRefs:
    - name: myblog-svc
      port: 80
```

Применим данные манифесты:
```bash
kubectl apply -f http-gw.yaml
kubectl apply -f http-routing.yaml
```
Проверить доступность можно всё также через файл hosts:
```bash
echo '10.0.0.11 myblog.info' >> /etc/hosts
kubectl get svc -n nginx-gateway
curl http://myblog.info:31399
```

### 16.4 Управление CNI и DNS в Kubernetes

Cетевое взаимодействие между Pod'ами в Kubernetes обеспечивается CNI-плагином. В нашем случае используется *Calico*. Именно CNI отвечает за выдачу IP-адресов Pod'ам, настройку маршрутизации между нодами и сетевую связность внутри кластера. Посмотреть текущие IP-пулы Calico можно следующим образом:

```bash
kubectl get ippool -A -o yaml
```

Конфигурация самого CNI-плагина обычно хранится на нодах в каталоге:

```bash
cd /etc/cni/net.d/
ls
vim 10-calico.conflist
```

Deployment `coredns` запускается в пространстве имён `kube-system` и отвечает за DNS-разрешение внутри Kubernetes-кластера. Пока в кластере не установлен и не настроен сетевой CNI-плагин, Pod’ы `coredns` будут находиться в состоянии `Pending`, поскольку для их запуска требуется рабочая сеть между узлами и Pod’ами. Конфигурация CoreDNS хранится в объекте `ConfigMap` с именем `coredns`. Любые изменения DNS-настроек — например, добавление внешнего DNS-forwarder’а — необходимо вносить именно в этот `ConfigMap`:

```bash
kubectl get cm -n kube-system coredns -o yaml
```
Подробнее о таком ресурсе как ConfigMap будет рассказано в следующих главах. Продемонстрируем как работает DNS-резолвинг между Pod'ами. Для начала создадим Pod и Service:

```bash
kubectl run webserver --image=nginx
kubectl expose pod webserver --port=80
```

Теперь создадим тестовый Pod и попробуем обратиться к Service по имени:

```bash
kubectl run testpod --image=busybox -- sleep 3600
kubectl exec -it testpod -- wget webserver --spider
```

Как можно заметить, Kubernetes автоматически разрешает имя `webserver` через CoreDNS и перенаправляет запрос на Service. Теперь посмотрим как работает DNS между разными namespace. Создадим отдельный namespace и два Pod'а:

```bash
kubectl create ns remote
kubectl run interginx --image=nginx
kubectl expose pod interginx --port=80
kubectl run remotebox --image=busybox -n remote -- sleep 3600
```

Посмотрим DNS-настройки внутри контейнера и попробуем выполнить DNS-запрос:

```bash
kubectl exec remotebox -n remote -- cat /etc/resolv.conf
kubectl exec remotebox -n remote -- nslookup interginx
```

Имя не разрешится, потому что Kubernetes по умолчанию ищет Service только внутри текущего namespace. Поэтому при обращении между namespace необходимо указывать полное DNS-имя Service:

```bash
kubectl exec remotebox -n remote -- nslookup interginx.default.svc.cluster.local
```

Здесь: `interginx` — имя Service; `default` — namespace; `svc` — тип ресурса;`cluster.local` — DNS-домен кластера Kubernetes. Как видим, nslookup успешно возвращает IP адрес Pod'a.

## 17. Network Policies

По умолчанию в Kubernetes отсутствуют ограничения на сетевое взаимодействие между Pod’ами. Любой Pod может свободно обмениваться трафиком с другими Pod’ами, даже если они находятся в разных Namespace. Для ограничения сетевого доступа используются ресурсы `NetworkPolicy`, которые позволяют явно определять, какой трафик разрешён, а какой должен быть заблокирован. Если для Pod’а не существует подходящего правила `NetworkPolicy`, трафик будет запрещён. Однако пока в Namespace вообще не используются `NetworkPolicy`, весь сетевой трафик между Pod’ами разрешён. В `NetworkPolicy` существуют два основных направления трафика:
* `ingress` — входящий трафик к Pod’у;
* `egress` — исходящий трафик от Pod’а.

Обращаем внимание, что `Ingress` в контексте `NetworkPolicy` не имеет никакого отношения к ресурсу Kubernetes `Ingress`, который используется для публикации HTTP/HTTPS-сервисов. Это просто название.

### 17.1 Разрешение доступа к Pod по метке

Создадим политику, которая разрешает доступ к Pod’у `nginx` только тем Pod’ам, у которых присутствует метка `access=true`.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: access-nginx
spec:
  podSelector:
    matchLabels:
      app: nginx
  ingress:
  - from:
    - podSelector:
        matchLabels:
          access: "true"
---
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  containers:
  - name: nwp-nginx
    image: nginx:1.17
---
apiVersion: v1
kind: Pod
metadata:
  name: busybox
  labels:
    app: sleepy
spec:
  containers:
  - name: nwp-busybox
    image: busybox
    command:
    - sleep
    - "3600"
```

Применяем манифест:

```bash
kubectl apply -f nwpolicy-complete-example.yaml
```

Создадим Service для Pod’а `nginx`:

```bash
kubectl expose pod nginx --port=80
```

Проверим подключение из `busybox`:

```bash
kubectl exec -it busybox -- wget --spider --timeout=1 nginx
```

Подключение будет отклонено, так как у Pod’а `busybox` отсутствует метка `access=true`. Добавим необходимую метку и повторим проверку:

```bash
kubectl label pod busybox access=true
kubectl exec -it busybox -- wget --spider --timeout=1 nginx
```
Теперь соединение будет успешно установлено, поскольку Pod соответствует правилам `NetworkPolicy`.

### 17.2 Запрет доступа из других Namespace

Следующая политика разрешает входящий трафик только от Pod’ов внутри текущего Namespace.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  namespace: default
  name: deny-from-other-namespaces

spec:
  podSelector:
    matchLabels: {}

  ingress:
  - from:
    - podSelector: {}
```

В данном примере `podSelector: {}` означает "выбрать все Pod’ы", аналогично правило `from -> podSelector: {}` разрешает трафик только от Pod’ов этого же Namespace.

Создадим отдельный Namespace:

```bash
kubectl create ns nwp-namespace
```

Создадим Service:

```bash
kubectl expose pod nwp-nginx --port=80
```

Проверим доступ из Pod’а в другом Namespace:

```bash
kubectl exec -it nwp-busybox -n nwp-namespace -- wget --spider --timeout=1 nwp-nginx
```

Это работать не будет. Как мы указывали из предыдущего параграфа, если под находится в другом пространстве имён к нему нужно обращаться по полному DNS-имени:

```bash
kubectl exec -it nwp-busybox -n nwp-namespace -- wget --spider --timeout=1 nwp-nginx.default.svc.cluster.local
```

Применим подготовленный манифест:

```bash
kubectl apply -f nwp-btween-ns.yaml
```

Проверим доступность Pod'a:

```bash
kubectl exec -it nwp-busybox -n nwp-namespace -- wget --spider --timeout=1 nwp-nginx.default.svc.cluster.local
```

Доступ из другого Namespace будет заблокирован. При этом Pod’ы внутри Namespace `default` продолжат иметь доступ:

```bash
kubectl exec -it busybox -- wget --spider --timeout=1 nwp-nginx
```

Таким образом, `NetworkPolicy` позволяет гибко управлять сетевым взаимодействием между Pod’ами и Namespace внутри Kubernetes-кластера.

## 18. Обеспечение Дискового Пространства
<!-- ВАЖНО. Добавить нюанс что без StorageClass невозможно будет автоматическое увеличение параметра -->
В Kubernetes существует несколько способов работы с хранилищем (`Volumes`). Обычные `Volumes` являются частью Pod’а и описываются непосредственно внутри его манифеста. Такие volume существуют только пока работает сам Pod и чаще всего используются для временных данных, обмена файлами между контейнерами или хранения кэша. Для постоянного хранения данных Kubernetes используется следующий набор ресурсов:
* `PersistentVolume (PV)` — описание физического или виртуального хранилища в кластере. Создаётся администратором кластера и представляет собой выделенное место для хранения данных: локальный диск, NFS, Ceph, cloud disk и т.д.
* `PersistentVolumeClaim (PVC)` — запрос приложения на использование этого хранилища;
* `StorageClass` — механизм автоматического создания storage с определёнными параметрами.

Сам Pod напрямую с PV не работает. Вместо этого приложение создаёт `PersistentVolumeClaim`. PVC запрашивает определённый объём storage и режим доступа, а Kubernetes автоматически подбирает подходящий `PersistentVolume` и связывает (Bound) их между собой. Такой подход делает приложение независимым от конкретного типа хранилища. Pod’у не важно, используется локальный диск, сетевое хранилище или cloud storage — он работает только с PVC. `StorageClass` используется для динамического создания storage. Если подходящего PV ещё не существует, Kubernetes может автоматически создать новый volume через configured storage provider. Это особенно часто используется в cloud-средах.

### 18.1 Обычные Pod Volume

Продемонстрируем volume типа `hostPath`, он совершает проброс, аналогичный bind mount в Docker (`-v /var/tmp:/tmp`):
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: hostpath-pod
spec:
  containers:
  - name: nginx
    image: nginx
    volumeMounts:
    - mountPath: /usr/share/nginx/html
      name: host-storage
  volumes:
  - name: host-storage
    hostPath:
      path: /mydata
```
Создадим для начала данные, которые необходимо пробросить. Для этого на обоих woker-нодах создадим каталог `/mydata` и запишем туда какую-нибудь информацию:
```bash
for i in 1 2 ; do \
  ssh root@worker$i 'mkdir /mydata' && \
  ssh root@worker$i 'echo "Hello from Simple Storage" >> /mydata/index.html' ; \
done ;
```
Запустим Pod, и произведём проверку:
```bash
kubectl apply -f hostpath.yaml
kubectl get pod hostpath-pod -o wide
curl http://192.168.235.185
```
Как мы видим, содержимое каталога `/mydata` на worker-ноде примонтировалось в контейнер.

Далее продемонстрируем пример простого проброса volume между контейнерами, который использует тип `emptyDir`:
```yaml
apiVersion: v1
kind: Pod
metadata: 
  name: emptydir-pod
spec:
  containers:
  - name: centos1
    image: centos:7
    command:
      - sleep
      - "3600" 
    volumeMounts:
      - mountPath: /centos1
        name: test
  - name: centos2
    image: centos:7
    command:
      - sleep
      - "3600"
    volumeMounts:
      - mountPath: /centos2
        name: test
  volumes: 
    - name: test
      emptyDir: {}
```
Применим манифест, и создадим файлы в общем хранилище:
```bash
kubectl apply -f emptydir-pod.yaml
kubectl exec -it emptydir-pod -c centos1 -- touch /centos1/lunchbrake.txt
kubectl exec -it emptydir-pod -c centos2 -- ls /centos2/
```
Как мы видим, оба контейнера используют один и тот же volume и имеют доступ к общей директории. И на самом деле этот volume хранится тоже на host системе, посмотреть это можно при помощи следующей команды:
```bash
ssh root@worker1 'ls /var/lib/kubelet/pods/*/volumes/kubernetes.io~empty-dir/test'
```
В моём случае Pod создался на worker1, но это не обязательно - Pod может быть и на worker2, перед выполнением команды необходимо это проверить. Теперь пересоздадим наш Pod:
```bash
kubectl delete pod emptydir-pod
kubectl apply -f emptydir-pod.yaml
kubectl exec -it emptydir-pod -c centos2 -- ls /centos2/
```
Как мы видим, данные удалились. Volume типа `emptyDir` существует только пока существует сам Pod. После удаления Pod Kubernetes автоматически очищает данный storage.

### 18.2 Ручное создание PersistentVolume и PersistentVolumeClaim

Как мы видим обычные Pod Volume, такие как `hostPath` или `emptyDir`, подходят разве что для простых сценариев и демонстраций. Они имеют серьёзные ограничения, ведь они привязаны к конкретной Node, не позволяют удобно переносить данные между узлами. Например, если Pod будет пересоздан на другой Node, данные `hostPath` могут оказаться недоступны, а `emptyDir` вообще полностью удалится вместе с Pod’ом. Для того чтобы "по-нормальному" использовать дисковое пространство, в Kubernetes используются ресурсы `PersistentVolume (PV)` и `PersistentVolumeClaim (PVC)`. `PersistentVolume` требует описания нескольких основных параметров:
1. `storageClassName` — используется как selector для привязки PVC к определённому типу storage;
2. `capacity` — объём доступного хранилища;
3. `accessModes` — режим доступа (`ReadWriteOnce`, `ReadOnlyMany`, `ReadWriteMany`);
  * `ReadWriteOnce (RWO)` — volume может быть подключён для чтения и записи только одним Node;
  * `ReadOnlyMany (ROX)` — volume может использоваться несколькими Node только для чтения;
  * `ReadWriteMany (RWX)` — volume может одновременно использоваться несколькими Node для чтения и записи.
4. тип backend-хранилища (`hostPath`, `NFS`, `CephFS`, cloud disk и т.д.).

Важно понимать, что `PersistentVolume` сам по себе не выделяет место на диске автоматически. PV является прежде всего API-ресурсом и абстракцией над storage. Именно благодаря тому, что storage описывается отдельными объектами Kubernetes, появляется возможность автоматизировать создание и подключение хранилища через `StorageClass` и storage provider’ы. Для понимания базового принципа работы сначала создадим `PersistentVolume` вручную. Мы можем использовать уже знакомый нам `hostPath`. Приведём манифест такого PV:

```yaml
kind: PersistentVolume
apiVersion: v1
metadata:
  name: demo-pv
spec:
  storageClassName: demo
  capacity:
    storage: 1Gi
  accessModes:
    - ReadWriteOnce
  hostPath:
    path: "/newdata"
```

Создадим описанный PV:
```bash
kubectl apply -f demo-pv.yaml
kubectl get pv
kubectl get pvc
```
Как видим, у нас есть persistentVolume, но нет claim на него. Попробуем создать Pod, который бы запрашивал этот volume:
```yaml
kind: Pod
apiVersion: v1
metadata:
   name: demo-pv-pod
spec:
  volumes:
    - name: pv-storage
      persistentVolumeClaim:
        claimName: demo-pv-claim
  containers:
    - name: pv-container
      image: nginx
      ports:
        - containerPort: 80
          name: "http-server"
      volumeMounts:
        - mountPath: "/usr/share/nginx/html"
          name: pv-storage
```
Применим манифест и посмотрим что происходит:
```bash
kubectl apply -f pv-demo-pod.yaml
kubectl get pods
kubectl describe pod demo-pv-pod
kubectl get pvc
```
Как видно, Pod находится в состоянии *pending* по причине `persistentvolumeclaim "demo-pv-claim" not found`. То есть для Pod'а остуствует доступный запрос на созданный storage. Создадим PVC:
```yaml
kind: PersistentVolumeClaim
apiVersion: v1
metadata:
  name: demo-pv-claim
spec:
  storageClassName: demo
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 32Mi
```
Применим манифест, и посмотрим что произошло:
```bash
kubectl apply -f demo-pv-claim.yaml
kubectl get pvc
kubectl get pod demo-pv-pod
```
Наш Pod заработал. Теперь точно также можно добавлять данные в `/newdata` и он будет работать как hostPath.
```bash
kubectl get pods -o wide
ssh root@worker1 'echo "Aboba!!!" >> /newdata/index.html'
curl 192.168.235.189
```
Теперь пересоздадим наш Pod и PVC и посмотрим что произойдёт:
```bash
kubectl get pv
kubectl get pvc
kubectl delete pod demo-pv-pod
kubectl delete pvc demo-pv-claim
kubectl apply -f pv-demo-pod.yaml
kubectl apply -f demo-pv-claim.yaml
kubectl get pvc
kubectl get pv
```
Как мы видим, Pod не создаётся несмотря на наличие доступного `PersistentVolumeClaim`. Причина заключается в политике `persistentVolumeReclaimPolicy`, которая по умолчанию имеет значение `Retain`. После удаления PVC Kubernetes не удаляет `PersistentVolume` и не возвращает его обратно в пул свободных volume. Вместо этого PV переходит в состояние `Released`, сохраняя привязку к предыдущему claim. Это сделано специально для защиты данных: Kubernetes предполагает, что внутри volume может находиться важная информация, и не должен автоматически передавать storage другому приложению.
```text
RECLAIM     POLICY STATUS
Retain      Released
```

Именно поэтому новый PVC не может автоматически подключиться к существующему PV. Для повторного использования такого volume администратор должен вручную очистить storage и пересоздать PV либо удалить поле `claimRef`. Политика `persistentVolumeReclaimPolicy` определяет поведение Kubernetes после освобождения volume. Существует несколько основных вариантов:
* `Retain` — значение по умолчанию. PV сохраняется в состоянии `Released` и требует ручного вмешательства администратора;
* `Delete` — Kubernetes автоматически удаляет volume после удаления PVC;
* `Recycle` — устаревшая политика, при которой volume очищается и возвращается в пул свободных PV.

Политика задаётся непосредственно внутри `PersistentVolume`:
```yaml
spec:
  persistentVolumeReclaimPolicy: Delete
```

### 18.3 Автоматическое Резервирование Пространства

Предыдущий пример с ручным созданием `PersistentVolume` и `PersistentVolumeClaim` в первую очередь был нужен для демонстрации принципа работы PV/PVC. На практике то что мы делали бесполезно, ведь backend storage основан на всё том же `hostPath`. Главное преимущество системы PV/PVC раскрывается при использовании `StorageClass` и механизма `Dynamic Provisioning`. `StorageClass` — это ресурс Kubernetes, который описывает способ автоматического создания storage. Внутри него указывается provisioner — специальный компонент, отвечающий за выделение дисков или storage-ресурсов.

Provisioner может работать cloud storage (`AWS EBS`, `GCE Persistent Disk`, `Azure Disk`), сетевыми storage (`NFS`, `Ceph`, `Longhorn`), CSI-драйверами (`Container Storage Interface`), локальными storage provider'ами. В отличие от статического подхода, администратору больше не требуется вручную создавать `PersistentVolume`. Достаточно создать `PersistentVolumeClaim`, указав нужный `StorageClass`, после чего Kubernetes автоматически выполнит процесс выделения storage.

Продемонстрируем на практике автоматическое резервирование при помощи NFS-сервера. Для начала необходимо поднять, собственно, NFS-сервер. В качестве такового может выступать любой сервер, не обязательно нода кластера. В моём случае это отдельная виртуальная машина `192.168.56.300 util`. Установим инструменты для nfs на каждой ноде:
```bash
dnf install nfs-utils
```
На машине *util* установим также *nfs-utils*, и настроим его:
```bash
dnf install nfs-utils
firewall-cmd --add-service=mountd --permanent
firewall-cmd --add-service=rpc-bind --permanent
firewall-cmd --add-service=nfs --permanent
firewall-cmd --reload
mkdir /storage
semanage fcontext -a -t nfs_t /storage
echo '/storage   *(rw,no_root_squash)' >> /etc/exports
systemctl enable --now nfs-server.service
touch /storage/test_message.eml
```
Проверим на worker-нодах доступен ли наш NFS:
```bash
showmount -e 192.168.56.300
mount -t nfs 192.168.56.300:/storage /mnt
ls /mnt/
umount /mnt
```
Теперь перейдём к созданию PVC, возвращаемся на control1 ноду. Для начала нам необходи provisioner - контроллер который будет автоматически создавать PV. Например установим следующий:
```bash
 helm repo add nfs-provisioner https://kubernetes-sigs.github.io/nfs-subdir-external-provisioner/
 helm install demo nfs-provisioner/nfs-subdir-external-provisioner --set nfs.server=192.168.56.300 --set nfs.path=/storage
 kubectl get deploy
 kubectl get storageclass
 kubectl get pv
```
Как мы видим, у нас появился deploy 'demo-nfs-subdir-external-provisioner', а также storageclass, Но PV пока что нет, потому что никто его не запрашивает. Теперь мы можем создать PVC:
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: demo-nfspv-claim
spec:
  storageClassName: nfs-client # SAME NAME AS THE STORAGECLASS
  accessModes:
    - ReadWriteMany #  must be the same as PersistentVolume
  resources:
    requests:
      storage: 32Mi
```
Применим его, и проверим:
```bash
kubectl apply -f demo-nfspv-claim.yaml
kubectl get pvc
kubectl get pv
```
Как мы видим, автоматически вместе с PVC создался и PV - наш provisioner сам выделил пространство по запросу. Теперь мы можем создать Pod, который бы использовал этот Claim.
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: demo-nfs-pod

spec:
  containers:
  - name: nginx
    image: nginx

    volumeMounts:
    - mountPath: /usr/share/nginx/html
      name: nfs-storage

  volumes:
  - name: nfs-storage
    persistentVolumeClaim:
      claimName: demo-nfspv-claim
``` 
Применим манифест и создадим тестовую страничку:
```yaml
kubectl apply -f demo-nfs-pod.yaml
kubectl get pods -o wide
kubectl exec -it demo-nfs-pod -- sh -c 'echo "Hello from NFS StorageClass!" > /usr/share/nginx/html/index.html'
```
Теперь проверим содержимое NFS-сервера:
```bash
ssh root@util 'ls -R /storage/'
```
Как мы видим, provisioner автоматически создал отдельную директорию внутри /storage и разместил там данные Pod’а.

Чтобы каждый раз вручную не указывать storageClassName в PVC, можно установить default StorageClass. Kubernetes автоматически выберет storage provider по умолчанию.
```bash
kubectl patch storageclass nfs-client -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'
kubectl get storageclass
```
Default StorageClass будет помечен символом (*). Теперь PVC можно создавать без указания storageClassName:
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: auto-pvc
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 64Mi
```

Таким образом provisioner - это контроллер, работающий внутри кластера. После установки Helm chart создал Deployment demo-nfs-subdir-external-provisioner, который постоянно взаимодействует с Kubernetes API. Он следит за появлением новых PersistentVolumeClaim, Проверяет storageClassName и если PVC относится к его StorageClass, provisioner автоматически создаёт директорию на NFS-сервере и PersistentVolume, затем связывает (Bound) PV и PVC. После этого Pod может использовать выделенное хранилище.

## 19. ConfigMap и Secret

В Kubernetes `ConfigMap` используется для хранения произвольных данных, чаще всего  конфигурационных. Это отдельный API-ресурс, который позволяет хранить site-specific параметры: переменные окружения, startup-аргументы, конфигурационные файлы и другие настройки, которые не должны быть жёстко зашиты внутрь Docker-образа. Благодаря `ConfigMap` один и тот же контейнер можно использовать в разных окружениях, изменяя только конфигурацию Kubernetes без пересборки image. Ресурс `Secret` работает по похожему принципу, но предназначен для хранения чувствительных данных: паролей, токенов, ключей и т.д. В отличие от `ConfigMap`, данные в `Secret` хранятся в base64-представлении. При этом важно понимать, что base64 не является шифрованием — это лишь способ кодирования данных.

Представим, что мы написали приложение. Ему нужно знать, где находится база данных. Самый простой вариант — захардкодить адрес прямо в коде:
```python
DATABASE_HOST = "localhost"
```

Однако база данных может находится не на `localhost`, а, например, доступна через Service `postgres.default.svc.cluster.local` Тогда придётся пересобрать Docker-образ под новую конфигурацию. Когда приложение развёрнуто при помощи простых средств оркестрации, вроде Docker-compose, мы можем просто пробросить volume на хост-систему, и таким образом менять конфигурацию. При работе с kubernetes, мы имеем дело с несколькими нодами, при этом предполагается что Pod может попасть на любую ноду - мы не должны этому препятствовать. В итоге один и тот же образ приходится пересобирать только потому, что поменялась конфигурация. Именно эту проблему и решают ConfigMap. При помощи ConfigMap мы можем устанавливать переменные среды для контейнера, и нам остаётся её только указать приложению:
```python
DATABASE_HOST = os.getenv("DATABASE_HOST")
```
Обьект `ConfigMap` и `Secret` могут подключаться к Pod’ам и как environment variables и как volume с файлами конфигурации.

Рассмотрим применение ConfigMap для проброса volume. Для начала создадим обычный файл:
```bash
echo 'hello world' > index.html
```

Теперь создадим `ConfigMap` на основе этого файла:
```bash
kubectl create cm webindex --from-file=index.html
```

После создания локальный файл больше не нужен:
```bash
rm -f index.html
```

Посмотрим содержимое ConfigMap:
```bash
kubectl get cm webindex -o yaml
```

В отличие от большинства ресурсов Kubernetes, ConfigMap практически целиком состоит из данных, поэтому отдельный раздел `spec` ему не требуется - тут сразу идёт раздел `data` в котором содержатся произвольные данные в формате `key: value`. В качестве key может выступать, например, имя файла или имя переменной среды: 
```yaml
apiVersion: v1
data:
  index.html: |
    hello world
kind: ConfigMap
metadata:
  creationTimestamp: "2026-06-12T02:47:00Z"
  name: webindex
  namespace: default
  resourceVersion: "443954"
  uid: 4b507288-e8bb-4e21-a2fc-ab40333d1876
```

Конкретная интерпритация данных определяется самим ресурсом использующим ConfigMap:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: newwebserver
spec:
  containers:
  - name: nginx
    image: nginx
    volumeMounts:
    - mountPath: /usr/share/nginx/html
      name: web-content
  volumes:
  - name: web-content
    configMap:
      name: webindex
```

После создания Pod проверим содержимое файла внутри контейнера:
```bash
kubectl apply -f newwebserver.yaml
kubectl exec -it newwebserver -- cat /usr/share/nginx/html/index.html
```

Как мы видим, файл из `ConfigMap` автоматически примонтировался внутрь контейнера и стал доступен nginx как обычный файл в filesystem.

Теперь рассмотрим пример, как мы можем загрузить переменные среды. Содержимое ConfigMap при этом остаётся таким же по формату:
```yaml
apiVersion: v1
data:
  MYVAR: Aboba!
kind: ConfigMap
metadata:
  creationTimestamp: null
  name: myvar
```

Использование этого ConfigMap в качестве переменных среды определяется самим Pod'ом:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: env-configmap
spec:
  containers:
    - name: app
      command: ["sleep", "inf"]
      image: busybox
      envFrom:
        - configMapRef:
            name: myvar
```

Проверим:
```bash
kubectl apply -f myvar.yaml
kubectl apply -f env-configmap.yaml
kubectl exec -it env-configmap -- sh -c 'echo $MYVAR'
```

Кроме того, мы можем загрузить значения из ConfigMap в стороннюю переменную:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: configmap-demo-pod
spec:
  containers:
    - name: demo
      image: alpine
      command: ["sleep", "inf"]
      env:
        - name: EDITOR
          valueFrom:
            configMapKeyRef:
              name: myvar
              key: MYVAR
```

Посмотрим что получится:
```bash
kubectl apply -f configmap-demo-pod.yaml
kubectl exec -it configmap-demo-pod -- sh -c 'echo $MYVAR'
kubectl exec -it configmap-demo-pod -- sh -c 'echo $EDITOR'
```
Обратите внимание: переменная MYVAR внутри контейнера отсутствует. Мы явно указали, что значение из ключа MYVAR должно быть помещено в переменную EDITOR. Это позволяет переименовывать переменные независимо от структуры ConfigMap.

То же самое можно выполнить и для файлов, мы можем размещать их содержимое по произвольному названию, не обязательно применять то название что указано в ConfigMap:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: configmap-demo-pod2
spec:
  containers:
    - name: demo
      image: alpine
      command: ["sleep", "3600"]
      volumeMounts:
      - name: message
        mountPath: "/tmp"
  volumes:
  - name: message
    configMap:
      name: webindex
      items:
      - key: "index.html"
        path: "message.txt"
```

Проверим:
```bash
kubectl apply -f configmap-demo-pod2.yaml
kubectl exec -it configmap-demo-pod2 -- cat /tmp/message.txt
```

## 20. Probes

Kubernetes считает контейнер “живым”, если сам процесс внутри него продолжает работать. То есть если приложение запущено и контейнер не завершился с ошибкой, kubelet воспринимает это как нормальное состояние. Но на практике, тот факт что процесс активен не обязательно означает что само приложение работает. Например, веб-сервер nginx может возвращать ответ HTTP 500 - фактически в работе самого процесса nginx ошибок нет - ответ 500 вполне легитимный, его возвращает PHP-backend, например из-за ошибок в коде. С точки зрения Kubernetes это всё ещё “Running контейнер”, потому что сам процесс не упал. Для пользователей, при этом, сайт недоступен.

Мы можем изменить это поведение, для этого в описании контейнера предусмотрены параметры **probes**. Они позволяют явно описать, что именно считается “здоровым” состоянием контейнера, и как Kubernetes должен это проверять.

Существует три основных типа probes:
- **liveness probe** определяет, жив ли контейнер с точки зрения работоспособности приложения — если она падает, Kubernetes перезапускает контейнер. 
- **readiness probe** определяет, готов ли контейнер принимать трафик; если она не проходит, Pod убирается из балансировки и перестаёт получать запросы от Service, но сам контейнер при этом не перезапускается.
- **startup probe** используется на этапе запуска приложения и проверяет, успело ли оно корректно стартовать; пока она не успешна, остальные проверки не выполняются, что особенно полезно для медленно стартующих приложений.

Проверки могут выполняться несколькими способами: 
- запуском команды внутри контейнера `exec` - в таком случае, если команда возвращает 0 проверка считается `success`,
- HTTP-запросом `httpGet` - успехом считается ответ 200, остальные ответы 404, 403, 500 - неудачей.
- TCP-подключением `tcpSocket` - проверка проходит если указанный порт открыт.
- gRPC health check `grpc`. 
Независимо от выбранного механизма, Kubernetes ожидает один из трёх результатов проверки: *Success* — проверка успешна; *Failure* — проверка не пройдена; *Unknown* — результат определить не удалось, Kubernetes не предпринимает действий и повторяет проверку позже.

Рассмотрим пример liveness probe с использованием `exec`:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: probe-demo
spec:
  containers:
  - name: app
    image: busybox
    command: [ 'sh', '-c', 'touch /tmp/healthy && sleep inf' ]
    livenessProbe:
      exec:
        command: [ 'test', '-f', '/tmp/healthy' ]
      initialDelaySeconds: 5
      periodSeconds: 1
      failureThreshold: 1
```
После запуска контейнер создаёт файл `/tmp/healthy`. Каждую секунду kubelet выполняет `test`. Здесь: **initialDelaySeconds** — сколько секунд ждать после запуска контейнера перед первой проверкой. **periodSeconds** — как часто выполнять проверку. По умолчанию — каждые 10 секунд. **failureThreshold** — сколько неудачных проверок подряд требуется, чтобы Kubernetes признал probe неуспешной.

Создадим Pod и убедимся что он работает корректно:

```bash
kubectl apply -f probe-demo.yaml
kubectl get pod probe-demo
sleep 10
```

Теперь удалим файл, и проверим что произойдёт:

```bash
kubectl get pod probe-demo
kubectl exec -it probe-demo -- rm /tmp/healthy
kubectl get pod probe-demo
kubectl get pod probe-demo
```

Можем заметить, что спустя некоторое время, `RESTARTS` изменилось на 1. После удаления файла, контейнер признается неисправным и перезапускается. Таким образом, liveness probe позволяет самостоятельно определить, что именно считать "живым" приложением, а не ограничиваться фактом существования процесса. Но такой тип probe применим не во всех случаях. Представим, если бы у нас веб-сервер nginx возвращал 503 по причине включенного режима технических работ Wordpress. В таком случае, наш контейнер начал бы пересоздаваться, что никак не помогло обеспечить доступность сайта для пользователей, так в добавок ещё и помешало веб-разработчикам получить доступ к сайту. 

Для такого случая, мы можем использовать `readinessProbe`. Продемонстрируем это на практике. Рассмотрим следующий манифест:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  creationTimestamp: null
  labels:
    app: website
  name: website
spec:
  replicas: 2
  selector:
    matchLabels:
      app: website
  strategy: {}
  template:
    metadata:
      creationTimestamp: null
      labels:
        app: website
    spec:
      containers:
      - image: nginx
        name: nginx
        volumeMounts:
          - name: webconfig
            mountPath: "/etc/nginx/conf.d"
      - image: 'bitnami/php-fpm'
        name: php
        volumeMounts:
          - name: webindex
            mountPath: "/var/www/html/index.php"
            subPath: index.php
            readOnly: false
          - name: persistent
            mountPath: /tmp
      volumes:
        - name: webindex
          configMap:
            name: website-cm
            items:
              - key: 'index.php'
                path: 'index.php'
        - name: webconfig
          configMap:
            name: website-cm
            items:
              - key: 'default'
                path: 'default.conf'
        - name: persistent
          hostPath:
            path: /tmp
            type: Directory
status: {}
---
apiVersion: v1
kind: ConfigMap
metadata:
  name: website-cm
data:
  index.php: |
    <?php
    if (file_exists("/tmp/error.txt")) {
        while (true) {
            sleep(1);
        }
    }
    echo "The fate of the world is safe in Crystal Castles !\n";
  default: |
    server {
        listen 80 default_server;
        root /var/www/html;
        index index.php index.html;
        location / {
          try_files $uri $uri/ /index.php?$query_string;
        }
        location ~ \.php$ {
          fastcgi_pass 127.0.0.1:9000;
          fastcgi_index index.php;
          fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
          fastcgi_connect_timeout 3s;
          fastcgi_send_timeout 3s;
          fastcgi_read_timeout 3s;
          include fastcgi_params;
        }
    }
---
apiVersion: v1
kind: Service
metadata:
  creationTimestamp: null
  labels:
    app: website
  name: website-svc
spec:
  ports:
  - port: 80
    protocol: TCP
    targetPort: 80
  selector:
    app: website
status:
  loadBalancer: {}
```
Тут создаётся deployment с двумя репликами, в каждой из которых по два контейнера. Первый контейнер - это веб-сервер nginx, второй - php-fpm. Применим манифест и посмотрим что у нас будет создано:
```bash
kubectl apply -f website.yaml
kubectl get svc website-svc
curl 10.97.201.67
```
Как мы видим, у нас есть сервис с нашим сайтом. При обращении к нему, он выдаёт сообщение. Представим, что после очередного обновления разработчики допустили ошибку, из-за которой обработка запросов в PHP-приложении начала зависать. Сам процесс php-fpm продолжает работать, однако больше не способен корректно обслуживать пользователей:
```bash
kubectl get pods --selector='app=website'
kubectl exec -it website-5d5fd6ccd9-5p64g -c php -- touch /tmp/error.txt
```
Проверим доступность сайта теперь. Сделать это необходимо несколько раз:
```
curl 10.97.201.67
curl 10.97.201.67
curl 10.97.201.67
```
Как мы видим, периодически запрос зависает, и через 3 секунды выдаёт 504 Timeout. Тогда удалим всё что мы создали, и попробуем добавить в `spec.template.spec.container[0]` livenessProbe:
```yaml
#...
    livenessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 1
      periodSeconds: 5
      timeoutSeconds: 4
      failureThreshold: 1
#...
```
Тут, *timeoutSeconds* — сколько ждать ответа от проверки перед тем, как считать её неуспешной.

Проверим что получится при заражении сайта:
```bash
kubectl delete -f website.yaml
ssh worker1 'rm /tmp/error.txt'
vim website.yaml
kubectl apply -f website.yaml
kubectl get pods --selector='app=website'
kubectl get svc website-svc
kubectl exec -it website-646f9c8c7f-bsr6j -c php -- touch /tmp/error.txt
kubectl get pods --selector='app=website'
curl 10.103.225.78
curl 10.103.225.78
curl 10.103.225.78
kubectl get pods --selector='app=website'
```
И ничего не поменялось. Зато Pod успел целых два раза рестартануть! Что, как можно догадаться, по сути только вредит человеку который будет пытаться это отладить. Изменим тип probe с `livenessProbe` на `readinessProbe` и пересоздадим деплой:
```bash
kubectl delete -f website.yaml
ssh worker1 'rm /tmp/error.txt'
vim website.yaml
kubectl apply -f website.yaml
kubectl get pods --selector='app=website'
kubectl get svc website-svc
```

Что же произойдёт теперь в случае заражения:
```bash
kubectl get endpoints website-svc
kubectl exec -it website-9869f8dc6-mr9vx -c php -- touch /tmp/error.txt
curl 10.99.227.171
curl 10.99.227.171
curl 10.99.227.171
kubectl get pods --selector='app=website'
kubectl get endpoints website-svc
```
Как мы видим, сайт не утратил доступности: Endpoint неисправного Pod исчез из списка Endpoints сервиса, и весь пользовательский трафик автоматически начал направляться только на исправную реплику. При этом сам Pod не был перезапущен и продолжил работать, что позволяет администраторам и разработчикам исследовать проблему непосредственно в работающем окружении.

## 21. Downward API

Иногда приложению требуется информация не о внешней конфигурации, а о самом Kubernetes-объекте, внутри которого оно работает. Например, нам необходимо заранее указать приложению имя своего Pod, имя Node, на которой был размещён Pod и т.д. Конечно, уже работающее приложение может получить эту информацию через Kubernetes API. Однако для этого необходимо реализовать взаимодействие с API, выдать Pod соответствующие права доступа и самостоятельно обработать полученные данные. Это довольно трудоёмко, и избыточно. Поэтому нам может потребовать заранее до запуска иметь некую "переменную" которая бы автоматически заполнялась при размещении Pod'а. Для передачи подобной информации Kubernetes предоставляет механизм Downward API. Kubernetes передаёт данные из описания Pod'а внутрь контейнера.

Рассмотрим что это всё значит на практике:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: playground
spec:
  containers:
    - name: app
      image: busybox
      command:
          - sleep
	  - inf
      env:
        - name: MY_POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name

        - name: MY_NAMESPACE
          valueFrom:
            fieldRef:
              fieldPath: metadata.namespace

        - name: MY_NODE_NAME
          valueFrom:
            fieldRef:
              fieldPath: spec.nodeName
```
Переменные среды `env` заполняются значениями `fieldRef.fieldPath` - это собственно поля самого Pod'а. Запустим Pod и посмотрим что произойдёт:
```bash
kubectl apply -f playground.yaml
kubectl exec -it playground -- sh -c 'echo $MY_POD_NAME'
kubectl exec -it playground -- sh -c 'echo $MY_NAMESPACE'
kubectl exec -it playground -- sh -c 'echo $MY_NODE_NAME'
```
Как мы видим, в переменных среды теперь есть данные о том на какой ноде запустился Pod, в каком namespace и какое у него имя. Если в приложении эти данные нужны, мы можем ими спокойно воспользоваться, не выполняя лишней работы. Кроме этого, через resourceFieldRef можно получить информацию о ресурсах контейнера:
```yaml
env:
  - name: CPU_LIMIT
    valueFrom:
      resourceFieldRef:
        resource: limits.cpu
```
Работает это аналогичным образом, просто в отличии от `fieldRef` нам не придётся угадывать индекс массива контейнера.

Иногда приложению удобнее получать информацию не через переменные окружения, а через файлы. Например, если программа умеет перечитывать конфигурацию без перезапуска.
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: playground-volume
  labels:
    app: demo
spec:
  containers:
    - name: app
      image: busybox
      command:
          - sleep
	  - inf
      volumeMounts:
        - name: podinfo
          mountPath: /etc/podinfo
  volumes:
    - name: podinfo
      downwardAPI:
        items:
          - path: pod-name
            fieldRef:
              fieldPath: metadata.name
          - path: pod-namespace
            fieldRef:
              fieldPath: metadata.namespace
          - path: app-label
            fieldRef:
              fieldPath: metadata.labels['app']
```
После запуска Pod внутри контейнера появятся указанные файлы. Проверим это:
```bash
kubectl exec playground-volume -- cat /etc/podinfo/pod-name
```
Команда выведет playground-volume. Таким образом, Downward API позволяет безопасно предоставить контейнеру информацию о самом Pod, не выдавая ему дополнительных прав доступа к кластеру Kubernetes.

Стоит подчеркнуть что **Downward API всегда работает только с объектом Pod**, а не с Deployment, ReplicaSet или другими контроллерами Kubernetes. Например, если Pod был создан Deployment'ом, то выражение:

```yaml
env:
- name: POD_NAME
  valueFrom:
    fieldRef:
      fieldPath: metadata.name
```

вернёт имя конкретного Pod, например `web-6b8d9f5d8c-7k2mv` а не имя Deployment `web`. Это связано с тем, что контейнеры запускаются внутри Pod, и именно спецификация Pod доступна kubelet во время запуска контейнера. По этой же причине через Downward API нельзя напрямую получить имя Deployment или ReplicaSet. Если необходимо передать имя родительского объекта, обычно используют labels или annotations, которые затем читаются через `fieldRef`.

При использовании `resourceFieldRef` по умолчанию ресурсы читаются из текущего контейнера. Однако можно явно получить ресурсы другого контейнера того же Pod, для этого необходимо указать `containerName`:

```yaml
env:
- name: SIDECAR_CPU_REQUEST
  valueFrom:
    resourceFieldRef:
      containerName: sidecar
      resource: requests.cpu
```

Таким образом контейнеры внутри одного Pod могут получать информацию о `requests` и `limits` друг друга. А вот тип `fieldRef` вообще не позволяет обращаться к произвольным полям других контейнеров, например к `spec.containers[0].resources.requests.cpu`; для доступа к ресурсам других контейнеров предусмотрен исключительно механизм `resourceFieldRef` с указанием параметра `containerName`.

## 22. Kubernetes аутентификация

В дополнение к главе 8.3, хотелось бы подробнее раскрыть всё что нам может пригодиться относительно сертификатов и авторизации. 

Чтобы понять, как Kubernetes использует сертификаты, полезно сначала разобраться, что вообще представляет собой сертификат. Зайдём издалека. Шифрование - это операция для превращения данных в нечитабельный вид таким образом, чтобы вернуть данные обратно в привычный вид можно только при помощи ключа. Можем вспомнить простой алгоритм шифрования - "шифр Цезаря" - суть в том чтобы сместить алфавит на некоторый шаг, тем самым превратив фразу в набор бессмысленных символов. В качестве ключа тут выступает число на которое мы смещаем алфавит. Опустим то что сам шифр Цезаря легко перебрать, проблема симметричного шифрования заключается в том, что шифрование и расшифровка выполняется одним и тем же ключом - после того как мы зашифровали данные, нам необходимо передать ключ, и любой кто его перехватит сможет расшифровать секретные данные. Поэтому возникает необходимость в ассимметричном шифровании - то есть, чтобы шифровка и расшифровка происходила разными ключами - один только шифрует, и называется *публичным*, другой расшифровывает и называется *приватным*. Запомнить легко: зашифровать данные может каждый, поэтому ключ которым шифруют можно рассказывать всем, а вот расшифровать секретные данные нельзя никому кроме владельца ключа, поэтому он и назван приватным. 

Так как все данные в компьютере можно представить в виде чисел, то мы можем представить алгоритм как некоторую математическую функцию. Предположим каким бы могла быть функция "шифрования". Большинство операций являются симметричными, например:
```txt
C = M xor e ; # шифрование
M = C xor d ; # раcшифровка
e = d ;
```
где С - это шифр, M - исходные данные, e - ключ к для шифрования, и d - ключ для расшифровки. Такие операции заведомо не подходят, ведь как уже было сказано, они симметричные.

Возмьём тогда операции которые не обладают свойством инволютивности. Например:
```
C = M × e ;
M = C × d ;
e = 1/d  <->  d = 1/e
```
Как мы видим, тут из ключа `e` легко получить `d`, что не добавляет смысла, ведь тогда от того что ключи формально по значению разные, но толку никакого - зная `e` можно получить и `d`, а значит это равносильно симметричному шифрованию, ведь мы можем прямо из вервого выражения выразить `M`, т.е. расшифровать и вообще без приватного ключа:
```txt
M = С/e
```
Таким образом, нам нужно придумать аналогичную функцию, но только чтобы вычислить `d = f⁻¹(e)` было невозможно. Например мы можем сделать функцию от нескольких переменных `d = g(e,s)`, ведь если у нас кроме `e` есть ещё и `s`, то мы не можем вычислить `d` не зная `s`. В таком случае возникает вопрос, а что это за `s`? Если это просто произвольно заданное "число", то возникает вопрос - каким образом о нём должен узнать шифровальщик. Ведь если `e = f(d,s)` то для шифрования нам нужно знать `s`. Мы можем передать его вместе с `e`:
```txt
C = M × e  s ;
M = С × d ;
e = 1/(d × s)  <->  d = 1/(e × s) 
```
Но если мы так сделаем, то теряется весь смысл ведь это всё равно что `d = f(e)` - можно считать что приватный ключ составной из `e,s`. Ничего не поменялось. Собственно, нам необходимо как-то убрать `s` из первого выражения. Это возможно в том случае, если функция`e = f(d,s)` одностороняя, то есть такая, что вычислить `e` зная `d` - легко, но вычислить `d` зная `e` практически невозможно. Сформулируем задачу:
```txt
C = M op e ;
M = С op d ;
e = f(d,s)   <->   d ≠ f⁻¹(e,s);
```
Как можно заметить, для выводе `e` мы выражали первое. Следовательно, найти одностороннюю функцию — значит найти такую операцию `C = f(M,e)`, которую легко вычислить по M и e, но практически невозможно обратить, зная только `C` и `e`. Обычные алгебраические функции нам не подойдут, потому что пратически все они вычислимы - если мы знаем `C` и `e` нам не сложно вычислить и `M`. Даже если возьмём операцию возведения в степень:
```txt
C = M^e ;
M = С^d ;

M = ᵉ√C   ->  ᵉ√C = С^d   ->  С^(d - 1/e) = 1
```
Решить эту задачу, конечно сложно, но сложность у нас только на бумаге. Функция `x = ᵉ√y` как известно - монотонна. Это значит что любое значение в её некотором интервале, пропорционально значению в интервали кратном этому. То есть, нам достаточно вычислить значения от 1..7, и потом поделить `y` столько раз, пока оно не поместиться в этот интервал, затем взять число по таблице, и домножить его на количество итераций. Для компьютеров эта задача лёгкая, в общем-то лбой калькулятор сможет вычислить это за мгновение. Обычная алгебра нам не подходит. На помощь приходит модульная арифметика (остатки от деления). Когда мы берем остаток, мы отбрасываем часть информации. Например, `15 mod 7 = 1`. Зная числа 1 и 7, мы никогда не узнаем, было это 15, 8 или 22. Иными словами:
```txt
a mod b = c
<=>
a = c + b · k
```
То есть остаток от деления `mod` ломает обратимость. В модульной арифметике операций не так много, всего три, попробуем поискать среди них односторонние:
1. Модульное сложение: `C ≡ M+e (mod n)`. Чтобы найти M, достаточно просто посчитать `M ≡ C − e (mod n)`. Следовательно функция не подходит.
2. Модульное умножение: `C ≡ M ⋅ e(mod n)` Обратная задача: `M = C · e⁻¹ mod n` нужно найти число, на которое при умножении на `e`, возвращает единицу. В математике есть расширенный алгоритм Евклида, который находит это для любых гигантских чисел. Не смотря на сложный внешний вид, функция обратима.
3. Модульное возведение в степень: `C ≡ M^e (mod n)` Обратная задача: `M^e ≡ C (mod n)`, то есть согласно теоремме Эйлера:
```txt
C = M^e mod n          # Прямая задача
M^e ≡ C (mod n)        # Обратная задача
d = e⁻¹ mod φ(n)       # Связь
```
где `φ(n)` - количество чисел от `n..n-1` взаимно простых с `n` (то есть их НОД = 1). Если число `n` само простое, то тут обратимность присутствует, ведь тогда `φ(n) = n - 1`. Но вот если число `n` не простое, то тут возникают сложности с вычислением. Мы можем попытаться решить задачу "в лоб" и сделать алгоритм, который ищет число φ(n):
```C
pn = 0
for(int k = 0; k < n; k++){
  if ( gcd(k,n) == 1 ){
    pn++;
  }
}
```
Современные процессоры работают довольно быстро, на частоте около 2,2 - 5 ГГц. Положим, что одна проверка gcd занимает 10 нс, а число `n` мы зададим равным `n ≈ 2^2048 = 10^617` тогда:
```
10^617 / 10^8 = 10^609 секунд = ≈ 10^601 лет
```
Для сравнения, возраст вселенной ≈ 10^10 лет. Иными словами, таким способом вычислить `n` невозможно никогда. Мы нашли `op`. Мы знаем, что шифровать будем так:
```txt
C = M^e mod n
M = С^d mod n
```
Теперь осталась задача собственно определить `e = f(d,s)` Подставляем первое во второе:
```txt
M = С^(ed) mod n
```
Необходимо определить при каких e и d условие выполняется. Согласно теореме Эйлера, если `gcd(M,n)=1,`, то
```txt
M^φ(n) ≡ 1 (mod n).
```
Следовательно:
```txt
M^(ed) = M^(1+kφ(n)) = M·(M^φ(n))^k  ≡ M mod n
```
Задача сводится к тому, чтобы подобрать e и d таким образом, что:
```txt
ed ≡ 1 (mod φ(n))
```
Функцию эйлера `φ(n)` мы можем вычислить так:
```txt
φ(n) = (p-1)(q-1)
```
Где `p` и `q` - простые, то есть ни на что не делятся. Доказательство тут cледующее. Представим ряд чисел. Чтобы число из нашего ряда имело с `n` общие делители (то есть не было взаимно простым с ним), оно должно делиться либо на `q` либо на `p`. Например, если `p=3`, фильтр отсеивающий *не-взаимно-простые* числа убивает каждое 3-е число. Так как `n` кратно `p` - каждое третье это и есть 1/3 всех чисел. Выживает 2/3 чисел. Аналогично можно сказать и про число `q`. То есть сумма чисел которые взаимно простые с φ(n) это `pq * (p-1)/p *(q-1)/q` сокращаем получаем `(p-1)(q-1)`. Любое число раскладывается на простые множители единственным образом, поэтому следующая операция обладает уникальным свойством
```txt
p * q = n
n = ?... # нам нужно подобрать уникальную пару p,q?
```
Таким образом, функция для генерации приватного ключа у нас получается следующая:
```txt
e = d⁻¹ mod φ(n)
d = e⁻¹ mod φ(n)
```
В качестве секрета `s` выступает пара `p,q`. Число `φ(n)` легко вычислить только если мы знаем `p` и `q`, однако по готовому `n` вычислить его проблематично. Поэтому мы можем генерировать публичный ключ на основании приватного, только если у нас есть числа `p` и `q`, при этом если у нас есть число `n`и `e`, мы можем только зашифровать информацию. Пара `n,e` называется **публичным ключом**. Приватным ключом считается пара `n,d`.

Операция цифровой подписи строится на тех же математических операциях, что и шифрование, но смысл у них противоположный. Если записать базовую форму подписи, то она выглядит как если бы мы шифровали приватным ключом, а расшифровывали публичным:

```txt
S = M^d mod n
M = S^e mod n
```

Здесь `M` — это хэш сообщения. Сначала отправитель вычисляет хэш сообщения и возводит его в степень `d` по модулю `n`, получая подпись `S`. Получатель, имея открытый ключ `(n, e)`, возводит подпись в степень `e` и восстанавливает тот же хэш `M`. То есть процесс противоположный, один человек может зашифровать, и все могут расшифровать. В чём тогда смысл? Смысл в том, чтобы доказать, что данные были отправлены владельцем приватного ключа. Если полученный результат совпадает с хэшем, вычисленным из сообщения, значит данные не были изменены и подпись действительно создана владельцем приватного ключа. Если же подпись была сделана кем-то другим - она не расшифруется, значит данные некорректные. Если же данные не совпадут с хэшем - значит подменили данные, а подпись оставили. Подпись проверяет сразу две вещи: 1. владелец ключа настоящий, и 2. данные подлинны. Подпись — это не “обратное шифрование”, а доказательство владения ключом

В отличие от шифрования, здесь не скрывается содержание сообщения — оно остаётся открытым. Цель подписи не в секретности, а в подтверждении двух свойств: **целостности** (сообщение не изменено) и **аутентичности** (его действительно подписал владелец приватного ключа). Именно поэтому подпись можно проверить любому, у кого есть публичный ключ, но подделать её без знания `d` практически невозможно, так как это эквивалентно решению обратной задачи.

Можно представить простую схему обмена ключами без доверенной инфраструктуры: сервер генерирует пару `server.key / server.pub`, клиент — `client.key / client.pub`. При соединении происходит handshake, где стороны обмениваются публичными ключами и дальше пытаются шифровать данные: клиент использует `server.pub` для отправки сообщений, сервер — `client.pub` для ответов. Однако такая модель сразу ломается в реальном мире, потому что публичные ключи передаются по сети, а значит атакующий может перехватить их и подменить на свои — клиент думает, что получил ключ сервера, но на самом деле использует ключ злоумышленника `evil.pub`, который в свою очередь расшифровывает данные своим ключём `evil.key`, читает, затем снова шифрует и отправляет на сервер, и вся схема превращается в MITM.

Чтобы решить проблему подмены, вводится сертификат. Сертификат — это публичный ключ + подпись доверенного сервера. Возникает вопрос, собственно что это нам даёт? Злоумышленник ведь может сам и подписать сертификат своим ключом. Особенность в том, что доверенный центр (в данном примере это мы сами), генерирует ключевую пару `ca.key`, `ca.pub` и подписывает так называемый корневой сертификат - `ca.pem`. Он заранее устанавливается в систему браузер, ОС или корпоративную инфраструктуру и никогда не передаётся по сети и используется только для проверки подписи. Его задача — определить, каким центрам сертификации вообще можно верить. Для шифрования трафика, мы уже другую ключевую пару `server.key`, `server.pub`, но CA (в данном случае это мы сами) подписывает публичный ключ не ключом `server.key` а ключом `ca.key`. В отличие от `ca.pem`, `server.pem` является “переменной” частью системы: сервер может выпускать, обновлять или заменять свои сертификаты, и он отправляется клиенту при каждом новом соединении. При установлении соединения клиент получает `server.pem` и проверяет его подпись с помощью локально сохранённого `ca.pem`. Если подпись валидна, это означает, что публичный ключ действительно выдан доверенным CA, после чего клиент принимает его как идентичность сервера. Важно, что CA не участвует в передаче данных и не присутствует в самом соединении — он используется только как критерий проверки, а не как участник обмена. Таким образом если злоумышленник подключается к нашей сети, он не сможет отправить нам поддельный сертификат - так как `ca.pem` хранится у нас и не изменяется.

В Kubernetes нет “одной пары ключей”. Каждый компонент кластера — это отдельный TLS-клиент и/или TLS-сервер со своими ключами и корневыми сертификатами.

В Kubernetes можно упростить устройство сертификатов до трёх основных “зон хранения”. Во-первых, /etc/kubernetes/pki — это центральное хранилище удостоверяющей инфраструктуры control plane: здесь лежит корневой CA (ca.key, ca.crt) и сертификаты всех ключевых компонентов кластера (apiserver, controller-manager, scheduler, etcd). Этот уровень отвечает за выпуск и подпись всех сертификатов в кластере, то есть формирует “единый источник доверия”.

Во-вторых, /var/lib/kubelet/pki — это область kubelet, где хранится его собственная идентичность (client certificate и ключ), используемая для аутентификации перед API server и иногда для предоставления kubelet как TLS-сервера. Kubelet всегда использует клиентский сертификат для аутентификации перед API server. Дополнительно он может иметь серверный сертификат, если включена защита HTTPS-интерфейса kubelet.

И в-третьих, ~/.kube/config — это конфигурация клиента (kubectl), где хранится его клиентский сертификат, приватный ключ и CA, которому он доверяет. Эти данные используются для установления TLS-соединения с kube-apiserver: клиент доказывает свою личность и проверяет, что сервер действительно подписан доверенным CA.

Когда мы выполняем:
```bash
kubectl get pods
```
kubectl не общается с кластером "магическим образом". Он читает kubeconfig-файл `~/.kube/config`. В kubeconfig хранится три типа информации:
* **clusters** — куда подключаться;
* **users** — как проходить аутентификацию;
* **contexts** — какая пара cluster/user используется в данный момент.

Структура конфига примерно следующая:
```yaml
apiVersion: v1

clusters:
- name: kubernetes
  cluster:
    server: https://10.0.0.11:6443
    certificate-authority-data: ...

users:
- name: kubernetes-admin
  user:
    client-certificate-data: ...
    client-key-data: ...

contexts:
- name: kubernetes-admin@kubernetes
  context:
    cluster: kubernetes
    user: kubernetes-admin
    namespace: default

current-context: kubernetes-admin@kubernetes
```

То есть, при любом запросе, будь то упомянутый выше `kubectl get pods`, происходит обычное TLS-соединение с API-сервером Kubernetes. Kubectl читает файл `~/.kube/config`, в котором хранится информация о кластерах - `clusters`, пользователях  - `users`. Из секции `clusters` он узнаёт адрес API-сервера - `clusters[0].cluster.server` и корневой сертификат `clusters[0].cluster.certificate-authority-data` или путь к `ca.crt`. Как можно догадаться, этот сертификат не используется для шифрования трафика — он нужен только для проверки подлинности API-сервера. Kubectl устанавливает TLS-соединение, API-сервер отправляет свой сертификат `/etc/kubernetes/apiserver.crt`, а клиент проверяет его подпись при помощи локально сохранённого корневого сертификата CA. Если подпись неверна, соединение немедленно разрывается: это защита от атаки "человек посередине" (MITM).

После того как подлинность API-сервера подтверждена, начинается взаимная аутентификация клиента. Если в kubeconfig указаны `client-certificate-data` и `client-key-data`, kubectl отправляет API-серверу свой клиентский сертификат и доказывает владение соответствующим приватным ключом. API-сервер проверяет подпись этого сертификата своим доверенным CA и извлекает из него сведения о пользователе. Поле `CN` (Common Name) сертификата интерпретируется Kubernetes как имя пользователя, а поля `O` (Organization) — как группы, например `system:masters`. Именно эти значения участвуют в RBAC и определяют, какие действия разрешены пользователю. А вот, запись: `users[0].name`**не является именем пользователя Kubernetes**. Это всего лишь локальный ключ записи внутри kubeconfig, позволяющее контекстам ссылаться на нужный набор учётных данных. Реальное имя пользователя определяется содержимым сертификата, а не именем записи в конфигурационном файле.

Секция `contexts` связывает между собой кластер и пользователя. Например: использовать кластер `kubernetes`, учётные данные `kubernetes-admin`, а если namespace явно не указан в команде, считать namespace `default` пространством имён по умолчанию. Таким образом, контекст — это просто удобный "профиль подключения", позволяющий быстро переключаться между разными кластерами и пользователями.

После того как становится понятно, что такое сертификаты, подписи и CSR, команды OpenSSL перестают выглядеть как магические заклинания. На практике для работы с Kubernetes обычно достаточно понимать назначение нескольких утилит. у OpenSSL есть довольно логичная структура команд, и её можно описать примерно так: `openssl <утилита> <опции_к_этой_утилите`. Например, при помощи утилиты x509 мы можем работать с сертификатами. Опция `-in` тут собственно отвечает за сам файл сертификата, а опция `-out` за получившийся файл. В большинстве случаев нам может потребоваться просто посмотреть информацию в сертификате не выполняя генерации каких-либо файлов, для этого существует опция `-noout`. В качестве практического упраждения возьмём наш сертификат и посмотрим его некоторые данные. Для начала найдём сертификат в конфигурационном файле и выведем его в отдельный файл. Обращаем внимание, что сертификат закодирован в base64:
```bash
kubectl config current-context  # Проверим какой контекст используется
grep -A4 'contexts:' ~/.kube/config # Найдём какой юзер используется в контексте
grep -A20 'users:' ~/.kube/config | grep -A4 'name: kubernetes-admin' | grep 'client-certificate-data:' | cut -d ':' -f2,2 | tr -d ' ' | base64 -d > client.crt
```
Теперь посмотрим его полное содержание, сделать это можно при помощи опции text:
```bash
openssl x509 -in client.crt -text -noout
```
Тут, нас может заинтересовать, например поле subject, в котором указано настоящее имя пользователя kubernetes.

Рассмотрим процесс создания другого пользователя. Для чистоты эксперимента, работать будем на другой ноде, например на worker1, при чём от обычного пользователи (не от root):
```
ssh petya@worker1
```
Что нам собственно нужно сделать, чтобы "зарегистрироваться" в kubernetes? Для начала необходимо сгенерировать ключевую пару. Сделать мы это можем при помощи утилиты genrsa:
```bash
openssl genrsa -out key.key 2048
cat key.key
```
В названии файла мы видим фразу "BEGIN PRIVATE KEY". Стало быть, это приватный ключ? Но как нам вычислить публичный, он ведь требует `e = d⁻¹ mod φ(n)`, а секрет `p * q` мы уже утратили! При помощи утилиты rsa мы можем посмотреть что содержится в файле:
```bash
openssl rsa -in key.key -text -noout
```
Тут содержатся следующие все данные:
- modulus (n)
- publicExponent (e)
- privateExponent (d)

То есть формат .pem содержит ключевую пару, а не только один приватный ключ. Теперь нам необходимо каким-то образом отправить публичный ключ нашему СА, чтобы он вернул нам уже сертификат - то есть добавил метаданные и подпись. Чтобы всё это удобно поместить в один файл, существует CSR - Certificate Signing Request - это контейнер в котором содержится публичный ключ и все необходимые данные. Сгенерировать его можно при помощи утилиты req:
```bash
openssl req -new -key key.key -out key.csr -subj "/CN=petya"
```
Сразу же посмотрим что там внутри:
```bash
openssl req -in key.csr -text -noout
```
Как мы видим, в наш запрос на подпись попала только публичная часть ключа - поля `Modulus` и `Exponent`. Далее, передаём пакет нашему CA:
```bash
scp key.csr root@control1:~/
ssh root@control1
```
Мы подключились обратно к control1 - тут находится наш доверительный центр CA. Подпишем наш публичный ключ:
```bash
openssl x509 -req -in key.csr -CAkey /etc/kubernetes/pki/ca.key -CA /etc/kubernetes/pki/ca.crt -out key.crt
openssl x509 -in key.crt -text -noout
```
Как можно заметить, мы зачем-то указали параметр `-CA`. Но зачем? Подпись же происходит приватным ключом, и чтобы проверить подлинность нам достаточно выполнить верификацию? На самом деле, ничего страшного не происходит. Несмотря на то что фактически подпись создаётся приватным ключом ca.key, OpenSSL также требует сертификат ca.crt, чтобы скопировать информацию об издателе - Issuer. В публичном ключе же нет метаданных, они есть только уже в сертификате. Фактически, публичный ключ в подписи не участвует. Отправим сертификат, и `ca.crt` обратно клиенту:
```bash
scp /etc/kubernetes/pki/ca.crt key.crt petya@worker1:~/
ssh petya@worker1
```
Сразу посмотрим как работает проверка. Выполнить её можно утилитой verify:
```bash
openssl verify -CAfile ca.crt client.crt
```
Результат должен быть `OK`. Представим, получил доступ у устройству и подменил публичный ключ у сертификата:
```bash
cp -a key.crt original.crt
sed -i 's|GvAwDQY|GdAwDQY|g' key.crt
openssl verify -CAfile ca.crt key.crt
```
Не удалось пройти проверку сертификата по содержимому. Хэш, что хранится в подписи не совпадает с фактическими данными. Смоделируем другую ситуацию, представим что сам публичний ключ корректный, но злоумышленник подменил его по пути:
```bash
openssl req -x509 -new -key key.key -out mtim.crt -subj "/CN=Fake-CA"
mv mtim.crt key.crt
openssl verify -CAfile ca.crt key.crt
```
Сертификат не прошёл проверку подлинности СА.

После того как сертификат получен, можем приступить к его установке:
```bash
openssl rsa -in key.key -out original.key -traditional
cat >  ~/.kube/config <<EOF
apiVersion: v1 
clusters: 
- cluster: 
    certificate-authority: /home/petya/ca.crt 
    server: https://10.0.0.11:6443 
  name: kubernetes 
contexts: 
- context: 
    cluster: kubernetes 
    user: petya 
  name: my-context 
current-context: my-context 
kind: Config 
preferences: {} 
users: 
- name: petya 
  user: 
    client-certificate: /home/petya/original.crt 
    client-key: /home/petya/original.key 
EOF
kubectl get pods
```
Успех! Мы получили ошибку `Forbidden`, а это значит мы смогли корректно установить SSL/TLS соедниение с api-server, которвый в свою очередь вернул нам ответ что не хватает прав на чтение подов. Исправим это при помощи RBAC:
```bash
ssh control1
kubectl create role guest --verb=list,get,watch --resource=pod,deploy,daemonset
kubectl create rolebinding petya-guest --user=petya --role=guest
ssh worker1
kubectl get pods
```
Таким образом, мы обеспечили доступ кластеру для стороннего юзера.

# 23. Конфигурация CoreDNS

За DNS отвечает deploy с названием coredns. Его конфигурация содержится в одноимённом ConfigMap. Посмотреть его можно следующей командой:
```bash
kubectl get cm -n kube-system coredns -o yaml
```
Конфигурационный файл состоит из набора **server block** — независимых блоков обработки DNS-запросов. Каждый блок начинается с указания зоны, за которую он отвечает, и порта, на котором слушает запросы:
```txt
.:53 {
    ...
}
```

Точка `.` обозначает корневую DNS-зону, то есть "все домены". Такой блок будет обрабатывать любые запросы, если для них не найден более специфичный обработчик. Например:

```txt
corp.local:53 {
    ...
}
.:53 {
    ...
}
```
При такой конфигурации, специфичные правила будут применяться только к запросам вида `db.corp.local`, `vpn.corp.local` и другим именам внутри зоны `corp.local`. Если запрос подходит сразу под несколько зон, используется наиболее специфичная. Например, запрос `db.corp.local` попадёт в блок `corp.local:53`, а не в `.:53`.

Внутри каждого блока располагаются **плагины**, которые выполняются сверху вниз и определяют поведение DNS-сервера. Например:

```txt
.:53 {
    # errors
    # cache 30
    forward . /etc/resolv.conf
}
```

Плагин `errors` включает вывод ошибок в журнал. `cache 30` кэширует DNS-ответы на 30 секунд, уменьшая нагрузку на вышестоящие DNS-серверы. Плагин `forward` пересылает запросы другому DNS-резолверу, который указан в виде IP или файла формата resolv.conf. Для нас важны плагины `forward` и `kubernetes` - означает обработать запрос собственным резолвером corends. Получается примерно следующий синтаксис:
```txt
<dns-zone.com>:53 {
    kubernetes <cluster-domain.local> [cluster-domain2.local] [... etc] {
    }
    forward <domain.info> <resolver-IP or IP:Port or resolv.conf>  [...same etc] {
    }
}
<dns-zone2.com>:53 {
    ...
}
```
В качестве практики выполним следующее упражнение. Допустим у нас есть тестовый Pod:
```bash
kubectl run test --image=curlimages/curl -- sleep inf
```
Мы можем из него отправлять запросы, например к google или к ресурсам в нашем кластере:
```
kubectl exec -it test -- curl -I https://www.google.com
kubectl exec -it test -- curl -I http://webstress.default.svc.cluster.local
```
Но попробуем допустим подключиться к сайту grep.geek:
```bash
kubectl exec -it test -- curl -I http://grep.geek
```
Должны увидеть ошибку `Could not resolve host`, потому что домены .geek относятся к любительской сети DNS "OpenNIC". Для нас не важно, что именно выступает сторонним DNS. Вместо OpenNIC это мог бы быть корпоративный BIND-сервер, Active Directory DNS или любой другой DNS-резолвер. Отредактируем coredns таким образом, чтобы сайт стал доступным `kubectl -n kube-system edit cm coredns`:
```txt
    .:53 {
        errors
        health {
           lameduck 5s
        }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
           pods insecure
           fallthrough in-addr.arpa ip6.arpa
           ttl 30
        }
        prometheus :9153
        forward . /etc/resolv.conf {
           max_concurrent 1000
        }
        cache 30 {
           disable success cluster.local
           disable denial cluster.local
        }
        loop
        reload
        loadbalance
    }
    geek.:53 {
      forward . 195.10.195.195
    }
```
Проверим теперь:
```bash
kubectl -n kube-system rollout restart deployment coredns
kubectl exec -it test -- curl -I http://grep.geek
```
Мы успешно смогли подключится к ресурсу. Также вместо отдельного блока можно использовать `froward geek 195.10.195.195` в основном блоке:
```txt
    .:53 {
        errors
        health {
           lameduck 5s
        }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
           pods insecure
           fallthrough in-addr.arpa ip6.arpa
           ttl 30
        }
        prometheus :9153
	forward geek 195.10.195.195
        forward . /etc/resolv.conf {
           max_concurrent 1000
        }
        cache 30 {
           disable success cluster.local
           disable denial cluster.local
        }
        loop
        reload
        loadbalance
    }
```

Кроме CoreDNS, настройки DNS могут быть заложены в спецификации самого Pod'а. Мы можем задавать Pod'ам hostname:
```yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: nginx
  name: web
spec:
  hostname: noginxxx
  subdomain: web-svc # subdomain должен матчится с именем service
  containers:
  - image: nginx
    name: test
```
Сам по себе hostname простро настраивает имя хоста, которое никак не влияет на DNS. Однако "hostname + subdomain + pod + service" совместно дают DNS запись для указанного hostname. Например, создадим вышеуказанный yaml:
```bash
kubectl apply -f web.yaml
kubectl expose pod web --port=80 --name='web-svc' --cluster-ip=None
```
Теперь мы можем обратиться к нему по hostname:
```bash
kubectl exec -it test -- curl -I http://noginxxx.web-svc.default.svc.cluster.local
```
А могли бы и обратиться по IP:
```bash
kubectl exec -it test -- curl -I http://192-168-189-127.default.pod.cluster.local
```
Но, как мы знаем, этот IP-адрес не стабилен, и может меняться при пересоздании Pod'а.

Важно отметить, что в случае с deployment и statefulset, подобное поведение не возникает - Pod'ы не начинают резолвиться по hostname. В данном случае явное указание hostname не даёт вообще **ничего**! Deploy сам устанавливает hostname для своих Pod. Кстати, в случае с statefulset, Pod'ы резолвятся по своему стабильному имени, например `web-0.web-svc.default.svc.cluster.local`. 

По умолчанию каждый Pod получает DNS-конфигурацию Kubernetes запросы к сервисам кластера `*.svc.cluster.local` обрабатываются CoreDNS, а все остальные запросы пересылаются на вышестоящие DNS-серверы.  С помощью поля `spec.dnsPolicy` можно изменить источник DNS-конфигурации Pod. Поддерживаются следующие значения:
* `ClusterFirst` — использовать CoreDNS Kubernetes (значение по умолчанию);
* `Default` — использовать DNS-конфигурацию **ноды**, на которой запущен Pod;
* `ClusterFirstWithHostNet` — использовать CoreDNS даже если Pod запущен с `hostNetwork: true` - параметр заставляет Pod использовать реальную сеть ноды вместо собственной сети Kubernetes.
* `None` - настройки DNS игнорируются, используются только настройки внутри самого Pod'а. Кстати, о них...

Настройки DNS для конкретного Pod можно задать в секции `spec.dnsConfig`:

```yaml
spec:
  dnsPolicy: None
  dnsConfig:
    nameservers:
    - 1.1.1.1
    searches:
    - corp.local
    options:
    - name: ndots
      value: "2"
```

Параметр `nameservers` задаёт дополнительные DNS-серверы, `searches` — список поисковых доменов, а `options` позволяет настроить параметры работы DNS-резолвера. Эти настройки объединяются с конфигурацией, полученной из `dnsPolicy`.
